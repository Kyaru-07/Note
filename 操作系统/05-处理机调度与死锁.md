# 第10讲 处理机调度

## 1. 概念引入

### 1.1 什么是处理器调度

处理器调度（Processor Scheduling）是将 CPU 时间以公平、合理方式分配给进程使用的机制。

调度对系统性能的意义：
- 类比学校排课、工厂排程，决定组织结构效率
- 类比社会财富分配，影响社会公平和运行效率
- Android/iOS 用户体验差异、服务器/PC 性能差异：处理器调度是关键因素之一

### 1.2 作业基本概念

**作业（Job）**：用户提交给系统的一项完整工作（如学生编程实验、银行数据备份），整个作业有时需要作为一个整体考虑。

**作业步（Job Step）**：一个作业划分为若干个相对独立又相互关联的顺序加工步骤，上一个作业步的输出经常作为下一个作业步的输入。

示例：`find / -name "*.c~" | rm -f` 是一个编程实验作业，包含查找和删除两个作业步。

### 1.3 作业阶段和状态

与进程类似，作业也有生命周期，包括三个阶段：

| 阶段 | 状态 | 说明 |
|------|------|------|
| 收容阶段 | 后备状态 | 用户提交作业后，系统创建作业控制块（JCB, Job Control Block）登记相关信息 |
| 运行阶段 | 运行状态 | 作业被选中载入内存，分配资源并建立进程 |
| 完成阶段 | 完成状态 | 作业运行完成或异常提前结束，系统回收资源 |

### 1.4 处理机调度机制的组成

- **排队器**：将就绪进程排队
- **分派器（Dispatcher）**：暂停当前进程执行，进行上下文（Context）切换，跳到新选出进程执行
- **上下文切换器**：负责保存/恢复进程现场

## 2. 三级调度

处理机调度分为三个层次：

| 调度类型 | 别名 | 功能 | 频率 |
|----------|------|------|------|
| 高级调度 | 作业调度、长程调度 | 将外存后备队列中的作业调入内存，创建进程，分配资源，放入就绪队列 | 低 |
| 中级调度 | 中程调度、内存调度、交换调度 | 将暂时不能运行的进程转移到外存等待；将外存上已具备运行条件的就绪进程重新调入内存 | 中 |
| 低级调度 | 短程调度、进程调度 | 从就绪队列中选取进程，将处理机分配给它，让进程真正执行 | 高 |

小结：交换调度属于内存管理功能（为进程腾内存）；作业调度是资源（内存）分配，也包括处理机调度；进程调度才真正分配处理器。

## 3. 调度时机与方式

### 3.1 处理机调度时机（何时分配 CPU）

五种触发调度的事件：

1. **进程结束**：正常结束或出错终止，让出 CPU，必须选择另一进程执行
2. **等待某个事件**：进程因 I/O（如 `scanf`）或定时等待（如 `sleep`）进入等待状态
3. **CPU 时间片用完**：分时系统中时间片耗尽但进程未结束，强行重新分配 CPU
4. **新进程产生**：新进程可能很紧急，需决定是否分配 CPU 给它
5. **等待事件已发生**：I/O 完成、睡眠结束、同步事件发生，进程被唤醒回就绪状态

分类归纳：
- 主动放弃 CPU：进程结束、等待事件
- 强制夺走 CPU：时间片用完
- 给新就绪进程执行机会：新进程产生、等待事件发生

### 3.2 非抢占式调度 vs 抢占式调度

| 类型 | 英文 | 特点 |
|------|------|------|
| 非抢占式调度 | Non-preemptive | 进程一旦获得 CPU，除非完成、出错或阻塞，不会被抢走 CPU |
| 抢占式调度 | Preemptive | 调度程序可根据某种原则暂停正在执行的进程，重新分配 CPU |

### 3.3 调度程序与分派程序

- **调度程序（Scheduler）**：按操作系统设计要求，实现特定的进程调度算法，决定"分配给谁"
- **分派程序（Dispatcher）**：暂停当前进程执行，进行上下文切换，跳到新选出进程执行，决定"怎样分配"

## 4. 调度算法目标

### 4.1 共同目标

1. **资源利用率**：CPU 利用率（CPU Utilization）越高越好，应使 CPU 和其他资源尽可能忙碌
2. **公平性**：各进程获得合理 CPU 时间，防止饥饿（Starvation）现象发生
3. **平衡性**：CPU 繁忙型作业和 I/O 繁忙型作业比例适当
4. **策略强制执行**：反映操作系统目标，哪怕某些工作延迟也要执行

### 4.2 批处理系统目标

- **平均周转时间（Turnaround）短**：用户从作业提交到作业完成的时间间隔
  - 周转时间 Ti = 完成时间 - 到达时间
  - 平均周转时间 = ΣTi / n
  - **带权周转时间** = Ti / TSi（TSi 为实际服务时间）
- **系统吞吐量（Throughput）高**：单位时间内完成的作业数或进程数
- **处理机利用率高**

### 4.3 分时系统目标

- **响应时间快**：从用户提交请求到屏幕显示结果的时间
- **均衡性**：响应时间与请求服务复杂性相适应

### 4.4 实时系统目标

- **满足截止时间要求**：必须开始执行的最早时间、必须完成执行的最迟时间
- **可预测性**：如多媒体系统需保证各帧连续均匀播放

## 5. 批处理系统调度算法

### 5.1 先来先服务（FCFS, First Come First Served）

**算法思想**：先进入就绪队列的进程被优先选中运行，进程一旦获得 CPU 便执行下去，直到完成或阻塞才释放。

**示例**：进程到达顺序 P1(24), P2(3), P3(3)

| 进程 | 到达顺序 | 服务时间 | 完成时间 | 等待时间 |
|------|----------|----------|----------|----------|
| P1 | 第1 | 24 | 24 | 0 |
| P2 | 第2 | 3 | 27 | 24 |
| P3 | 第3 | 3 | 30 | 27 |

- 平均等待时间：(0 + 24 + 27) / 3 = **17**
- 平均周转时间：((24-0) + (27-0) + (30-0)) / 3 = **27**

**改变到达顺序** P2(3), P3(3), P1(24)：
- 平均等待时间：(6 + 0 + 3) / 3 = **3**（比前例好得多）

**优点**：简单，易于实现
**不足**：短作业所需等待时间过长，对短作业不公平（"护航效应"）

### 5.2 短作业优先（SJF, Short Job First）

**算法思想**：从就绪队列中选出所需 CPU 时间（burst 时间，需估算）最短的进程或作业优先调度。

两种模式：

| 模式 | 英文 | 说明 |
|------|------|------|
| 非抢占式 SJF | SJF (non-preemptive) | 进程拥有 CPU 后，可在本次 burst 结束后才让出 CPU |
| 抢占式 SJF | SRTF (Shortest Remaining Time First) | 有比当前进程剩余时间更短的进程到达时，立即抢占 CPU |

**示例**：P1(0,7), P2(2,4), P3(4,1), P4(5,4)

**非抢占式 SJF**：
- 调度顺序：P1(0→7) → P3(7→8) → P2(8→12) → P4(12→16)
- 平均等待时间 = (0 + 6 + 3 + 7) / 4 = **4**

**抢占式 SRTF**：
- 调度过程：P1(0→2) → P2(2→4) → P3(4→5) → P2(5→7) → P4(7→11) → P1(11→16)
- 平均等待时间 = (9 + 1 + 0 + 2) / 4 = **3**

**优点**：
- 具有最小的平均周转时间和平均等待时间（已证明）
- 调度性能比 FCFS 好

**不足**：
- 需要知道作业所需运行时间，实现上有困难
- 对长作业不利，甚至导致饥饿问题

### 5.3 响应比最高者优先（HRRF, Highest Response Ratio First）

**引入原因**：FCFS 与 SJF 都是片面的调度算法。

**算法原理**：同时考虑等待时间和作业大小两个因素。

**响应比公式**：

```
响应比 = 1 + 等待时间 / 估计运行时间
```

每次分配处理机时，将 CPU 分配给响应比最高的作业。

**算法特点**：
- 短作业估计运行时间短，容易得到较高响应比
- 等待时间足够长后，也将获得足够高响应比，不会发生饥饿
- 既照顾短作业又不使长作业等待过久

**示例**：作业1(0,20), 作业2(5,15), 作业3(10,5), 作业4(15,10)

| 时刻 | 事件 | 作业2响应比 | 作业3响应比 | 作业4响应比 |
|------|------|-------------|-------------|-------------|
| 0 | 作业1开始执行 | - | - | - |
| 20 | 作业1完成 | 1+15/15=2 | 1+10/5=3 | 1+5/10=1.5 |
| 25 | 作业3完成 | 1+20/15≈2.33 | - | 1+10/10=2 |
| 40 | 作业2完成 | - | - | 选中 |

调度顺序：作业1 → 作业3 → 作业2 → 作业4

三种算法对比：

| 算法 | 平均周转时间 T | 平均带权周转时间 W |
|------|---------------|-------------------|
| FCFS | 38.75 | 3.13 |
| SJF | 25 | 2.25 |
| HRRF | 26.25 | 2.46 |

HRRF 是 SJF 与 FCFS 的折衷，调度性能介于二者之间。

### 5.4 基于优先级调度法

**算法思想**：根据确定的优先级选取作业，每次选择优先数高的作业。

**优先级确定方式**：

| 方式 | 说明 | 示例 |
|------|------|------|
| 外部优先数 | 由用户规定，涉及收费时优先级越高收费越高 | 用户自行指定 |
| 内部优先数 | 由系统计算，考虑运行时间、等待时间、作业类型等 | SJF 和 HRRF 是特殊的优先级调度 |

## 6. 分时系统与实时系统调度算法

### 6.1 分时系统特点

- 系统把 CPU 运行时间分成很短的时间片，按时间片轮流分配给各联机作业使用
- 多用户同时性、交互性、独立性
- 主要设计目标：交互性（响应性）
- 典型系统：Linux、Unix

### 6.2 时间片轮转（RR, Round Robin）

**算法思想**：OS 为每个进程分配一个时间片（Time Slice，通常 10-100ms），进程在时间片内拥有 CPU。时间片用完时，OS 强行夺取 CPU，并将进程放到就绪队列末尾。

**示例**：P1(0,7), P2(2,4), P3(4,1), P4(5,4)，时间片 q=4

调度过程：
- 0-4：P1 执行（剩余 3）
- 4-5：P2 执行（剩余 3，P3 此时到达）
- 5-8：P3 执行完成
- 8-11：P2 执行完成
- 11-16：P1 执行完成
- 16-20：P4 执行完成

RR 的平均周转时间比 SJF 长，但响应时间更短。

**时间片大小对系统开销的影响**：
- 假设 10 个就绪进程，上下文切换开销 10ms
- 时间片 200ms：系统开销比率 = 10/(200+10) = **4.8%**
- 时间片 100ms：系统开销比率 = 10/(100+10) = **9.1%**（增加 4.3%）

时间片越小，响应性能越好，但切换频繁增大调度开销，需折衷。

### 6.3 优先权调度算法（HPF, Highest-Priority-First）

为每个进程设置优先级数字，系统总是选择优先级最高的进程投入运行。

**静态优先级**：在创建进程时确定，整个运行期间不变。

确定原则：
- 进程类型：系统进程优先级高，用户进程优先级低
- 资源需求：CPU 时间多→优先级低；I/O 时间多→优先级高
- 用户要求：紧急程度高、付费多→优先级高

**动态优先级**：在进程运行过程中不断变化。

确定原则：
- 等待时间长→优先级高（防止饥饿）
- 运行时间长→优先级低
- 考虑进程使用资源的类型

动态优先级资源利用率高、公平性好，但开销大、实现复杂。

**非抢占式优先级算法示例**：

| 进程 | 到达时间 | 服务时间 | 优先数（小=高） |
|------|----------|----------|-----------------|
| P1 | 0 | 7 | 3 |
| P2 | 2 | 4 | 2 |
| P3 | 4 | 1 | 4 |
| P4 | 5 | 4 | 1 |

调度顺序：P1(0→7) → P2(7→11) → P4(11→15) → P3(15→16)

- 平均周转时间 = ((7-0) + (15-2) + (16-4) + (11-5)) / 4 = **9.5**
- 平均响应时间 = [(0-0) + (7-2) + (15-4) + (7-5)] / 4 = **4.5**

**抢占式优先级算法示例**（同上数据）：

调度过程：P1(0→2) → P2(2→6) → P4(6→10) → P1(10→15) → P3(15→16)

- 平均周转时间 = ((15-0) + (10-2) + (16-4) + (9-5)) / 4 = **9.75**
- 平均响应时间 = [(0-0) + (2-2) + (15-4) + (5-5)] / 4 = **2.75**

### 6.4 多级反馈队列（MFQ, Multilevel Feedback Queue）

**算法思想**：
1. 设置多个就绪队列，队列 1 优先级最高，其余逐个降低
2. 各队列时间片大小不同，优先级越高时间片越短
3. 新进程进入队列 1 末尾，按 FCFS 等待调度
4. 若一个时间片未完成，超时后转入队列 2 末尾，如此下去
5. 仅当高优先级队列为空时，才调度低优先级队列
6. 若高优先级新进程到达，抢占当前运行的低优先级进程

**算法性能分析**：

| 用户类型 | 特点 | 获得的性能 |
|----------|------|-----------|
| 终端型用户 | 交互型作业，通常较小 | 第一队列时间片内完成，周转时间短 |
| 短批处理用户 | 稍长作业 | 在第二、第三队列各执行一时间片完成，周转时间仍较短 |
| 长批处理用户 | 大型作业 | 依次在各队列中运行，不必担心长期得不到处理 |

多级反馈队列是分时系统中应用最广泛的调度算法。

### 6.5 公平原则调度算法

**保证调度算法**：向用户做出明确的性能保证。
- 跟踪每个进程自创建以来已执行的处理时间
- 计算每个进程应获得的处理机时间
- 调度程序选择比率最小的进程，让其运行到超过最接近它的进程比率为止

**公平分享调度算法**：使所有用户获得相同的处理机时间。
- 示例：用户 1 有 4 个进程 A/B/C/D，用户 2 有 1 个进程 E
- 调度序列：A E B E C E D E A E B E C E D E...

### 6.6 实时调度

**实时调度条件**：
1. 提供必要的信息（就绪时间、截止时间、处理时间、资源要求、优先级）
2. 系统处理能力：m 个周期性硬实时任务，单处理机下需满足 Σ(处理时间/周期) ≤ 1
3. 采用抢占式调度机制
4. 具有快速切换机制

**实时调度算法分类**：

| 算法 | 英文 | 特点 |
|------|------|------|
| 最早截止时间优先 | EDF (Earliest Deadline First) | 截止时间越早优先级越高，可抢占或非抢占 |
| 最低松弛度优先 | LLF (Least Laxity First) | 松弛度 = 必须完成的时间点 - 本身所需运行时间 - 当前时间 |

### 6.7 优先级倒置

**问题**：高优先级进程被低优先级进程延迟或阻塞。

场景：P1（高优先级）、P2（中）、P3（低）共享临界资源。P3 进入临界区后，P2 抢占 P3，导致 P1 被 P2 间接阻塞。

**解决方法**：
1. 简单方法：P3 进入临界区后不允许被抢占（临界区短时可行）
2. 实用方法：**优先级继承**——P1 要进入临界区时，若 P3 正在使用该资源，则 P3 继承 P1 的优先级，直到 P3 退出临界区

## 7. 总结复盘

### 7.1 调度算法对比总结

| 算法 | 类型 | 优点 | 缺点 | 适用系统 |
|------|------|------|------|----------|
| FCFS | 批处理 | 简单 | 短作业等待长，不公平 | 批处理 |
| SJF/SRTF | 批处理 | 平均周转时间最短 | 需预知运行时间，长作业饥饿 | 批处理 |
| HRRF | 批处理 | FCFS 与 SJF 折衷 | 需计算响应比 | 批处理 |
| 优先级 | 通用 | 灵活，可反映紧急程度 | 可能饥饿 | 通用 |
| RR | 分时 | 响应时间短 | 平均周转时间长，开销大 | 分时 |
| MFQ | 分时 | 兼顾各类用户 | 实现复杂 | 分时 |
| EDF/LLF | 实时 | 满足截止时间 | 计算复杂 | 实时 |

### 7.2 核心概念速记

- **三级调度**：高级（作业）→ 中级（内存交换）→ 低级（进程）
- **非抢占 vs 抢占**：关键区别在于是否允许强制剥夺 CPU
- **FCFS**：简单但有"护航效应"
- **SJF/SRTF**：最优平均等待时间，但可能导致饥饿
- **RR**：时间片大小是关键——太小开销大，太大退化为 FCFS
- **MFQ**：综合多种算法优点，分时系统首选
- **优先级倒置**：通过优先级继承解决

### 7.3 课后要点

- 掌握各类调度算法的 Gantt 图绘制和性能指标计算
- 理解非抢占/抢占模式下同一算法的差异
- 区分批处理/分时/实时系统的调度目标差异
- 了解 Linux CFS 调度器的基本原理（vruntime 概念）

# 第11讲 死锁

## 1. 概念引入

### 1.1 生活中的死锁现象

- 城市塞车：每辆车都被其它车辆挡住前行道路，卡在路口无法动弹
- 过桥例子：只能一个方向通行，桥的每一部分都可看成资源。若死锁发生，可由一辆车返回解决（抢占资源并回退），也可能很多车都不得不返回，还可能产生饥饿

### 1.2 进程死锁定义

如果两个或两个以上进程中，每一个进程都在等待获得由其它进程把持的某种系统资源，导致这批进程间**相互永久等待**的情况，称为死锁（Deadlock）。

示例代码（两个线程以不同顺序获取锁）：

```c
// 线程1：先锁 first_mutex，再锁 second_mutex
pthread_mutex_lock(&first_mutex);
pthread_mutex_lock(&second_mutex);
// ... 工作 ...
pthread_mutex_unlock(&second_mutex);
pthread_mutex_unlock(&first_mutex);

// 线程2：先锁 second_mutex，再锁 first_mutex
pthread_mutex_lock(&second_mutex);
pthread_mutex_lock(&first_mutex);
// ... 工作 ...
pthread_mutex_unlock(&first_mutex);
pthread_mutex_unlock(&second_mutex);
```

若两个线程同时执行，线程 1 持有 first_mutex 等待 second_mutex，线程 2 持有 second_mutex 等待 first_mutex，即产生死锁。

## 2. 资源分类与死锁原因

### 2.1 资源分类

| 资源类型 | 英文 | 说明 | 是否可能引起死锁 |
|----------|------|------|------------------|
| 可抢占资源 | Preemptable | 可被其他进程或系统强行收回（如 CPU、主存） | 不会 |
| 不可抢占资源 | Non-preemptable | 不能强行收回，只能使用完自行释放（如磁带机、打印机） | 会 |
| 可消耗资源 | Consumable | 进程运行期间动态创建和消耗（如消息、信号） | 会 |

### 2.2 死锁原因

**原因一：进程推进顺序不当（争夺不可抢占资源）**

场景：系统有打印机、读卡机各一台，进程 P 和 Q 共享。

```
进程P：P(s1) → P(s2) → 使用 → V(s1) → V(s2)
进程Q：P(s2) → P(s1) → 使用 → V(s2) → V(s1)
```

若 P、Q 几乎同时分别请求读卡机和打印机并获得成功，然后再分别请求另一设备时，因已无可用资源而双双陷入等待——死锁。

**原因二：等待不存在的消耗性资源**

三个进程使用消息通信：
```
P1：receive(p3,m3) → send(p2,m1)
P2：receive(p1,m1) → send(p3,m2)
P3：receive(p2,m2) → send(p1,m3)
```

P1 等 P3 的 m3，P2 等 P1 的 m1，P3 等 P2 的 m2——循环等待，死锁。

### 2.3 死锁与资源数量的关系

**问题**：m 个资源被 n 个进程共享，每个进程至少申请 k 个资源，不发生死锁的条件是什么？

**分析**：
- 发生死锁时，各进程拥有的资源数为 k-1，且系统无可用资源
- 可能发生死锁的最大资源数 C = n(k-1)
- 不发生死锁的条件：**m ≥ n(k-1) + 1**，即 **m > n(k-1)**

**示例**：哲学家就餐问题
- n=5（5 个哲学家/进程），m=5（5 根筷子/资源），k=2（每人需 2 根筷子）
- C = 5×(2-1) = 5，m=5 不满足 m > 5，可能死锁
- 增加一根筷子（m=6），满足 m > 5，不会死锁

## 3. 死锁的必要条件

### 3.1 四个必要条件（高频考点）

产生死锁的四个必要条件**必须同时满足**：

| 条件 | 英文 | 含义 |
|------|------|------|
| ①互斥条件 | Mutual Exclusion | 一个资源在同一时刻只能分配给一个进程，若另一进程请求已被占用的资源，必须等待 |
| ②请求和保持条件 | Hold and Wait | 进程已保持至少一个资源，又提出新的资源请求，而该资源已被其它进程占有 |
| ③不可抢占条件 | No Preemption | 进程已获得的资源，在未使用完之前不能被抢占，只能在使用完时由自己释放 |
| ④循环等待条件 | Circular Wait | 存在一个循环等待链，每一个进程分别等待它前一个进程所持有的资源，造成永远等待 |

**重要性质**：
- 四个条件是必要条件，不是充分条件（满足四个条件不一定死锁，但死锁一定满足四个条件）
- 只要破坏其中任何一个条件，就可以预防死锁

### 3.2 死锁特征

- 参与死锁的进程最少是两个
- 参与死锁的进程至少有两个已经占有资源
- 参与死锁的所有进程都在等待资源
- 参与死锁的进程是当前系统中所有进程的子集

## 4. 处理死锁的方法总览

| 方法 | 英文 | 特点 |
|------|------|------|
| 预防死锁 | Deadlock Prevention | 设置限制条件，破坏四个必要条件中的一个或几个 |
| 避免死锁 | Deadlock Avoidance | 不事先限制，动态分配时防止系统进入不安全状态 |
| 检测死锁 | Deadlock Detection | 允许发生死锁，通过检测机构及时发现并解除 |
| 解除死锁 | Deadlock Recovery | 检测到死锁后采取措施，将进程从死锁状态中解脱出来 |

## 5. 死锁预防

### 5.1 破坏"请求和保持"条件（部分分配）

**静态资源分配法（资源预分配法）**：

所有进程在开始运行之前，必须一次性地申请其在整个运行过程中所需的全部资源。若能满足则全部分配，否则一个也不分配，进程等待。

- 优点：简单、易行且安全
- 缺点：资源利用率低；对很多应用，一次提出所有资源申请困难

**实例：哲学家就餐问题**

```c
// 原方案（可能死锁）：分别拿两根筷子
wait(s[i]);           // 拿左边筷子
wait(s[(i+1)%5]);     // 拿右边筷子
// ... 吃饭 ...
signal(s[i]);
signal(s[(i+1)%5]);

// 静态分配方案（无死锁）：同时申请两根筷子
swait(s[i], s[(i+1)%5]);   // 同时拿起左右两根筷子
// ... 吃饭 ...
ssignal(s[i], s[(i+1)%5]); // 同时放下
```

### 5.2 破坏"不可抢占"条件

**请求未果则释放已有资源**：

已保持某些不可被抢占资源的进程，提出新的资源请求而不能得到满足时，必须释放已保持的所有资源，待以后需要时再重新申请。

- 缺点：程序复杂，代价高，进程执行被无限推迟

### 5.3 破坏"循环等待"条件

**有序资源分配法**：

系统将所有资源按类型进行线性排序并赋予不同序号，所有进程对资源的请求必须严格按资源序号**递增**的次序提出。

资源集：R = {r1, r2, ..., rn}，函数 F: R → N

规则：进程 pi 可以申请资源 rj 的条件是：对于 pi 已占有的所有 rk，F(rk) < F(rj)

**证明无死锁**（反证法）：
- 假定死锁，则 p1 等待 rk1（被 p2 占有），p2 等待 rk2（被 p3 占有），...，pn 等待 rkn（被 p1 占有）
- 由有序申请假设：F(rk1) < F(rk2) < ... < F(rkn) < F(rk1)
- 矛盾，故不可能死锁

**实例：哲学家就餐问题**

```c
// 有序资源分配：编号小的先拿
if (i ≠ 4) {
    wait(s[i]);      // 先拿编号小的筷子
    wait(s[i+1]);    // 再拿编号大的筷子
} else {
    wait(s[0]);      // 哲学家4先拿筷子0（编号小）
    wait(s[4]);      // 再拿筷子4（编号大）
}
// ... 吃饭 ...
signal(s[i]);
signal(s[(i+1)%5]);
```

### 5.4 破坏"互斥"条件

将互斥资源改造成共享资源。

**实例**：Spooling 系统（Simultaneous Peripheral Operations On-Line，外部设备联机并行操作），用文件代替独享设备。

虚拟概念的广泛应用：虚拟设备、虚拟 CPU、虚拟地址、虚拟存储器、虚拟终端、虚拟机、虚拟现实。

## 6. 死锁避免——银行家算法

### 6.1 安全状态概念

**安全状态**：如果系统中存在一种执行路径，使可用资源能使所有进程的资源请求得到满足从而顺利结束，称系统处于安全状态。这个执行路径称为**安全序列**。

**安全序列定义**：一个进程序列 (P1, ..., Pn) 是安全的，如果对于每个 Pi (1≤i≤n)，它尚需要的资源量不超过系统当前剩余资源量与所有 Pj (j < i) 当前占有资源量之和。

**重要性质**：
- 处于安全状态的系统**一定没有**发生死锁
- 不安全状态**不一定**导致死锁，但有死锁风险

### 6.2 安全状态示例

系统：3 个进程 P1/P2/P3，共 12 台磁带机。

| 进程 | 最大需求 | 已分配 | 尚需 | 可用 |
|------|----------|--------|------|------|
| P1 | 10 | 5 | 5 | 3 |
| P2 | 4 | 2 | 2 | - |
| P3 | 9 | 2 | 7 | - |

安全序列：P2 → P1 → P3
- P2 需 2 ≤ 可用 3，P2 完成后释放 4，可用 = 3+2 = 5
- P1 需 5 ≤ 可用 5，P1 完成后释放 5，可用 = 5+5 = 10
- P3 需 7 ≤ 可用 10，P3 完成

**由安全到不安全的转换**：

若 T0 后 P3 请求 1 台磁带机，系统分配后：
- P3 已分配 3，尚需 6，可用 2
- 此时无法找到安全序列（P1 需 5 > 2，P2 需 2 = 2 但完成后释放 4 < 5 或 6），系统进入不安全状态

### 6.3 银行家算法数据结构

```
Available[1..m]     // 系统可用资源向量
Max[1..n][1..m]     // 进程最大需求矩阵
Allocation[1..n][1..m]  // 当前分配矩阵
Need[1..n][1..m]    // 尚需资源矩阵（Need = Max - Allocation）
Request[1..n][1..m] // 当前请求向量
Work[1..m]          // 临时工作向量
Finish[1..n]        // 临时完成标记
```

### 6.4 资源分配算法

当进程 Pi 提出资源请求 Request[i] 时：

**步骤 1**：分配能力检查
- 若 Request[i] ≤ Need[i]，转步骤 2；否则报错（请求超过声明的最大值）

**步骤 2**：可用性检查
- 若 Request[i] ≤ Available，转步骤 3；否则 Pi 等待

**步骤 3**：预分配（试探性分配）
```
Available  = Available - Request[i]
Allocation[i] = Allocation[i] + Request[i]
Need[i]    = Need[i] - Request[i]
```

**步骤 4**：安全性检测
- 调用安全性算法判断分配后系统是否安全
- 安全 → 真正实施分配；不安全 → 取消预分配，Pi 等待

### 6.5 安全性检测算法

**步骤 1**：设置向量
- Work = Available（工作向量）
- Finish[i] = false（对所有 i）

**步骤 2**：找到满足条件的进程
- 条件：Finish[i] = false 且 Need[i] ≤ Work

**步骤 3**：若找到，执行
```
Work = Work + Allocation[i]
Finish[i] = true
回到步骤 2
```

**步骤 4**：判断
- 若所有 Finish[i] = true，系统处于安全状态
- 否则，系统处于不安全状态

### 6.6 银行家算法完整示例

**初始状态**：资源 R = {A(10), B(5), C(7)}，进程 P0-P4

| 进程 | Max | Allocation | Need | Available |
|------|-----|------------|------|-----------|
| P0 | (7,5,3) | (0,1,0) | (7,4,3) | (3,3,2) |
| P1 | (3,2,2) | (2,0,0) | (1,2,2) | |
| P2 | (9,0,2) | (3,0,2) | (6,0,0) | |
| P3 | (2,2,2) | (2,1,1) | (0,1,1) | |
| P4 | (4,3,3) | (0,0,2) | (4,3,1) | |

**(1) T0 时刻安全性检测**：

安全序列：<P1, P3, P4, P2, P0>

| 步骤 | 进程 | Work | Need ≤ Work? | Work + Allocation | Finish |
|------|------|------|--------------|-------------------|--------|
| 1 | P1 | (3,3,2) | (1,2,2)≤(3,3,2) ✓ | (5,3,2) | true |
| 2 | P3 | (5,3,2) | (0,1,1)≤(5,3,2) ✓ | (7,4,3) | true |
| 3 | P4 | (7,4,3) | (4,3,1)≤(7,4,3) ✓ | (7,4,5) | true |
| 4 | P2 | (7,4,5) | (6,0,0)≤(7,4,5) ✓ | (10,4,7) | true |
| 5 | P0 | (10,4,7) | (7,4,3)≤(10,4,7) ✓ | (10,5,7) | true |

系统安全。

**(2) P1 请求资源 Request(1,0,2)**：

① 检查：Request(1,0,2) ≤ Need(1,2,2) ✓
② 检查：Request(1,0,2) ≤ Available(3,3,2) ✓
③ 预分配：Available = (2,3,0), Allocation1 = (3,0,2), Need1 = (0,2,0)
④ 安全性检测：找到安全序列，系统安全，真正分配。

**(3) P4 请求资源 Request(3,3,0)**：

① 检查：Request(3,3,0) ≤ Need(4,3,1) ✓
② 检查：Request(3,3,0) > Available(2,3,0) → **P4 等待**

**(4) P0 请求资源 Request(0,2,0)**：

① 检查：Request(0,2,0) ≤ Need(7,4,3) ✓
② 检查：Request(0,2,0) ≤ Available(2,3,0) ✓
③ 预分配：Available = (2,1,0)
④ 安全性检测：Available(2,1,0) 不能满足任何进程的 Need → **不安全**，取消预分配，P0 等待。

## 7. 死锁的检测与解除

### 7.1 死锁检测

允许死锁发生，操作系统不断监视系统进展，判断死锁是否发生。一旦发生则采取措施解除。

**资源分配图**：
- 进程结点 P = {p1, ..., pn}，用圆圈表示
- 资源结点 R = {r1, ..., rm}，用方框表示
- 资源请求边：进程 → 资源（表示进程请求一个单位的该资源）
- 资源分配边：资源 → 进程（表示将一个单位的资源分配给进程）

**死锁定理**：

利用资源分配图简化方法检测死锁：
1. 找出一个既不阻塞又非独立的进程结点 Pi
2. Pi 可获得所需资源而继续运行，直至完成，释放全部资源——消去其请求边和分配边
3. 重复直到所有边都被消去（可完全简化）→ 无死锁；不能完全简化 → 死锁

**充分条件**：当且仅当状态 S 的资源分配图是不可完全简化时，S 为死锁状态。

### 7.2 死锁检测的数据结构

类似于银行家算法：
```
Work = Available
L = {Li | Allocation[i] = 0 且 Request[i] = 0}
for all Li in L do
    for all Request[i] ≤ Work do
        Work = Work + Allocation[i]
        Li 加入 L
    end
end
deadlock = (L ≠ {p1, p2, ..., pn})
```

### 7.3 死锁的解除

当发现死锁后，常采用的方法：

**方法一：剥夺资源**
- 选择一个与死锁相关的进程，将进程恢复到某个时间点的状态
- 该点之后新申请的资源由系统收回
- 时间点选择准则：系统能恢复到安全状态
- 缺点：需要记录进程之前的状态，增加系统复杂性

**方法二：撤消进程**
- 简单方法：终止所有死锁进程（代价大）
- 优化方法：每次撤消一个代价最小的进程，直到死锁消除（实现复杂）

### 7.4 相关概念

- **活锁（Livelock）**：进程不断改变状态以响应对方，但都无法继续执行（如两人在走廊相遇反复让路）
- **饥饿（Starvation）**：进程长时间得不到所需资源（不一定由死锁引起）

## 8. 总结复盘

### 8.1 死锁处理方法对比

| 方法 | 策略 | 优点 | 缺点 |
|------|------|------|------|
| 预防 | 破坏必要条件 | 简单、可靠 | 限制多，资源利用率低 |
| 避免 | 银行家算法 | 不需限制，利用率较高 | 需预知最大需求，开销大 |
| 检测+解除 | 允许发生再处理 | 灵活，无限制 | 检测开销，恢复代价大 |

### 8.2 四个必要条件速记口诀

**互斥-保持-不可抢-循环**：
- **互斥**：资源独占
- **保持**：持有旧的，请求新的
- **不可抢**：到手的不能被抢
- **循环**：你等我、我等你

### 8.3 核心要点

- 死锁定义：两个以上进程相互永久等待对方持有的资源
- 四个必要条件缺一不可，破坏任一即可预防死锁
- 安全状态一定无死锁，不安全状态不一定死锁但有风险
- 银行家算法核心：分配前先做安全性检查
- 安全序列的构造是银行家算法的关键步骤
- 资源分配图简化法是死锁检测的理论基础
- 不发生死锁条件：m ≥ n(k-1) + 1

- 掌握银行家算法的完整计算步骤
- 能够对资源分配图进行简化分析
- 理解预防/避免/检测三种策略的区别和适用场景
- 注意区分死锁、活锁、饥饿三个概念
