# 第13讲 I/O系统

## 1. 概念引入

计算机有两个主要任务：**计算**和**I/O操作**。I/O操作更加频繁，是系统中最常运行的部分。

I/O设备的特点：
- 种类繁多，不断有新设备出现
- 接口趋向标准化（如USB）
- 如何适应I/O设备的多样性是操作系统的一大挑战

**I/O系统任务**：I/O系统受输入/输出速率限制，经常成为系统性能瓶颈（如磁盘、串口）。操作系统需要：
- 有效管理各种输入/输出设备
- 提升设备运行效率
- 提供方便应用程序使用设备的标准编程接口，通过设备驱动程序（Device Driver）操作不同类型设备

**本讲内容**：I/O硬件系统、I/O编程接口、设备驱动程序结构与硬件操作流程、I/O控制方式、I/O缓冲技术、磁盘调度。

## 2. I/O硬件系统

### 2.1 I/O硬件构成

I/O硬件由三部分组成：
- **I/O设备**：打印机、键盘、鼠标、显示器、硬盘、U盘等
- **I/O控制器**（I/O接口/I/O模块）：UART串口、并口、I2C接口、USB接口、ADC接口、GPIO接口等
- **I/O总线**：PCI、USB、ATA、ISA、I2C等

硬件层次结构（自上而下）：
1. CPU通过**北桥芯片**与内存、I/O总线连接
2. **I/O总线**通过**南桥芯片**或扩展卡连接**I/O控制器**
3. I/O控制器通过**通信总线**连接**I/O设备**
4. 还包括：中断机制、DMA、通道、I/O处理器

### 2.2 I/O系统总线

系统总线上传输的信息包括：
- 数据（指令、操作数、中断号）
- 地址
- 其他控制/状态/定时信号

### 2.3 I/O设备分类

**按使用特性分类：**
- **存储设备**（外存/辅存）：存取速度较内存慢，但容量大、价格低
- **I/O设备**：输入设备、输出设备、交互式设备

**按传输速率分类：**
- 低速设备：每秒几个字节至数百字节（如键盘、鼠标）
- 中速设备：每秒数千字节至数十万字节（如行式打印机、激光打印机）
- 高速设备：每秒数十万字节至千兆字节（如磁盘机、光盘机、网卡）

**按传输特性分类：**
- 字符设备（Character Device）：以字节为单位传输（如键盘、鼠标）
- 块设备（Block Device）：以数据块为单位传输（如硬盘、CDROM）
- 网络设备（Network Device）

### 2.4 设备控制器结构及I/O编程

**设备控制器的一般结构：**
- 通过**命令字**向设备发送命令（写入I/O控制寄存器）
- 通过**状态字**获取设备状态（读取状态寄存器）
- 通过**数据交换**与外设传输数据
- CPU能访问的各类寄存器称为**I/O端口（I/O Port）**

设备控制器中一般有三类寄存器：
- **状态寄存器（Status Register）**：获知前一操作是否完成、是否出错、输入数据是否就绪
- **控制寄存器（Control Register）**：写入命令字，初始化装置或改变操作模式/参数
- **数据寄存器（Data Register）**：读取输入数据或写入输出数据

**I/O口编址方法：**

| 编址方式 | 说明 | 示例 |
|---------|------|------|
| 独立I/O空间寻址 | I/O端口地址与存储器地址独立，可重叠 | x86 CPU（64K地址空间，00000~0FFFFH） |
| 存储器映射I/O寻址 | I/O端口与存储器共用一个地址空间，设备寄存器如同存储器单元 | ARM、MIPS |
| 混合方式 | 传统设备用独立I/O空间，新型I/O控制器用存储器映射 | 现代PC |

**I/O编程示例：**

x86独立编址（in/out指令）：
```asm
IN  AL, 3F8H    ; 从串口寄存器读取一个字节
OUT AL, 378H    ; 将AL内容写入并口寄存器
```

ARM存储器映射I/O（C语言指针运算）：
```c
int *p = (int *)0x58000000;
*p = *p | 2;    // 启动A/D转换
```

## 3. I/O系统层次结构和模型

### 3.1 I/O系统层次结构

I/O软件采用**分层结构设计**，自上而下：

1. **用户层I/O软件**：实现与用户交互的接口，用户可直接调用与I/O操作有关的库函数
2. **设备独立性软件**（设备无关软件）：统一接口、设备命名、保护、分配与释放，提供必要存储空间
3. **设备驱动程序**：与硬件直接相关，具体实现系统对设备的操作指令
4. **中断处理程序**：处理设备产生的中断（数据传输完成、数据到达、设备出错等），恢复被中断进程现场后返回

### 3.2 I/O系统构架与编程接口

**I/O子系统**：由输入/输出函数库、音频视频函数库、图形库、缓冲、I/O调度、错误处理、格式转换等与设备无关功能构成，多数模块位于OS内核。

**I/O应用程序编程接口**：根据设备特征归纳出的标准化API：
- 字符设备接口（流设备接口）：用设备文件表示，如串口`/dev/ttyS0`，用open/read/write/close操作
- 块设备接口：提供open/read/write/close/seek等接口，允许原始I/O（raw I/O）
- 网络通信接口：如send/recv等API函数

**两个关键接口：**
- 内核与设备驱动程序接口：一组满足特定命名规范的函数
- 设备驱动程序与设备硬件接口：I/O端口、设备寄存器描述、中断机制、DMA机制

## 4. I/O子系统

I/O子系统完成与设备物理特性无关的功能：

### 4.1 设备管理与分配

**设备管理**：对各种I/O设备、驱动程序进行登记造册（使用表格、链表、结构体）。

Linux字符设备管理：使用`device_struct`数组管理系统中所有字符设备：
```c
struct device_struct {
    const char *name;
    struct file_operations *fops;
};
static struct device_struct chrdevs[MAX_CHRDEV];
```

**物理设备、软设备与逻辑设备：**
- **软设备**：无对应硬件，纯软件实现（如`/dev/null`空设备、`/dev/zero`零设备）
- **逻辑设备**：物理设备的一部分或多个设备的组合（如`/dev/sda`主硬盘为物理设备，`/dev/sda1`、`/dev/sda2`为分区逻辑设备）

**设备名与驱动映射**：通过**主设备号**（内核设备数组下标）和**次设备号**（同类设备中的序号）实现。

**设备分配：**
- **独占设备**（串口、打印机）：互斥使用，上锁方式打开
- **共享设备**（显示器、硬盘、网卡）：多进程同时使用
  - 显示器：内核分配虚拟窗口
  - 硬盘：请求排队处理
  - 网卡：操作系统直接操作，封装帧发送

### 4.2 缓冲技术（Buffering）

**缓冲技术的引入：**
- 最早为解决CPU与I/O设备**速度不匹配**矛盾
- 凡是数据到达和离去速度不匹配的地方均可采用

**缓冲技术在I/O系统中的作用：**
1. 缓解CPU与I/O设备速度不匹配的矛盾
2. 提高CPU与I/O设备之间的**并行性**
3. 减少I/O设备对CPU的**中断请求次数**，放宽CPU对中断响应时间的要求

**缓冲区设置：**
- **硬缓冲**：由硬件寄存器实现（如设备中的缓冲区）
- **软缓冲**：在内存中开辟空间作为缓冲区

**缓冲区管理方式：**

| 管理方式 | 说明 |
|---------|------|
| **单缓冲** | 每次I/O请求分配一个缓冲区。处理时间：Max(C, T) + M（C为CPU处理时间，T为设备传输时间，M为内存复制时间） |
| **双缓冲**（缓冲对换） | 数据先送第一缓冲区，满后转第二缓冲区，同时CPU处理第一缓冲区数据。处理时间：Max(C, M, T) |
| **环形缓冲区** | 多个缓冲区+多个指针（Nextg、Nexti、Current），分为输入数据空缓冲区R、已满数据缓冲区G、现行工作缓冲区C |
| **缓冲池（Buffer Pool）** | 公用缓冲池，多个进程共享。包含空缓冲队列emq、输入队列inq、输出队列outq，以及收容/提取输入/输出数据的工作缓冲区 |

### 4.3 SPOOLing系统（假脱机技术）

SPOOLing（Simultaneous Peripheral Operation On-Line，外部设备联机并行操作）：
- 由**输入井和输出井**、**输入缓冲和输出缓冲**、**输入进程SPi和输出进程Spo**组成
- 进程输出时，数据以文件形式存入磁盘输出井，形成输出队列
- 由输出SPOOLing系统控制打印机依次打印
- **核心思想**：把独占设备改造为共享设备，提高系统效率

### 4.4 错误处理

- **暂时性错误**：OS内保护机制可修复（如磁盘读取失败→重读；网络传输失败→重传）
- **永久性错误**：OS无法修复，只能报告错误信息给应用程序（如Linux通过变量`error`获知最近错误）

## 5. 设备驱动程序

**设备驱动程序功能：**
- 设备控制与数据输入输出
- 与操作系统内核提供标准化接口函数（屏蔽设备硬件特性）
- 处理外部事件（中断处理程序模块）

### 5.1 设备驱动程序工作流程

1. 接收设备无关软件发来的命令和参数，将抽象要求转换为与设备相关的低层操作序列
2. 检查用户I/O请求的合法性，了解设备状态，传递参数，设置工作方式
3. 发出I/O命令：设备空闲则立即启动；设备忙碌则将请求挂在设备队列等待
4. 及时响应设备控制器发来的中断请求，执行中断处理程序

### 5.2 设备驱动程序分类

- **字符设备驱动程序**：为每个字符设备创建设备节点，提供类似文件操作的函数
- **块设备驱动程序**：提供接口函数让内核创建文件系统
- **网卡驱动程序**：上层为网络协议栈IP层，提供成帧、帧发送、帧接收功能
- **总线驱动程序**（如USB总线）：实现设备加入/移走侦测、配置，以及总线数据帧收发

### 5.3 设备驱动程序接口规范（Linux字符设备）

字符设备驱动程序提供标准函数：
- `open`：设备打开
- `read`：数据输入
- `write`：数据输出
- `ioctl`：设备控制
- `release`：设备关闭

系统I/O函数与驱动函数的映射关系：
```
open  → s3c2410_adc_open
close → s3c2410_adc_release
read  → s3c2410_adc_read
write → s3c2410_adc_write
ioctl → s3c2410_adc_ioctl
```

## 6. I/O控制方式

I/O控制方式有四种，性能逐步提升：

### 6.1 程序直接控制方式（Programmed I/O）

**基本思想：** I/O设备将自己的状态放到状态寄存器中，OS不断查询状态寄存器以决定下一步动作。

**控制流程：**
1. OS发命令字启动I/O操作
2. OS反复读取控制器状态（**轮询/Polling**）
3. 若状态为就绪，则读走数据或写入下一个数据

**示例（字符串打印）：**
```c
copy_string_to_kernel(strbuf, kernelbuf, n);
for (i = 0; i < n; i++) {
    while (printer_status != READY);    // 等待就绪
    *printer_data_port = kernelbuf[i];  // 输出字符
    *printer_control_port = START;      // 启动打印
}
return_to_user();
```

**特点：**
- **优点**：简单、易控制、外围接口控制逻辑少
- **缺点**：CPU与外设串行工作，效率低、速度慢；CPU大部分时间在做无意义查询（**忙等待/Busy Waiting**），是极大浪费
- 适合慢速设备

### 6.2 中断控制I/O方式（Interrupt-driven I/O）

**基本思想：** CPU发送控制命令启动I/O后，转去执行其他工作；外设就绪或完成时向CPU发中断请求，CPU响应中断执行数据传送，然后返回断点。实现了**CPU与外设并行工作**。

**中断处理过程（详细）：**

1. 进程要输入数据时，向设备命令寄存器发I/O命令
2. 进程**阻塞**自己，CPU分配给其他进程
3. I/O设备按命令执行输入/输出操作（CPU与I/O并行）
4. 操作完成，数据进入设备数据寄存器，I/O设备向CPU发**中断信号**
5. CPU**响应中断**，执行**中断处理程序**，唤醒等待进程
6. 被唤醒的进程从设备数据寄存器读取输入数据

**关键细节：**
- 程序切换（响应中断）由硬件完成，执行"**中断隐指令**"
- 中断处理完成后需**中断回答（acknowledge）**清除中断请求
- 由于进程避免了循环等待，中断驱动方式可**成百倍提高CPU利用率**

**示例（字符串打印）：**
```c
// sys_write系统调用服务例程
copy_string_to_kernel(strbuf, kernelbuf, n);
enable_interrupts();              // 开中断
while (printer_status != READY);  // 等待就绪
*printer_data_port = kernbuf[i];  // 输出第一个字符
*printer_control_port = START;    // 启动打印
scheduler();                      // 阻塞进程P，调度其他进程

// 中断服务程序
if (n == 0) {
    unblock_user();               // 打印完，解除阻塞
} else {
    *printer_data_port = kernelbuf[i];  // 输出下一个字符
    *printer_control_port = START;
    n = n - 1;
    i = i + 1;
}
acknowledge_interrupt();          // 中断回答
return_from_interrupt();          // 中断返回
```

### 6.3 直接存储器访问方式（DMA, Direct Memory Access）

**引入原因：** 中断方式以字（节）为单位干预，数据传输率仍然很低。DMA进一步减少CPU对I/O的干预。

**DMA基本思想：**
- 由专门硬件（**DMA控制器**）控制总线进行传输
- 数据传输的基本单位是**数据块**
- 仅在传送一个或多个数据块的**开始和结束时**，才通过中断请求CPU干预

**DMA控制器结构：**
- 主机（存储器）与DMA控制器的接口
- DMA控制器与块设备的接口
- I/O控制逻辑

**DMA寄存器：**
- **命令/状态寄存器CR**：存放读写命令
- **内存地址寄存器MAR**：存放内存起始地址
- **数据寄存器DR**：暂存传输数据
- **数据计数器DC**：记录待传输字节数

**DMA工作过程（以读数据为例）：**
1. CPU向DMA控制器写入：CR（读磁盘命令）、MAR（内存目标地址）、DC（字节数）、磁盘块地址
2. CPU发送"启动DMA传送"命令
3. DMA自动传输数据，每传输一个字：MAR←MAR+1，DC←DC-1
4. 传输完毕（DC=0），DMA控制器向CPU发送**DMA结束中断请求**

**常用技术——周期挪用法（Cycle Stealing）：** 设备每准备好一个数据，挪用一次存储周期，使用一次总线事务进行数据传送。

**示例（字符串输出）：**
```c
// sys_write系统调用服务例程
copy_string_to_kernel(strbuf, kernelbuf, n);
initialize_DMA();              // 初始化DMA控制器
*DMA_control_port = START;     // 启动DMA传送
scheduler();                   // 阻塞进程P

// DMA结束中断服务程序
acknowledge_interrupt();       // 中断回答
unblock_user();                // 解除阻塞
return_from_interrupt();       // 中断返回
```

**CPU开销极小：** 仅在DMA初始化和处理"DMA结束中断"时介入，传送过程中不参与。

### 6.4 I/O控制方式对比

| 特性 | 程序查询 | 中断方式 | DMA方式 | 通道方式 |
|------|---------|---------|---------|---------|
| 干预单位 | 字（节） | 字（节） | 数据块 | 一组数据块 |
| CPU介入频率 | 持续查询 | 每个字一次中断 | 仅首尾中断 | 通道完成全部工作 |
| CPU利用率 | 最低（忙等待） | 较高 | 很高 | 最高 |
| 并行性 | 无（串行） | CPU与外设并行 | CPU与外设并行 | CPU、通道、外设三方并行 |
| 适用场景 | 简单慢速设备 | 中速设备 | 高速设备（磁盘等） | 大量I/O操作 |
| 硬件复杂度 | 最简单 | 中等 | 较复杂（DMA控制器） | 最复杂（通道处理器） |

**计算示例：中断 vs DMA的CPU开销**

条件：处理器500MHz，磁盘控制器16B缓存，磁盘传输率4MB/s。

- **中断方式**：每次中断开销500个时钟周期，硬盘仅5%时间传送
  - 每秒中断次数：4MB / 16B = 250K次
  - 每秒中断时钟数：250K × 500 = 125×10⁶
  - CPU占用率：125×10⁶ / (500×10⁶) = 25%
  - 实际（仅5%时间传送）：25% × 5% = **1.25%**

- **DMA方式**：初始化1000时钟 + 结束中断500时钟，每次传送8000B
  - 每秒DMA次数：4MB / 8000B = 500次
  - 每秒时钟数：(1000+500) × 500 = 750×10³
  - CPU占用率：750×10³ / (500×10⁶) = **0.15%**

## 7. 总结复盘

**I/O系统核心要点：**
1. **I/O硬件**由设备、控制器、总线三层组成，设备分为字符设备、块设备、网络设备
2. **I/O编程**通过读写设备控制器中的寄存器实现，有独立编址和存储器映射两种方式
3. **I/O软件层次**：用户层→设备独立性软件→设备驱动程序→中断处理程序，各层分工明确
4. **I/O子系统**负责设备管理、缓冲、SPOOLing、错误处理、I/O调度等功能
5. **四种I/O控制方式**性能递增：程序查询（忙等待）→ 中断（CPU并行）→ DMA（数据块级）→ 通道（处理器级）
6. **缓冲技术**缓解速度矛盾、提高并行性、减少中断次数，有单缓冲/双缓冲/环形缓冲/缓冲池等形式
7. **SPOOLing**将独占设备改造为共享设备
8. **设备驱动程序**是内核与硬件之间的桥梁，提供标准接口函数

**关键计算：** 磁盘地址转换（柱面号/磁头号/扇区号与盘块号的互转）、磁盘访问时间（寻道时间+旋转延迟+传输时间）、缓冲区处理时间计算。

# 第14讲 程序代码优化

## 1. 概念引入

程序代码优化关注**机器无关优化（Machine-Independent Optimization）**，即不依赖特定硬件架构的通用优化方法。

**为什么需要优化：**
- 常量因子（Constant Factors）也很重要：代码编写方式不同，性能差异可达**10:1**
- 必须在多个层次进行优化：算法、数据表示、过程、循环
- 需要理解系统如何编译和执行程序，如何衡量性能并识别瓶颈
- 在不破坏代码模块性和通用性的前提下提高性能

**优化器（Optimizer）的核心目标：** 将程序高效映射到机器代码，包括寄存器分配、代码选择与排序、消除低效之处。

---

## 2. 性能度量：CPE

**CPE（Cycles Per Element，每元素时钟周期数）** 是衡量向量/列表操作程序性能的便捷方式。

对于长度为 n 的向量操作：
```
T = CPE × n + Overhead
```

典型时钟周期参考：
- 100MHz → 时钟周期 10ns
- 2GHz → 时钟周期 0.5ns

---

## 3. 代码移动（Code Motion）

**核心思想：** 将循环中**不随迭代变化的计算**移到循环外部。

**示例1：向量组合函数**

`combine1`（优化前）—— 每次循环都调用`vec_length()`：
```c
void combine1(vec_ptr v, data_t *dest) {
    long int i;
    *dest = IDENT;
    for (i = 0; i < vec_length(v); i++) {  // vec_length每次迭代都调用
        data_t val;
        get_vec_element(v, i, &val);
        *dest = *dest OP val;
    }
}
```

`combine2`（优化后）—— `vec_length()`移到循环外：
```c
void combine2(vec_ptr v, data_t *dest) {
    long int i;
    long int length = vec_length(v);  // 只调用一次
    *dest = IDENT;
    for (i = 0; i < length; i++) {
        data_t val;
        get_vec_element(v, i, &val);
        *dest = *dest OP val;
    }
}
```

**示例2：字符串转小写**

`lower1`（慢）—— 每次循环调用`strlen()`：
```c
void lower1(char *s) {
    int i;
    for (i = 0; i < strlen(s); i++)   // strlen每次迭代都调用
        if (s[i] >= 'A' && s[i] <= 'Z')
            s[i] -= ('A' - 'a');
}
```

`lower2`（快）—— `strlen()`移到循环外：
```c
void lower2(char *s) {
    int i;
    int len = strlen(s);             // 只调用一次
    for (i = 0; i < len; i++)
        if (s[i] >= 'A' && s[i] <= 'Z')
            s[i] -= ('A' - 'a');
}
```

`strlen()`本身是一个O(n)的遍历操作，嵌入循环后复杂度升至O(n²)，移出后降回O(n)。

---

## 4. 强度削减（Reduction in Strength）

**核心思想：** 用**更廉价的操作**替代昂贵的操作（如用指针引用替代函数调用）。

`combine3`（优化后）—— 直接用指针访问数组，避免`get_vec_element`函数调用：
```c
void combine3(vec_ptr v, data_t *dest) {
    long int i;
    long int length = vec_length(v);
    data_t *data = get_vec_start(v);   // 获取数组首地址
    *dest = IDENT;
    for (i = 0; i < length; i++) {
        *dest = *dest OP data[i];      // 直接指针引用，无函数调用
    }
}
```

**代价：** 牺牲了数据抽象性（不再做边界检查）。

---

## 5. 消除不必要的内存引用

**核心思想：** 使用局部变量（寄存器）累加，避免每次循环都进行内存读写。

`combine3`的问题：每次循环执行3条内存操作（读dest、写dest、读data[i]）：
```asm
.L498:
    movss  (%rbp), %xmm0       # 从dest读取累加值
    mulss  (%rax,%rdx,4), %xmm0 # 乘以data[i]
    movss  %xmm0, (%rbp)       # 写回dest
    addq   $1, %rdx
    cmpq   %rdx, %r12
    jg     .L498
```

`combine4`（优化后）—— 使用局部变量`acc`在寄存器中累加：
```c
void combine4(vec_ptr v, data_t *dest) {
    long int i;
    long int length = vec_length(v);
    data_t *data = get_vec_start(v);
    data_t acc = IDENT;                // 局部变量，驻留寄存器
    for (i = 0; i < length; i++)
        acc = acc OP data[i];          // 寄存器操作，无内存读写
    *dest = acc;                        // 循环结束后才写内存
}
```

```asm
.L488:
    mulss  (%rax,%rdx,4), %xmm0   # 乘以data[i]
    addq   $1, %rdx
    cmpq   %rdx, %rbp
    jg     .L488
```

**效果：** 每次循环减少1次内存读和1次内存写，性能显著提升。

## 6. 优化障碍（Optimizing Blockers）

优化编译器在某些情况下无法进行优化，主要障碍有两类：

### 6.1 内存别名（Memory Aliasing）

**问题：** 两个不同的内存引用指向同一位置时，编译器无法安全优化。

```c
void twiddle1(int *xp, int *yp) {
    *xp += *yp;
    *xp += *yp;    // 若xp == yp，此语句与上句重复
}

void twiddle2(int *xp, int *yp) {
    *xp += 2 * *yp; // 仅一次写入
}
```

当xp == yp时，`twiddle1`的结果是 `*xp = 4*(*xp)`，而`twiddle2`的结果是 `*xp = 3*(*xp)`，两者不同！

**C语言中的别名示例：**
```c
combine3(v, get_vec_start(v)+2)  // dest与data可能指向同一位置
```

**应对策略：** 养成引入局部变量在循环中累加的习惯——这是告诉编译器"不用检查别名"的方式。

### 6.2 函数调用的副作用（Side Effect in Function Call）

```c
int func1(x) { return f(x) + f(x) + f(x) + f(x); }
int func2(x) { return 4 * f(x); }
```

若`f`有副作用（如修改全局变量），则`func1`和`func2`不等价。编译器不能将`func1`优化为`func2`。

```c
int counter = 0;
int f(int x) { return counter++; }  // 副作用：修改全局变量
```

### 6.3 优化编译器的局限性

- 编译器必须保证**在任何可能条件下不改变程序行为**，因此常采取保守策略
- 程序员显然了解的约束（如数据范围），编译器可能无法推断
- 例如C语言中用`int`表示本可用枚举类型表达的数据范围

---

## 7. 总结复盘

**程序代码优化核心要点：**

1. **代码移动（Code Motion）**：将不随循环迭代变化的计算（如`vec_length()`、`strlen()`）移到循环外部，是最基本也最有效的优化
2. **强度削减（Reduction in Strength）**：用指针直接引用替代函数调用（如`get_vec_element`），以牺牲抽象性换取性能
3. **消除不必要的内存引用**：用局部变量（寄存器）替代内存读写，减少每周期的内存操作次数
4. **优化障碍**：
   - **内存别名**：两个指针指向同一位置时，编译器无法安全优化
   - **函数副作用**：函数修改全局状态时，编译器无法合并/消除调用
5. **最佳实践**：在循环中使用局部变量累加，既避免别名问题又提升性能
6. **优化层次**：应在算法、数据表示、过程、循环等多个层次综合优化，常量因子与渐近复杂度同样重要

**优化效果示例（向量累加 combine系列）：**
- combine1 → combine2（代码移动）：消除循环内的重复函数调用
- combine2 → combine3（强度削减）：消除元素访问的函数调用开销
- combine3 → combine4（消除内存引用）：每周期减少1次内存读+1次内存写
