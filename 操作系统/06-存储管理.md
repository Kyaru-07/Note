# 第12讲 存储管理

## 一、存储管理概念与程序装入

### 1. 概念引入

#### 存储管理功能
操作系统对内存的管理需要实现五大功能：
- **内存分配与回收**：为每个进程分配一定数量的内存空间
- **地址转换**：将程序地址（逻辑地址）转换为物理地址
- **内存信息共享**：实现进程间有条件地共享某些内存区域
- **内存信息保护**：保证各进程在各自的内存空间中活动，不能相互干扰，更不能侵犯操作系统空间
- **内存扩充**：实现小内存运行大程序的要求

#### 为什么要设置多级存储体系？

计算机一般需设置多级存储体系，自上而下包括：

| 存储层次 | 组织方式 | 速度 | 容量 | 价格 |
|---------|---------|------|------|------|
| 寄存器（Register） | SRAM | 最快 | 最小 | 最贵 |
| 高速缓存（Cache） | SRAM | 很快 | 较小 | 较贵 |
| 主存储器（Main Memory） | DRAM | 快 | 中等 | 中等 |
| Flash闪存 | 闪存 | 较快 | 较大 | 较便宜 |
| 固定磁盘（Disk） | 磁盘 | 慢 | 大 | 便宜 |
| 可移动存储介质 | 磁盘/光盘 | 最慢 | 最大 | 最便宜 |

**设置多级存储体系的原因**：以较低的成本获得容量大、速度快的存储资源。

各级存储器由操作系统的不同模块进行管理：
- 寄存器和Cache由硬件直接管理
- 主存由操作系统内存管理模块管理
- 磁盘由文件系统和I/O管理系统管理

### 2. 核心知识点

#### 程序编辑、编译、装入过程

程序（可执行程序）平常驻留在外存，需要装入到内存才能执行。一个程序从源代码到最终执行需要经历以下步骤：

1. **编辑**：将程序源代码输入到文件中（如 `hello.c`）
2. **编译**：将源程序编译成汇编语言文件（`hello.s`），再汇编成目标代码文件（`hello.o`）
3. **链接（Linking）**：将编译后的一组目标模块及它们所需的库函数装配成一个完整的装入模块的过程（`hello.o` + `xx.dll` → `hello.exe`）
4. **装入运行（Loading）**：将程序装入到内存中，产生进程

#### 逻辑地址与物理地址

**逻辑地址（Logical Address）**：又称程序地址、虚拟地址（Virtual Address，VA），是用户能看到的程序代码、变量、堆栈的地址，按用户方便使用的方式安排（如从0开始编码）。用户地址空间中的地址是逻辑地址。

**物理地址（Physical Address）**：简称PA，是代码、变量、堆栈所在实际内存区域的实际地址，用户看不到。

一般来说，**虚拟地址 ≠ 物理地址**。将逻辑地址与物理地址分开的好处包括：实现进程隔离、提高内存利用率、实现进程和内核保护。

#### 地址空间

1. **虚拟地址空间（Virtual Address Space）**：程序地址（虚拟地址）可能的取值范围。由表示最大地址所需位数来决定，通常由通用寄存器宽度决定。
   - 例：32位处理器，虚拟地址空间大小为 $2^{32}$ = 4GB
   - 若用 v 位表示虚拟地址，则虚拟地址空间大小为 $2^v$ bytes

2. **物理地址空间**：计算机系统中实际存在的物理存储器单元的集合。
   - 若系统地址总线位数为 p 位，物理地址空间大小是 $2^p$ bytes

#### 内核空间与用户空间

在Linux中：
- **用户空间**：程序地址空间中安排给用户程序使用的部分（0 ~ 0xC0000000，共3GB）
- **内核空间**：程序地址空间中安排给操作系统使用的部分（0xC0000000 ~ 0xFFFFFFFF，共1GB）

**保护机制**：
- 页表中有一个状态位标识该页面为核心态页面还是用户态页面（如x86页表中U/S位）
- CPU的标志寄存器中有一个模式位，标识当前CPU所处的特权模式
- 当前CPU处于核心态才允许访问内核态页面，CPU处于用户模式时仅能访问用户态页面
- MMU对逻辑地址所属区域与CPU工作模式的检查，可阻止用户程序读写OS代码、数据、堆栈

#### 程序装入方式

1. **绝对装入方式**：编译时已知程序在内存中的位置，编译器直接生成绝对地址代码。适用于单道程序环境。
2. **可重定位装入方式（静态重定位）**：装入时将所有逻辑地址转换为物理地址，之后不再改变。地址变换在装入时一次完成。
3. **动态运行装入方式（动态重定位）**：程序在执行过程中才将逻辑地址转换为物理地址，需要硬件（重定位寄存器）支持。

#### 程序链接方式

1. **静态链接**：在程序运行前，将所有目标模块及库函数链接成一个完整的可执行文件。
2. **装入时动态链接**：边装入边链接，目标模块边装入内存边链接。
3. **运行时动态链接**：在程序执行过程中需要某个模块时才进行链接，便于实现模块共享。

### 3. 关键公式/原理

- 虚拟地址空间大小 = $2^v$ bytes（v为虚拟地址位数）
- 物理地址空间大小 = $2^p$ bytes（p为物理地址总线位数）
- 页面大小 = $2^k$ bytes（k为页内地址位数）

### 4. 计算示例

**地址转换计算基础**：已知虚拟地址空间大小是V，物理地址空间大小是M：
- 表示虚拟地址需要的位数 = $\log_2(V)$
- 物理地址总线位数 = $\log_2(M)$

### 5. 实际应用/实现

#### Linux环境下C语言程序编译链接过程

```
源代码(.c) → 预处理 → 编译(.s) → 汇编(.o) → 链接 → 可执行文件
```

Linux可执行文件结构（以objdump为例）包含：
- `.text`：程序代码段（READONLY, CODE）
- `.rodata`：只读数据段（READONLY, DATA）
- `.data`：全局变量段（有初值）
- `.bss`：全局变量（无初值）

### 6. 常见问题/易错点

- **局部变量**存储在栈（stack）中，不在全局变量段
- **动态申请（malloc）的存储器**在堆（heap）中分配
- 逻辑地址通常从0开始，一个程序段的逻辑地址是连续的
- 物理地址是连续的还是不连续的，取决于存储管理方式

### 7. 总结复盘

| 要点 | 说明 |
|-----|------|
| 多级存储体系 | 以低成本获得大容量、快速度的存储资源 |
| 逻辑地址 vs 物理地址 | 用户看到的是逻辑地址，实际访问的是物理地址 |
| 地址空间 | 虚拟地址空间由地址位数决定，物理地址空间由地址总线决定 |
| 程序装入 | 绝对装入、可重定位装入、动态运行装入 |
| 程序链接 | 静态链接、装入时动态链接、运行时动态链接 |
| 内核空间 vs 用户空间 | Linux中用户空间3GB，内核空间1GB |
| 保护机制 | MMU检查CPU模式位和页表U/S位，防止越权访问 |

## 二、分页式存储管理

### 1. 概念引入

#### 碎片问题与分页的引入

在连续分区存储管理中存在严重的碎片问题：
- 固定分区：内部碎片严重（分配给进程的空间中有未使用的部分）
- 动态分区：外部碎片问题大（大量小的空闲区无法利用）

碎片大量存在，经常出现"可用存储器总容量足够大"但"大作业不能运行"现象。内存利用率难以提高，有些作业序列可能使内存利用率低于10%。

**可变分区碎片问题的根本原因**：对可用内存地址连续的要求。若能打破这种限制，碎片问题可能迎刃而解。这就是**分页式存储管理**的设计思想。

#### 存储器地址模式

1. **实地址模式**：
   - 编程时使用的地址为逻辑地址，CPU执行时无需地址转换
   - 程序地址 = 物理地址
   - 特点：硬件简单，指令执行快
   - 问题：进程间不能隔离，不能解决内存容量不足

2. **保护地址模式**：
   - 编程时使用的地址为逻辑地址（VA），存储器实际地址为物理地址（PA），VA ≠ PA
   - CPU给出VA，通过**存储器管理部件（Memory Management Unit，MMU）**将逻辑地址转化为主存的物理地址
   - MMU功能：地址转换、检查访问信息是否在主存、检查地址越界、检查访问越权

### 2. 核心知识点

#### 分页式存储管理的基本思想

**地址空间划分**：
- 主存储器（物理存储器）和虚拟地址空间都被划分成**大小相等的页面（Page）**
- 磁盘和主存之间也按页为单位交换信息
- 每页大小通常为512B、1024B、2048B和4096B等

**术语区分**：
- **虚拟页/逻辑页（Virtual Page，VP）**：虚拟地址空间中的页
- **页框/页帧（Page Frame，PF/PP）**：主存空间中的页，又称物理页或实页

**存储分配与回收**：
- 加载程序时，OS从可用主存中分配若干物理页面给进程
- 分配的存储大小可以等于或小于进程大小
- 进程结束时，归还所有物理页

**页面映射和地址转换**：
- 建立一个逻辑页与物理页映射关系的表格，称为**页表（Page Table）**
- 程序执行过程中，CPU给出逻辑地址，由MMU查页表实时转换成物理地址

#### 地址结构

**虚拟地址（VA）结构**：

| VPN（虚拟页号） | VPO（页内偏移地址） |
|:---:|:---:|
| v 位 | k 位 |

其中：
- 页面大小 = $2^k$
- 页面数 = $2^v$
- VA = VPN | VPO = VPN × $2^k$ + VPO
- VPN = VA / $2^k$（取整）= VA >> k
- VPO = VA mod $2^k$
- VPN的起始地址 = VPN × $2^k$ = VPN << k

**物理地址（PA）结构**：

| PPN（页框号） | PPO（页内偏移地址） |
| :------: | :---------: |
|   p 位    |     k 位     |

其中：
- PPO = VPO（页内偏移不变）
- PA = PPN | PPO = PPN × $2^k$ + PPO

#### 页表结构

- 页表由操作系统在加载用户程序、分配好内存时创建
- 页表存在于内存中，记录虚拟页号VPN与物理页号PPN的对应关系
- 页表实际上可看成一个一维数组，VPN是元素索引号，元素值是PPN
- **每个进程有一个独立的页表**

页表项的标志位：

| 标志位  | 含义 |
|--------|------|
| Valid（有效位/装入位） | 1：该页在内存中；0：不在内存中（缺页） |
| Dirty（修改位） | 1：该页被修改过；0：未被修改 |
| Ref/Access（访问位） | 1：最近被访问过；0：最近未被访问 |
| Access(R/W)（访问权限） | 读/写/只读/不可访问 |

页表项的三种类型：
1. **未分配页**：进程虚拟地址空间中"空洞"对应的页（如VP0、VP4）
2. **已分配的缓存页**：有内容且已装入主存的页（如VP1、VP2、VP5等）
3. **已分配的未缓存页**：有内容但未装入主存的页（如VP3、VP6）

#### 地址转换过程

地址转换的实质：**将逻辑地址中的虚拟页号替换成相应的页帧号**。

转换过程（由MMU自动完成）：
1. 从页表基址寄存器，找到页表起始位置
2. 从VA中提取虚拟页号VPN查页表
3. 若装入位（Valid）为1，则取出PPN，与VPO拼接，形成PA
4. 若装入位为0，则说明缺页，引发**缺页中断（Page Fault）**，转缺页处理，调入该页

#### 缺页处理过程

指令 `load AX, VA` 需访问的页面VPN不在内存的处理过程：
1. CPU控制器将VA传给MMU，MMU从中提取VPN，查页表进行地址转换
2. MMU检测到页面VPN的状态位为0，引发缺页中断，记录页号VPN，暂停当前进程
3. 操作系统（缺页中断处理程序）响应中断，从页表中获取页面VPN的外存地址
4. OS从可用物理内存分配一个空闲物理页面PPN，将页面VPN的内容载入页帧PPN
5. OS将物理块号PPN写入页面VPN的页表项，并将状态位改为1
6. 系统重启进程，重新执行因缺页而中断的指令

**缺页处理效率问题**：缺页处理中页面加载过程涉及磁盘转动等机械动作，时间在ms级。缺页多会严重影响系统性能，应尽量减少缺页率。

#### 快表（TLB，Translation Lookaside Buffer）

**问题引入**：每次存储器引用要访问几次主存？不使用TLB时，取一条指令需2次访存（1次查页表 + 1次取指令），访问操作数也需2次。

**TLB的作用**：把经常要查的页表项放到Cache中，减少到内存查页表的次数。

**TLB结构**：TLB中的页表项 = tag + 主存页表项

**TLB查找方式**：
- **全相联（Fully Associative）**：VPN需和每个tag比较，没有index，只有Tag
- **组相联（Set Associative）**：VPN高位为Tag，低位为index，用作组索引

#### 引入TLB后的地址转换过程

1. 先由虚页号VPN到TLB中查找
2. 若TLB中的V=0或Tag≠VPN（TLB缺失），则到内存页表中查找
3. 若页表中的V=0，则缺页，到磁盘中找
4. 若页表中的V=1，取出PPN，更新TLB，与VPO拼接形成PA

**主存页表中不需要Tag**：因为页表用VPN直接索引（数组方式），不需要比较。

#### 多级页表

**引入原因**：虚拟地址空间很大（如4GB），全空间映射导致页表很大。但虚拟地址空间大部分是空洞（未使用），对应的页表项可以节省。

**二级页表结构**：逻辑地址由三部分组成：**页目录号 + 页表页号 + 页内偏移**

**二级页表的好处**：
- 解决了单级页表需要很大连续内存空间的问题
- 节省存储页表所需内存
- 例如：某进程只使用第0、1页和第1M-1页
  - 单级页表：需要4MB连续内存
  - 二级页表：仅需3个物理块（12KB内存）

对于64位机等系统，二级页表已不够用，可采用三级以上页表。

#### 倒置页表（Inverted Page Table）

**引入原因**：现代系统允许进程逻辑地址空间非常大，导致页表项过多，占用大量内存。

**改进方法**：为每一个物理块设置一个页表项，按物理块编号排序，内容为页号和进程标识符。

**地址变换**：根据进程标识符和页号去检索反置页表，匹配到的表项序号即为物理块号。可利用**哈希表技术**进行组织。

#### 页式存储管理的内存保护

三种保护方法：
1. **越界保护**：将页号与页表长度比较，若页号更大则地址越界
2. **标志位保护**：在页表中增加标志位，指出只读/读写/只可执行/不可访问
3. **键保护方法**：设置保护键，只有获知保护键的进程才能访问页面

#### 空闲页面管理

1. **位示图法**：每位与一帧相对应，用0/1表示对应块为空闲/已占用，用另一专门字记录当前空闲块数
2. **链表方法**：每个表项包含：进程占用区/空闲区标识、起始地址、长度、指向下一表项的指针

### 3. 关键公式/原理

**地址计算公式**：
- VPN = VA >> k（k为页内地址位数）
- VPO = VA mod $2^k$
- PPN = PA >> k
- PA = PPN × $2^k$ + VPO

**内存有效访问时间（EAT）计算**：

| 情况 | EAT公式 |
|------|---------|
| TLB命中，页在内存 | EAT = λ + t |
| TLB未命中，页在内存（先查TLB再查页表） | EAT = λ + t + λ + t = 2(λ + t) |
| 缺页（页不在内存） | EAT = λ + t + ε + λ + t = ε + 2(λ + t) |

其中：λ = TLB查找时间，t = 内存访问时间，ε = 缺页中断处理时间

**综合EAT公式**（TLB命中率a，缺页率f）：

$$EAT = a \times (\lambda + t) + (1-a) \times [(1-f) \times 2(\lambda + t) + f \times (\varepsilon + 2(\lambda + t))]$$

### 4. 计算示例

#### 示例1：逻辑地址→物理地址转换

**题目**：页面大小1024B，页表如下（VPN→PPN）：0→4, 1→5, 2→6, 3→2。求逻辑地址1524、4123、3172的物理地址。

**解答**：
- 逻辑地址1524：VPN = 1524 / 1024 = 1，VPO = 1524 mod 1024 = 500
  - 查表VPN=1→PPN=5，物理地址 = 5 × 1024 + 500 = **5620**
- 逻辑地址4123：VPN = 4123 / 1024 = 4，VPO = 4123 mod 1024 = 27
  - VPN=4超出页表范围（仅4页），**地址越界异常**
- 逻辑地址3172：VPN = 3172 / 1024 = 3，VPO = 3172 mod 1024 = 100
  - 查表VPN=3→PPN=2，物理地址 = 2 × 1024 + 100 = **2148**

#### 示例2：EAT计算

**题目**：内存访问时间1μs，TLB检索时间0.2μs，TLB命中率85%。

**解答**：
- 不使用TLB时：EAT = 2 × t = 2 × 1 = 2μs
- 使用TLB时：EAT = a(λ + t) + (1-a)(λ + 2t)
  = 0.85 × (0.2 + 1) + 0.15 × (0.2 + 2) = 0.85 × 1.2 + 0.15 × 2.2 = 1.02 + 0.33 = **1.35μs**
- 有效访存时间降低了：(2 - 1.35) / 2 × 100% = **32.5%**

#### 示例3：带TLB和Cache的完整访存过程

**题目**：14位虚拟地址，12位物理地址，页大小64B，TLB 16项4路组相联，16行Cache直接映射，主存块大小4B。逻辑地址0x0BD4。

**解答**：
- VPN = 0x0BD4 >> 6 = 0x2F（取高8位中有效部分）
- VPO = 0x0BD4 mod 64 = 0x34
- TLBI = VPN mod 4 = 0x2F mod 4 = 3（4路组相联）
- TLBT = VPN / 4 = 0x0B
- 查TLB：TLB[3]中tag=0x0B，命中，取出PPN=0x0D
- PA = PPN | PPO = 0x0D << 6 | 0x34 = 0x354
- CI = (PA >> 2) mod 16 = 0x354 >> 2 mod 16 = 0xD5 mod 16 = 0x5
- CT = PA >> 6 = 0x0D
- 查Cache：Cache[5]中tag=0x0D，命中，取出数据0x7236

### 5. 实际应用/实现

#### Nachos页式存储管理实现

```cpp
class AddrSpace {
  TranslationEntry *pageTable;  // 页表
  unsigned int numPages;        // 页表长度
};

// 地址翻译
ExceptionType AddrSpace::Translate(unsigned int vaddr, unsigned int *paddr, int isReadWrite) {
    unsigned int vpn = vaddr / PageSize;      // 提取虚拟页号
    unsigned int offset = vaddr % PageSize;   // 提取页内偏移
    if (vpn >= numPages)                       // 越界检查
        return AddressErrorException;
    pte = &pageTable[vpn];                     // 查页表
    pfn = pte->physicalPage;                   // 获取物理页号
    *paddr = pfn * PageSize + offset;          // 计算物理地址
    return NoException;
}
```

#### 软硬件分工

| 角色 | 职责 |
|------|------|
| 软件（OS） | 创建页表（装载程序时）、越界异常处理 |
| 硬件（MMU） | 地址变换、异常监测 |

地址变换机构组成：
1. 内存页表
2. 页表地址寄存器（如CR3）
3. 地址转换电路
4. 越界检测与中断处理

### 6. 常见问题/易错点

- 页表大小的计算：26位VPN × 32位/项 = 256MB（对于40位虚拟地址、16KB页面的系统）
- 如果按实际计算出的页表大小构建页表，页表过大可能无法一次装入内存（需要多级页表解决）
- 页式存储管理解决了外部碎片，但引入了内部碎片（最后一页可能不满）
- TLB缺失可由硬件也可由OS处理；Cache缺失由硬件处理；缺页由OS处理（缺页异常）
- 页表项数 = $2^{VPN位数}$，每项位数 = PPN位数 + 标志位数

### 7. 总结复盘

| 要点 | 说明 |
|-----|------|
| 分页核心思想 | 打破连续地址要求，将地址空间和物理空间都等分为页 |
| 页表作用 | 建立VPN→PPN的映射关系 |
| 地址转换 | VPN替换为PPN，VPO保持不变 |
| 缺页处理 | Valid=0时触发缺页中断，从磁盘调入页面 |
| TLB | 将常用页表项缓存在Cache中，加速地址转换 |
| 多级页表 | 解决大地址空间下页表过大的问题 |
| 倒置页表 | 按物理块组织页表项，减少页表内存占用 |
| EAT计算 | 考虑TLB命中率和缺页率的综合公式 |

## 三、段式存储管理

### 1. 概念引入

#### 分段存储管理的引入

编译后的应用程序通常有多个独立段构成。采用分区和分页存储管理存在一些问题：

1. **信息共享问题**：没有在逻辑上划分程序，有时在一个页面中会包含毫不相关的两段代码，使得信息共享不方便
2. **动态链接问题**：一个大程序可能包含数百、数千个程序模块，很多用户希望在程序执行过程中装入、动态连接所需模块，降低对内存需求，而分区管理和页式管理只能采用静态链接

**段式存储管理**可以解决上述问题。

### 2. 核心知识点

#### 分段的基本思想

- 把程序按内容或过程（函数）关系分成**段（Segment）**，每段有自己的名字，每个段一个地址空间
- 在装入内存后，每个段占用一片连续的存储区域，而各段之间可以不连续
- 可按段进行存储共享和保护
- 段式管理给用户提供了更方便灵活的程序设计环境
- 可将经常访问的段驻留内存，把不常用的段放入外存，较好地支持虚拟存储器

#### 分段地址结构（二维地址）

逻辑地址由两个成分组成：**段号s + 段内地址d**

即在分段存储情况下，作业的逻辑地址空间是**二维的**。

| 段号s | 段内地址d |
|:---:|:---:|
| 16位 | 16位 |

- 段号范围：0 ~ $2^{16}$-1
- 段内地址范围：0 ~ $2^{16}$-1（最大段长64KB）

**讨论**：若地址长度32位，段号占8位，最大段长为 $2^{24}$ = 16MB。

#### 内存分配

- 在分段存储管理中，内存以段为单位进行分配
- 给每个段分配一块连续的内存分区
- 各分区的大小由对应段的大小决定
- 类似于动态分区分配，但一个作业可以有多个段，这些段可以离散地放入内存不同的分区中

#### 段表

为每个进程建立了一个**段映射表（段表，Segment Map Table）**：
- 每个段在段表中占有一项
- 段表项包含：**段号、段的长度、段在内存中的起始地址（基址）**

| 段号 | 基地址(Base) | 段长(Limit) |
|------|-------------|------------|
| 0 | 219 | 600 |
| 1 | 2300 | 14 |
| 2 | 90 | 100 |
| 3 | 1327 | 580 |
| 4 | 1952 | 96 |

#### 地址映射与越界保护

地址转换过程：
1. 从逻辑地址中提取段号s和段内地址d
2. 用段号s查段表，获得该段的基地址b和段长l
3. **越界检查**：若 d ≥ l，则地址越界，产生中断
4. 若未越界，物理地址 = b + d

**快表支持**：段表也可以使用TLB加速查找。

#### 分段系统中程序和数据的共享

- 只要让两个进程各自段表中的有关表项指向共享的段，即可实现共享
- 相比分页系统，分段系统的共享更方便，因为段是按逻辑模块划分的

#### 段页式存储管理方式

**引入原因**：
- 分页系统：能有效提高内存利用率，但信息共享不方便
- 分段系统：能满足用户多方面需要，但存在碎片问题
- **段页式**：对两种方式"各取所长"

**基本思想**：
- 作业的地址空间仍按其逻辑结构分段
- 每个段又被进一步分成若干大小相同的页面
- 内存空间分成与页面大小相等的物理块

**地址结构**：段号s + 页号p + 页内偏移d

| 段号s | 页号p | 页内偏移d |
|:---:|:---:|:---:|

**地址变换过程**：
1. 用段号s查段表，获得该段的页表起始地址
2. 用页号p查该段的页表，获得物理块号PPN
3. 物理地址 = PPN × 页面大小 + 页内偏移d

**地址变换实例**：每页大小4KB，逻辑地址(0, 2, 500)
- 段号0，查段表获得第0段页表起始地址
- 页号2，查第0段页表，获得物理块号19
- PA = 19 × 4K + 500 = 78324

**段页式存储管理的段表和页表**：
- 先有一张段表，每个段表项指向该段的页表
- 每段有自己的页表，页表项与分页系统相同

### 3. 关键公式/原理

**段式地址转换**：PA = 段基地址 + 段内偏移（前提是段内偏移 < 段长）

**段页式地址转换**：PA = PPN × 页大小 + 页内偏移

### 4. 计算示例

#### 示例1：段式地址转换

**题目**：段表如下，求逻辑地址(2, 88)、(4, 100)、(3, 123)、(5, 1)的物理地址。

| 段号 | 基地址 | 段长 |
|------|--------|------|
| 0 | 219 | 600 |
| 1 | 2300 | 14 |
| 2 | 90 | 100 |
| 3 | 1327 | 580 |
| 4 | 1952 | 96 |

**解答**：
1. (2, 88)：段号2，基地址=90，段长=100，88 < 100，合法。PA = 90 + 88 = **178**
2. (4, 100)：段号4，基地址=1952，段长=96，100 ≥ 96，**地址越界**，产生越界中断
3. (3, 123)：段号3，基地址=1327，段长=580，123 < 580，合法。PA = 1327 + 123 = **1450**
4. (5, 1)：段号5，超出段表范围（仅5段，编号0~4），**越界**

#### 示例2：段页式地址转换

**题目**：虚地址空间16个段，每段最大 $2^{16}$ 字节，每页512字节。求地址各部分位数。

**解答**：
- 段号：$\log_2(16)$ = 4位
- 页号：$\log_2(2^{16}/512)$ = $\log_2(2^7)$ = 7位（每段 $2^{16}$/512 = 128页）
- 页内偏移：$\log_2(512)$ = 9位
- 虚拟地址总位数：4 + 7 + 9 = 20位

### 5. 实际应用/实现

段页式系统取一条指令或一个操作数通常需要三次访存：
1. 第一次：访问段表，获得页表地址
2. 第二次：访问页表，获得物理块号
3. 第三次：访问目标内存单元

**解决方法**：使用快表（TLB），将段表项和页表项缓存，可将三次访存减少到一次。

### 6. 常见问题/易错点

- 分段系统的逻辑地址空间是**二维的**（段号+段内地址），而分页系统是一维的
- 段页式系统中，段内共享比分页系统方便，因为段按逻辑模块划分
- 段页式系统的段表中每项包含该段的页表起始地址和页表长度
- 段式系统会产生外部碎片（段之间不连续，段内连续但段间有空隙）

### 7. 总结复盘

| 要点 | 说明 |
|-----|------|
| 分段思想 | 按逻辑模块（函数/过程）划分程序，每段独立命名 |
| 二维地址 | 段号s + 段内地址d |
| 段表 | 每段有基地址和段长，用于地址转换和越界保护 |
| 地址转换 | PA = 段基地址 + 段内偏移（需越界检查） |
| 信息共享 | 分段系统比纯分页系统更容易共享代码和数据 |
| 段页式 | 分段的逻辑性 + 分页的内存利用率，"各取所长" |
| 段页式地址 | 段号s + 页号p + 页内偏移d |
| 段页式变换 | 先查段表获得页表地址，再查页表获得物理块号 |

## 四、虚拟存储器

### 1. 概念引入

#### 虚拟存储器的引入背景

由于技术和成本原因，主存容量受到限制（从最早512KB到目前的4GB/8GB/16GB）。人们编程时不希望受到特定计算机物理内存大小的制约；多道程序运行要求有效而安全地共享主存。

**虚拟存储器的基本思想**：
1. 程序员在一个不受物理内存空间限制且比物理内存大得多的**虚拟逻辑地址空间**中编写程序（IA-32的逻辑空间大小为 $2^{32}$B = 4GB）
2. 程序执行过程中，把当前执行到的一部分程序和相应数据调入主存，其他暂不用的部分暂时存放在磁盘上
3. 实质：借用部分外存当做逻辑上的内存

#### 程序局部性原理

程序运行时通常具有**局部性（Locality）**特征，这是虚拟存储器能够高效工作的基础。

**时间局部性（Temporal Locality）**：如果程序中的某条指令一旦执行，则不久以后该指令可能再次执行；如果某数据被访问过，则不久以后该数据可能再次被访问。典型例子：循环结构。

**空间局部性（Spatial Locality）**：一旦程序访问了某个存储单元，在不久之后，其附近的存储单元也将被访问。典型例子：数组结构。

**举例分析**：

循环求和程序：
```c
sum = 0;
for (i = 0; i < n; i++)
    sum += a[i];
*v = sum;
```
- for循环体：时间局部性好
- 变量i、sum：时间局部性好
- 数组a：空间局部性好
- 程序代码：一般空间局部性较好

**数组按行访问 vs 按列访问**：
- 按行访问（P1）：访问顺序与存放顺序一致，空间局部性好
- 按列访问（P2）：每次跳过多个单元，空间局部性差
- 实测结果：按行访问比按列访问快约21.5倍！

#### 虚拟存储器实现原理

虚拟存储机制的实质：就像Cache是主存的缓存，主存可看成**磁盘的缓存**。

需要考虑的问题：
1. 存储器分配：如何将物理存储器有效分配给各进程使用
2. 地址映射：如何将虚拟地址VA转换成物理地址PA
3. **替换问题**：当无空闲主存时，如何淘汰信息腾出空间
4. 交换块大小：主存与磁盘之间交换的数据块大小
5. 写一致性问题：同一数据在主存和磁盘上的副本不一致如何处理

实现方案：
- **分页式**（一般为请求分页，广为使用）
- 分段式（一般为请求分段）
- 段页式（一般为请求段页式）

### 2. 核心知识点

#### 请求分页存储管理

**实现思想**：
- 应用程序运行前，仅将当前要运行的少数页面装入内存即可运行，其余部分暂留在外存
- 运行时，如果访问的页已调入内存，可继续执行；如果未调入内存（**缺页**），发出缺页中断请求，由OS调入该页
- 若内存已满，OS将内存中暂时不用的页调至外存（**页面置换**），腾出页帧后调入所需页面

**请求分页系统页表项结构**：Valid | Dirty | Ref(Access) | Access(R/W) | PCD

#### 请求分页系统内存分配

**内存页帧分配方案**：
1. **平均分配方法**：将所有可供分配的物理块平均分配给各进程
2. **按比例分配方法**：根据进程大小按比例分配物理块
   - 每个进程分到的物理块数 $b_i$ = $S_i$ / $\sum S_i$ × 总物理块数
3. **考虑优先权的分配方法**：一部分按比例分配，另一部分根据优先权分配

**页面替换策略**：
1. **固定分配局部置换**：为每个进程分配固定数目的物理块，缺页时只能从本进程的页面中选择换出
2. **可变分配全局置换**：OS保留空闲物理块，缺页时可从所有进程的全部物理块中选择换出
3. **可变分配局部置换**：缺页时只从本进程页面中选择换出，但缺页频繁时系统可为该进程增加附加物理块

#### 页面置换算法

##### 最佳置换算法（OPT，Optimal）

**算法思想**：总是选择以后永不使用的，或许是在最长时间内不再被访问的页面进行替换。

**示例**：系统分配3个物理块，页面引用串：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1

| 步骤 | 引用页 | 内存状态（3块） | 缺页？ |
|------|--------|----------------|--------|
| 1 | 7 | 7, -, - | 缺页 |
| 2 | 0 | 7, 0, - | 缺页 |
| 3 | 1 | 7, 0, 1 | 缺页 |
| 4 | 2 | 2, 0, 1 | 缺页（替换7，因为7最久不用） |
| 5 | 0 | 2, 0, 1 | 命中 |
| 6 | 3 | 2, 0, 3 | 缺页（替换1，因为1最久不用） |
| 7 | 0 | 2, 0, 3 | 命中 |
| 8 | 4 | 4, 0, 3 | 缺页（替换2，因为2最久不用） |
| 9 | 2 | 4, 2, 3 | 缺页（替换0，因为0最久不用） |
| 10 | 3 | 4, 2, 3 | 命中 |
| ... | ... | ... | ... |

**性能**：命中率 = 11/17 = 64.7%，缺页率 = 6/17 = 35.3%

**特点**：通常可保证获得最低缺页率，但**理论上无法实现**，因为无法预测进程未来的页面走向。

##### 先进先出页面置换算法（FIFO）

**算法思想**：当需要调入新页面但无空闲物理块时，选择在内存中驻留时间最长的页淘汰。

**示例**：同上引用串，分配3个物理块。

**性能**：命中率 = 6/17 = 37.5%，缺页率 = 10/17 = 62.5%

**Belady异常**：有时分配的物理块愈多，缺页率反而升高！

**示例**：地址流 A = 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5
- 3个物理块时：命中3次
- 4个物理块时：命中仅2次（反而减少）

**原因**：FIFO没有考虑程序执行的局部性特征。因此FIFO具有不稳定特征，一般不用作页面置换算法，但因其简单，被用于高速缓存替换算法。

##### 最近最久未使用算法（LRU，Least Recently Used）

**算法思想**：根据程序局部性原理，选择最后一次访问时间距离当前时间最长的页面淘汰，即淘汰没有使用的时间最长的页。

**性能**：命中率 = 8/17 = 47%，缺页率 = 9/17 = 53%

LRU缺页率比FIFO少，能带来更高系统性能，被广泛看好。

**LRU的硬件实现方法**：

1. **移位寄存器方法**：
   - 为每个内存页面设置一个移位寄存器 $R = R_{n-1}R_{n-2}...R_1R_0$
   - 每次内存访问都将寄存器最高位 $R_{n-1}$ 置1
   - 系统定时（如每隔100ms）将各寄存器右移一次
   - 页面置换时，选择寄存器值最小的页面替换
   - 优点：地址转换增加开销小
   - 缺点：每个页表项需增加n位（通常n≥8），存储开销大

2. **堆栈法**：
   - MMU设置一个堆栈，维护当前进程内存页面的访问顺序
   - 栈底到栈顶，访问时间递增，栈顶为最近访问过的页面，栈底为最近最久未使用页面
   - 每次内存访问时，将被访问页面调至栈顶
   - 每次页面置换时，选择栈底页面置换
   - 缺点：若进程页面数较多（如1000个），堆栈调整代价大至不可接受

##### Clock置换算法（近似LRU）

**引入原因**：LRU性能好但硬件支持需求高、实现代价大。实际应用中大多采用性能下降不大但实现代价小的LRU近似算法。

**A、简单的Clock置换算法**：
- 为每个页面增加一个**访问位A**
- 将内存中所有页面通过链接指针链接成一个**循环队列**
- 每次访存时，MMU将页面访问位置1
- 页面置换时检查访问位A：
  - 若A=0，直接替换
  - 若A=1，将其置0，检查下一个页面
  - 若所有页面A全为1，置0后返回队首继续检查
- 结果：上一轮检查以来A没有被置1的第一个页面被选做淘汰页
- 页表开销仅增加1 bit，地址转换过程仅增加一次访问位置位操作

**B、改进型Clock置换算法**：
- 在访问位A基础上增加**修改位M**
- 由A和M组合成四类页面：

| 类型 | A | M | 说明 | 淘汰优先级 |
|------|---|---|------|-----------|
| 1类 | 0 | 0 | 最近未被访问，也未修改 | **最佳淘汰页** |
| 2类 | 0 | 1 | 最近未被访问，但已修改 | 较好淘汰页 |
| 3类 | 1 | 0 | 最近已被访问，但未修改 | 可能再被访问 |
| 4类 | 1 | 1 | 最近已被访问且被修改 | 最不想淘汰 |

**三步扫描过程**：
1. 从指针位置开始，寻找A=0且M=0的1类页面作为淘汰页（第一次扫描不改变A）
2. 若失败，寻找A=0且M=1的2类页面作为淘汰页，扫描过程中将遇到的A复位为0
3. 若仍失败，指针返回开始位置，将所有A复0，然后重复第一步

**性能**：减少了页面回写辅存（非常耗时）的频率，提高置换效率

**应用**：Linux、Windows等现有系统均采用Clock置换算法

#### 内存有效访存时间（EAT）详细计算

**符号定义**：
- λ：查找快表时间
- t：实际物理内存访问时间
- ε：缺页中断处理时间
- a：快表命中率
- f：缺页率

**综合EAT公式**：

$$EAT = a \times (\lambda + t) + (1-a) \times [(1-f) \times 2(\lambda + t) + f \times (\varepsilon + 2(\lambda + t))]$$

**计算实例1**：内存访问时间9ns，TLB检索1ns，TLB命中率75%
- 无TLB：EAT = 2 × 9 = 18ns
- 有TLB：EAT = 0.75 × (1+9) + 0.25 × 2 × (1+9) = 7.5 + 5 = 12.5ns
- 地址转换时间下降：(9 - (12.5-9)) / 9 = 61%

**计算实例2**：TLB命中率98%
- 有TLB：EAT = 0.98 × (1+9) + 0.02 × 2 × (1+9) = 9.8 + 0.4 = 10.2ns
- 地址转换时间下降：(9 - (10.2-9)) / 9 = 86%

**计算实例3**：TLB命中率95%，内存读写周期10ns，TLB查表2ns，内外存传送1ms

| 缺页率f | EAT |
|---------|-----|
| 2% | 约1011ns（几乎是实际物理内存访问时间的100倍） |
| 0.1% | 约17.6ns（缺页率下降到0.1%，EAT仅为情况一的1/50） |
| 0.00001% | 约15.5ns（页面置换影响甚微） |

**结论**：降低缺页率对提高访存效率非常重要。

#### "抖动"（Thrashing）与工作集

**抖动的产生过程**：
1. 若一个进程未获得足够页帧，缺页率居高不下
2. 导致CPU利用率很低
3. 操作系统错误认为需要在内存中增加程序道数
4. 更多进程增加到系统中
5. 每个进程获得的页帧更少，缺页率更高，形成恶性循环

**抖动（Thrashing）的定义**：系统大部分时间忙于进程页面的导入和导出，内存中程序道数增加，但CPU利用率却下降。

**判断系统是否处于抖动**：系统花费在页面导入导出的时间比实际执行进程处理工作时间更多。

**导致抖动的根本原因**：工作集（局部性）所需内存之和超过实际分配给进程的内存。

#### 工作集（Working Set）模型

**定义**：进程在时间间隔（t-Δ，t）中引用页面的集合，记为 $w(t, Δ)$。

其中：
- Δ为**工作集窗口（Working-Set Window）**：一个固定数量的页引用（如10,000条指令）
- $WSS_i$（进程 $P_i$ 的工作集）= 在最近Δ次页面引用中引用的页面总数

**窗口大小Δ的选择**：
- 若Δ太小：不能包含整个局部性区域
- 若Δ太大：会包含多个局部性区域
- 若Δ = ∞：将包含整个程序

**抖动判断**：设D = Σ $WSS_i$（总需求页帧数），m为可用页帧总数
- 若 D > m，则发生抖动
- 策略：若 D > m，则暂停其中一个进程

#### 预防抖动的方法

1. **采取局部置换策略**：可变分配时采用局部置换，防止一个进程的缺页影响其他进程
2. **把工作集算法融入到处理机调度中**：调度程序从外存调入作业前，先检查每个进程在内存的驻留页面是否足够多
3. **利用"L=S"准则调节缺页率**：L是缺页之间的平均时间，S是平均缺页服务时间。当L≈S时，系统处于最佳状态
4. **选择暂停的进程**：当多道程序度偏高时，减少并发进程数

#### 页面失效率与EAT的关系

**练习**：TLB命中率95%，内存读写周期10ns，TLB查表2ns，内外存传送1ms

| 缺页率 | EAT |
|--------|-----|
| 2% | 1011.77ns |
| 0.1% | 17.6ns |
| 0.00001% | 15.45ns |

**结论**：缺页率从2%降到0.1%，EAT降低了约57倍。降低缺页率对提高访存效率非常重要。

### 3. 关键公式/原理

**页面置换算法比较**（3个物理块，页面引用串：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1）：

| 算法 | 缺页次数 | 缺页率 |
|------|---------|--------|
| OPT | 6 | 35.3% |
| LRU | 9 | 53% |
| FIFO | 10 | 62.5% |

**EAT综合公式**：
$$EAT = a(\lambda + t) + (1-a)[(1-f) \times 2(\lambda + t) + f(\varepsilon + 2(\lambda + t))]$$

**工作集判断**：
$$D = \sum WSS_i > m \Rightarrow 抖动$$

#### 示例2：带TLB的请求分页系统综合计算

**题目**：页表内容如下，页面大小4KB，内存访问时间10ns，TLB访问时间1ns，缺页处理1ms，驻留集大小固定为2，LRU置换，局部淘汰。TLB初始为空。虚地址序列：2362H、1565H、25A5H。

**解答**：

**(1) 物理地址计算**：
- 页面大小4KB = $2^{12}$，页内偏移占12位，页号占高位
- 2362H：页号P=2，页内偏移=362H
- 1565H：页号P=1，页内偏移=565H
- 25A5H：页号P=2，页内偏移=5A5H

**(2) 各地址访问时间**：

**2362H（P=2）**：
- TLB为空，TLB未命中 → 访问内存页表 → 得到页框号 → 更新TLB → 访问主存
- 时间 = 1(TLB) + 10(页表) + 1(更新TLB) + 10(主存) = **22ns**

**1565H（P=1）**：
- TLB未命中 → 访问内存页表 → 页面不在内存 → 缺页中断处理 → 调入页面 → 更新页表和TLB → 重新执行
- 时间 = 1(TLB) + 10(页表) + 1000000(缺页处理) + 1(更新TLB) + 10(主存) ≈ **1000022ns**

**25A5H（P=2）**：
- TLB命中（第一次访问2362H时已将P=2放入TLB）
- 时间 = 1(TLB) + 10(主存) = **11ns**

### 5. 实际应用/实现

#### Linux虚存空间管理

**Linux在X86上的虚拟地址空间**（其他Unix系统类似）：
- 0x00000000 ~ 0x08048000：未使用
- 0x08048000：只读代码段（.init, .text, .rodata）
- 0x08049000：读写数据段（.data, .bss）
- 向上增长：堆（heap），由malloc动态生成
- 向下增长：用户栈（User Stack）
- 共享库区域
- 0xC0000000 ~ 0xFFFFFFFF：内核虚存区（1GB）

**Linux将虚存空间组织成"区域（area）"**：
- 使用 `vm_area_struct` 结构管理
- `vm_start`：区域起始地址
- `vm_end`：区域结束地址
- `vm_prot`：读写权限
- `vm_flags`：共享/私有标志
- `vm_next`：指向下一个区域
- `pgd`：全局页目录地址（指向L1页表，装入CR3）

**Linux页故障类型**：
1. **Segmentation fault**：访问不存在的页面（非正常缺页）
2. **Normal page fault**：正常缺页，页面不在内存中
3. **Protection exception**：保护异常，如写只读页面

#### 请求分页系统地址转换完整流程

```
CPU给出虚拟地址VA
    ↓
提取VPN，查TLB
    ↓
TLB命中？→ 是 → 取出PPN，与VPO拼接为PA → 访问主存
    ↓ 否
查内存页表
    ↓
页面在内存？→ 是 → 取出PPN，更新TLB，拼接PA → 访问主存
    ↓ 否
触发缺页中断
    ↓
OS从磁盘调入页面 → 分配空闲物理块 → 更新页表和TLB → 重新执行指令
```

### 6. 常见问题/易错点

- **Belady异常**仅出现在FIFO算法中，OPT和LRU不会出现
- LRU是OPT的最佳近似，但硬件实现代价高
- **Clock算法**是LRU的近似，用1个访问位代替完整的访问时间记录
- **改进型Clock**用A和M两个位，优先淘汰未修改的页面（减少磁盘写操作）
- 缺页率对EAT影响巨大：缺页率从2%降到0.1%，EAT可降低数十倍
- 抖动的根本原因是工作集所需内存超过分配的物理内存
- 工作集窗口Δ的选择：太小不能包含整个局部性区域，太大会包含多个局部性区域
- 按行访问数组比按列访问快得多（空间局部性差异）

### 7. 总结复盘

| 要点 | 说明 |
|-----|------|
| 虚拟存储器 | 借用部分外存当做逻辑上的内存，部分调入即可运行 |
| 局部性原理 | 时间局部性（循环）+ 空间局部性（数组顺序访问） |
| 请求分页 | 运行时按需调页，缺页时从磁盘调入 |
| OPT | 理论最优，选择最久不用的页面，无法实际实现 |
| FIFO | 选择最早进入内存的页面，简单但有Belady异常 |
| LRU | 选择最近最久未使用的页面，性能好但硬件开销大 |
| Clock | LRU近似算法，用访问位实现，Linux/Windows采用 |
| 改进型Clock | 在Clock基础上增加修改位，优先淘汰未修改页面 |
| EAT公式 | 综合考虑TLB命中率和缺页率 |
| 抖动 | 工作集所需内存超过实际分配，CPU利用率急剧下降 |
| 工作集模型 | D = ΣWSSi > m 则抖动，需暂停进程 |
| 抖动预防 | 局部置换、工作集调度、L=S准则、减少进程数 |
