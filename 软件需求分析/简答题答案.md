# 软件需求分析 - 四个核心问题答案整理

---

## 一、需求层次按照层次来说主要有哪几个

需求层次主要有**三个层次**（从高到低）：

| 层次 | 英文 | 说明 |
|------|------|------|
| **业务需求** | Business Requirement | 反映组织机构或客户对系统的概括目标要求，描述组织为什么要开发系统，建立战略出发点 |
| **用户需求** | User Requirement | 描述用户使用系统完成的任务，描述系统能帮助用户做什么 |
| **系统需求** | System Requirement | 包括**功能需求**和**非功能需求**，定义开发人员必须实现的软件功能 |

---

## 二、需求定义中如何判定优秀需求的特性

优秀需求具有以下**7个特性**：

1. **完整性** — 每项需求必须将功能描述清楚，使开发人员获得必要的所有信息
2. **正确性** — 每项需求必须准确陈述要开发的功能，依据是用户或高层系统需求规格说明
3. **无歧义性** — 所有读者对需求只能有一个明确统一的解释
4. **可行性** — 每项需求必须在已知系统和环境的权能和限制范围内可以实施
5. **有优先级** — 需区分需求的优先级别，可从业务、技术开发、项目管理三个角度划分
6. **必要性** — 每项需求应记录客户真正需要和系统需遵从的标准
7. **可验证性** — 每项需求能通过测试用例或其他验证方法来确定产品是否实现

---

## 三、需求发生变更时的流程

需求变更控制流程遵循**变更申请－审批－更改－重新确认**的流程：

1. **变更申请** — 提出需求变更请求
2. **审批** — 评估变更影响，进行审批决策
3. **更改** — 按审批结果实施变更
4. **重新确认** — 验证变更后的结果

**变更控制策略包括三个阶段**：

- 项目启动阶段的变更预防
- 项目实施阶段的需求变更
- 项目收尾阶段的总结

---

## 四、需求管理中应该包含哪些内容

需求管理包括以下核心内容：

| 内容 | 说明 |
|------|------|
| **需求确认** | 开发方和客户共同对需求文档进行评审，双方达成共识后作出书面承诺，使需求文档具有商业合同效果 |
| **需求跟踪** | 建立与维护需求与后续工作成果的对应关系，包括**正向跟踪**（检查需求是否在后继成果中找到对应点）和**逆向跟踪**（检查后继成果是否在需求中找到出处） |
| **需求变更控制** | 依据变更申请－审批－更改－重新确认的流程处理变更，防止变更失去控制导致项目混乱 |
| **版本控制** | 对配置项的不同版本进行标志和跟踪，保证在需要时可回到旧版本 |

> 需求管理的目的是在客户与开发方之间建立对需求的共同理解，维护需求与其它工作成果的一致性，并控制需求的变更。
