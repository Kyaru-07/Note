# 第03章 用例图

---

## 一、概念引入

### 1.1 什么是用例图

用例图（Use Case Diagram）是 UML 中最重要的行为图之一，用于描述系统的功能需求。它从用户的角度出发，展示系统提供的服务和功能单元。

**核心思想：** 用例图回答的是"系统能为谁做什么"这个问题。

- **"谁"** = 参与者（Actor）
- **"做什么"** = 用例（Use Case）

### 1.2 用例的起源

- 用例（Use Case）由 **Ivar Jacobson** 在爱立信公司开发 AXE 系统时发明
- 在其博士论文（1985年）和专著（1992年）中有详细介绍
- 被称为"第二代面向对象技术的标志"
- 用例是 Rational 统一过程（RUP）的核心概念——"用例驱动"

### 1.3 用例图为什么重要

| 作用 | 说明 |
|------|------|
| 需求捕获 | 从用户视角捕获系统功能需求，避免开发人员凭空想象 |
| 沟通桥梁 | 为用户和开发人员提供共同的语言，减少理解偏差 |
| 驱动开发 | 用例贯穿软件开发全过程：需求分析、设计、测试都以用例为中心 |
| 范围界定 | 通过系统边界明确系统做什么、不做什么 |

> **类比：** 用例图就像一本书的目录——它告诉读者这本书包含哪些章节（功能），但不会展开每个章节的具体内容。用例描述才是具体内容。

## 二、核心概念

### 2.1 参与者（Actor）

#### 2.1.1 参与者的概念

**定义：** 参与者是指系统以外的、需要使用系统或与系统交互的外部实体。

**关键理解：** 参与者在系统**外部**，不在系统内部。

**参与者的分类：**

| 类型 | 说明 | 示例 |
|------|------|------|
| 人 | 使用系统的用户 | 银行客户、图书管理员、系统管理员 |
| 外部设备 | 与系统交互的硬件设备 | 打印机、扫描仪、传感器 |
| 外部系统 | 与本系统交互的其他软件系统 | 短信服务平台、旧的课程目录系统 |

**参与者的图形符号：**

```
    O
   /|\    ← 人形图标（简笔画小人）
   / \
  Actor名称
```

#### 2.1.2 识别参与者的方法

**案例：银行柜台业务系统**

```
参与者分析：
- 银行客户：在系统中办理了账户的居民，通过系统进行金融交易
- 柜台营业人员：负责具体操作事务的办事人员，为客户办理存款、取款等
- 短信服务平台：负责向个人发送短信的外部系统平台
```

**案例：大学图书管理系统**

```
需求描述：
读者通过系统管理员在某大学图书管理系统中办理借书证。
读者分学生和教师。
读者借还书籍要通过图书管理员来完成。
系统管理员负责图书管理系统的借书证维护和图书数据维护等操作。

参与者识别：
- 图书管理员
- 系统管理员
- 读者（注意：读者本身是否是参与者？要看系统边界）
```

> **易错点：** 识别参与者时，要站在系统外部看。如果"读者"不直接操作系统（借还书通过图书管理员完成），则读者可能不是参与者。但如果需求说"读者可以自己在终端机上直接进行借还书"，那读者就是参与者。

### 2.2 参与者之间的关系

参与者之间的关系主要是**泛化关系**。

#### 2.2.1 泛化关系的概念

**定义：** 泛化关系是一般参与者与特殊参与者之间的关系。特殊参与者继承一般参与者的所有行为，并可以拥有自己的额外行为。

**图形表示：**

```
      一般参与者
          △
          |        ← 空心三角箭头，指向一般参与者
          |
      特殊参与者
```

#### 2.2.2 泛化关系的作用

**核心作用：** 有效地减少参与者与用例之间的关联关系的个数，简化用例模型。

**案例：网上购物系统**

```
需求描述：
普通用户只能浏览网站上陈列的商品。
注册会员除了普通用户具有的操作之外，还有权限进行在线购物。

分析：
- 共同点：普通用户操作的所有功能，注册会员也能操作
- 不同点：注册会员还能操作普通用户不能操作的功能
- 结论：普通用户作为一般参与者，注册会员作为特殊参与者

泛化关系：
    普通用户（一般参与者）
        △
        |
    注册会员（特殊参与者）

注册会员 → 在线购物（额外用例）
普通用户 → 浏览商品（继承用例）
注册会员 → 浏览商品（继承自普通用户）
```

**案例：大学图书管理系统**

```
需求新增：
读者可以自己在终端机上直接进行借还书，也可以通过图书管理员来完成。

此时可以引入泛化关系：
    读者（一般参与者）
        △
        |
    图书管理员（特殊参与者）

图书管理员继承了读者的某些行为（如查找图书），
同时还有自己的特殊行为（如借还书登记）。
```

---

### 2.3 参与者与用例之间的关系

#### 2.3.1 关联关系

**定义：** 参与者与用例之间的关系是**关联关系**，起通信的作用，描述了一个参与者完成系统的一项功能。

**图形表示：**

```
参与者 ——————— 用例     ← 实线连接
```

**理解要点：**
- 关联关系是参与者与用例之间唯一的关系类型
- 每条关联线表示"该参与者可以使用该功能"
- 参与者必须与至少一个用例有关联
- 用例必须与至少一个参与者有关联（否则该用例无意义）

---

### 2.4 用例（Use Case）

#### 2.4.1 用例的概念

**定义：** 用例是对一个参与者使用系统的一项功能时所进行的交互过程的一个文字描述序列。

**双重含义：**
1. **功能**——参与者可以感受到的系统服务或功能单元
2. **交互过程**——功能涉及的交互过程

> **关键理解：** 功能及功能涉及的交互过程，两者都重要。用例不仅仅是"一个功能名称"，更是对"如何使用这个功能"的完整描述。

**用例的特征：**
1. 站在系统外部，看到的系统功能
2. 总是与参与者交互
3. 是一个行为上相关的步骤序列
4. 用例分析本质上是一种功能分解技术
5. 交互过程的描述中，可以发现类的特征和类的行为

**用例的图形符号：**

```
    ╭─────────────────╮
    │                 │    ← 椭圆形
    │   用例名称       │
    │                 │
    ╰─────────────────╯
```

#### 2.4.2 识别用例的方法

**案例：银行业务系统**

```
可能的用例：
- 浏览账户余额
- 列出交易清单
- 存款
- 取款
- 转账
```

**案例：大学图书管理系统**

```
需求描述：
读者在大学图书管理系统中可以查找图书。
读者分学生和教师。
读者借还书籍要通过图书管理员来完成。
为了方便工作，图书管理员也可以查找图书。
系统管理员负责大学图书管理系统的借书证维护和图书数据维护等操作。

用例识别：
- 查找图书（读者、图书管理员都使用）
- 借书登记（图书管理员使用）
- 还书登记（图书管理员使用）
- 借书证维护（系统管理员使用）
- 图书数据维护（系统管理员使用）
```

**案例：超市购买商品系统**

```
需求描述：
顾客带着所要购买的商品到达超市的一个销售点终端。
营业员通过销售点终端录入每项商品的通用产品代码。
系统确定商品的价格，并将商品代码、数量信息加入到正在运行的系统。
系统显示当前商品的描述信息和价格。

参与者识别：顾客、营业员
用例识别：
- 接收数据（接收顾客购买的商品的信息）
- 显示数据（显示顾客购买的商品的信息）
- 打印购物单（打印顾客购买的商品的信息）
- 商品信息维护（确定每件商品的代码及单价）
```

### 2.5 用例之间的关系

用例之间存在三种关系：**泛化关系**、**包含关系**、**扩展关系**。

#### 2.5.1 用例间的泛化关系

**定义：** 特殊用例继承、覆盖或增加一般用例的行为。

**案例：在线股票经纪人系统**

```
        交易行为（一般用例）
           △
      ┌────┼────┐
      |    |    |
    买入  卖出  卖空（特殊用例）

特殊用例继承一般用例的行为，并可能有自己的特殊行为。
```

**案例：学校信息管理系统**

```
        查找人（一般用例）
           △
      ┌────┼────┐
      |    |    |
   查找学生 查找教师 查找职工（特殊用例）
```

#### 2.5.2 用例间的包含关系（include）

**定义：** 基本用例的行为**必然执行**包含用例的行为。包含用例补充了基本用例的行为，且是**必须执行**的。

**语法：** `<<include>>`

**图形表示：**

```
基本用例 - - - - -> 被包含用例
            <<include>>
         （虚线箭头，指向被包含用例）
```

**案例：网上购物系统**

```
需求描述：
当注册会员在线购物时，网上购物系统需要对顾客的信用进行检查，
检查输入的信用卡号是否有效。

用例关系：
"在线购物" --<<include>>--> "信用检查"

含义：每次在线购物，都必须执行信用检查。
```

#### 2.5.3 用例间的扩展关系（extend）

**定义：** 基本用例的行为**条件执行**扩展用例的行为。扩展用例在满足特定条件时才被执行。

**语法：** `<<extend>>`

**图形表示：**

```
扩展用例 - - - - -> 基本用例
            <<extend>>
         （虚线箭头，指向基本用例）
```

**案例：图书借阅系统**

```
需求描述：
当读者还书时，如果借书时间超期，则需要交纳一定的滞纳金，作为罚款。

用例关系：
"交纳滞纳金" --<<extend>>--> "还书"

含义：只有在借书超期的情况下，才需要执行"交纳滞纳金"。
"还书"是基本行为，"交纳滞纳金"是有条件的扩展行为。
```

#### 2.5.4 包含关系与扩展关系的区别（重点！）

| 比较项 | 包含关系（include） | 扩展关系（extend） |
|--------|---------------------|---------------------|
| 执行条件 | **必然执行** | **条件执行** |
| 箭头方向 | 基本用例 → 被包含用例 | 扩展用例 → 基本用例 |
| 补充性质 | 补充基本用例的必要行为 | 补充基本用例的可选行为 |
| 依赖方向 | 基本用例依赖于被包含用例 | 基本用例依赖于扩展用例 |

**共同点：**
- 包含用例和扩展用例都补充了基本用例的行为
- 基本用例的行为都依赖于包含用例/扩展用例的行为
- 包含关系和扩展关系都是依赖关系的特例

**案例对比：ATM 取款操作**

```
中国农业银行 ATM：
"取款" --<<include>>--> "登录"
含义：每次取款都必须先登录。登录是取款的必要步骤。

中国建设银行 ATM：
"登录" --<<extend>>--> "取款"
含义：只有在需要时才登录。登录是可选的扩展行为。

东莞银行 ATM：
"取款" --<<include>>--> "登录"
含义：每次取款都必须登录。
```

> **考试重点：** 包含关系和扩展关系的区别是考试高频考点。记住口诀：
> - **包含 = 必然执行**（箭头从基本用例指向被包含用例）
> - **扩展 = 条件执行**（箭头从扩展用例指向基本用例）

### 2.6 系统边界

#### 2.6.1 系统边界的概念

**定义：** 系统边界是指系统与系统之间的界限。它定义了由谁或什么（即参与者）来使用系统，系统能够为参与者提供什么特定服务。

**确定系统边界的意义：**
- 定义什么是系统的组成部分（边界内）
- 定义什么是系统的外部（边界外）

**图形表示：**

```
┌─────────────────────────────────┐
│         系统名称                 │  ← 实线方框表示系统边界
│                                 │
│   用例1    用例2    用例3        │  ← 用例绘制在方框里面
│                                 │
└─────────────────────────────────┘
     ↑           ↑           ↑
  参与者A      参与者B      参与者C    ← 参与者绘制在方框外面
```

#### 2.6.2 系统边界决定参与者

**案例：基金交易系统**

```
情况一：系统边界较小
基金交易系统只提供买卖功能，基金品种由外部系统管理。

┌──────────────┐
│ 基金交易系统   │
│              │
│  买入基金     │
│  卖出基金     │
└──────────────┘
  ↑        ↑
交易客户  基金管理系统（作为参与者）

此时基金管理系统是参与者（外部系统）。
```

```
情况二：系统边界扩大
新的基金交易系统同时包含买卖和品种管理功能。

┌──────────────────────┐
│ 新的基金交易系统       │
│                      │
│  买入基金             │
│  卖出基金             │
│  基金品种管理          │
└──────────────────────┘
  ↑            ↑
交易客户    基金公司员工

此时基金管理系统不再是参与者，而是系统内部的一部分。
```

> **关键结论：** 系统边界决定了参与者。如果系统边界规划得不一样，系统的参与者就会发生很大的变化。相应地，系统的用例也会发生很大的变化。只有弄清楚了系统边界，才能更好地确定系统的参与者和用例。

### 2.7 用例描述

#### 2.7.1 为什么需要用例描述

没有描述的用例就像是一本书的目录——人们只知道该目录标题，但并不知道该目录的具体内容是什么。

不同的问题域，相同名字的用例可能完成不同的操作过程。

#### 2.7.2 用例描述的主要内容

| 描述项 | 说明 |
|--------|------|
| **用例名称** | 动宾结构或主谓结构命名的用例名字 |
| **标识符** | 用例的唯一标识符，在文档的别处可以用标识符来引用该用例 |
| **用例描述** | 对用例作用和目的的简要描述，包括最终任务和期望结果 |
| **参与者** | 与此用例相关联的参与者列表 |
| **前置条件** | 访问用例之前必须满足的条件列表 |
| **后置条件** | 用例完成后必须满足的条件列表 |
| **基本操作流程** | 各项工作都正常进行时用例的工作方式 |
| **可选操作流程** | 变更工作方式、出现异常或发生错误时的路径 |
| **特殊需求** | 非功能性需求和设计约束 |
| **被泛化的用例** | 此用例所泛化的用例列表（父用例） |
| **被包含的用例** | 此用例所包含的用例列表 |
| **被扩展的用例** | 此用例所扩展的用例列表 |

#### 2.7.3 用例描述案例

**案例：中国农业银行"取款"用例**

```
用例名称：取款
参与者：银行客户

基本流程：
1. 要求用户重新输入密码进行验证
2. 密码正确后，用户输入取款金额
3. 判断该储户的账户余额是否足额
4. 若足额，则输出等额的人民币现金，修改账户余额
5. 若不足额，则本次取款操作失败

被包含的用例：登录（每次取款都必须先登录）
```

**案例：中国建设银行"取款"用例**

```
用例名称：取款
参与者：银行客户

基本流程：
1. 要求用户输入取款金额
2. 判断该储户的账户余额是否足额
3. 若足额，则输出等额的人民币现金，修改账户余额
4. 若不足额，则本次取款操作失败

被扩展的用例：登录（需要时才登录）
```

**案例：东莞银行"取款"用例**

```
用例名称：取款
参与者：银行客户

基本流程：
1. 判断本次登录以来，是否是第一次进行取款操作
2. 根据判断结果执行以下操作：
   2.1 若是第一次，用户输入取款金额
   2.2 否则需要验证，通过后，用户输入取款金额
3. 判断该储户的账户余额是否足额
4. 若足额，则输出等额的人民币现金，修改账户余额
5. 若不足额，则本次取款操作失败

被包含的用例：登录
```

## 三、方法速通

### 3.1 识别参与者的三步法

```
第一步：确定系统边界
    ↓
第二步：找出系统外部的实体
    ↓
第三步：分类参与者（人、外部设备、外部系统）
```

**识别技巧：**
1. 谁使用系统的主要功能？（人）
2. 谁来维护系统的数据？（人）
3. 系统使用哪些外部设备？（外部设备）
4. 系统需要与哪些外部系统交互？（外部系统）

### 3.2 识别用例的三步法

```
第一步：对每个参与者，问"他需要系统做什么？"
    ↓
第二步：将功能描述为动宾短语（如"查找图书"、"借书登记"）
    ↓
第三步：检查用例是否与参与者有交互
```

**识别技巧：**
1. 参与者要完成什么任务？
2. 参与者需要什么信息？
3. 参与者需要提供什么信息给系统？
4. 系统需要通知参与者什么事件？

### 3.3 判断包含关系 vs 扩展关系的口诀

```
包含：每次都要做 → include → 箭头指向被包含用例
扩展：有时才要做 → extend → 箭头指向基本用例
```

### 3.4 绘制用例图的步骤

```
1. 确定系统边界
2. 识别所有参与者
3. 识别所有用例
4. 建立参与者与用例的关联关系
5. 分析用例之间的关系（包含、扩展、泛化）
6. 添加系统边界框
7. 检查和优化
```

### 7.2 常见错误与陷阱

**错误1：把系统内部的实体当作参与者**

```
错误示例：在图书管理系统中，把"图书"当作参与者。
正确理解："图书"是系统内部的数据，不是参与者。
参与者必须是系统外部的实体。
```

**错误2：混淆包含关系和扩展关系的箭头方向**

```
错误示例：把包含关系的箭头画反。
正确记忆：
- 包含：基本用例 → 被包含用例（箭头指向被包含用例）
- 扩展：扩展用例 → 基本用例（箭头指向基本用例）
```

**错误3：把可选行为建模为包含关系**

```
错误示例："还书" --<<include>>--> "交纳滞纳金"
正确建模："交纳滞纳金" --<<extend>>--> "还书"
原因：交纳滞纳金是有条件的（超期时），不是每次都执行。
```

**错误4：用例粒度不合适**

```
粒度过细：把每个操作步骤都建模为一个用例
  例如：输入密码、验证密码、显示主菜单... 都作为独立用例

粒度过粗：把整个系统功能作为一个用例
  例如：银行系统的所有功能都放在一个"银行业务"用例中

合适的粒度：一个用例对应用户的一个完整任务
  例如：存款、取款、转账、查询余额
```

**错误5：忽视系统边界**

```
错误做法：不考虑系统边界，直接识别参与者和用例
正确做法：先确定系统边界，再识别参与者和用例
原因：系统边界决定了谁是参与者，边界不同，参与者不同
```

**错误6：不恰当的用例描述**

```
错误做法：用例描述过于简单，只有功能名称
正确做法：用例描述应包含完整的交互过程
  - 前置条件
  - 基本操作流程
  - 可选操作流程
  - 后置条件
```

### 7.3 考试重点提示

1. **参与者与用例的识别**——给出需求描述，要求识别参与者和用例
2. **包含关系与扩展关系的区别**——给出场景，要求判断是包含还是扩展
3. **用例描述的编写**——给出用例，要求编写完整的用例描述
4. **系统边界的作用**——理解系统边界如何影响参与者和用例的确定
5. **参与者泛化关系的作用**——理解如何用泛化关系简化用例模型

### 7.4 UML 模型总览

用例图在 UML 模型体系中的位置：

```
UML 模型
├── 功能描述
│   ├── 用例图          ← 本章内容：系统的整个功能
│   └── 用例描述        ← 本章内容：一个功能的详细描述
├── 结构描述
│   ├── 类图            ← 后续章节
│   └── 对象图
├── 行为描述
│   ├── 顺序图          ← 后续章节：一个用例的实现方案
│   ├── 协作图          ← 后续章节：一个用例的实现方案
│   ├── 活动图          ← 后续章节：一个用例的实现方案
│   └── 状态图          ← 后续章节：一个类的所有可识别的变化
└── 体系结构描述
    ├── 组件图          ← 后续章节：源代码文件放在哪个文件夹
    ├── 包图            ← 后续章节
    └── 部署图          ← 后续章节：源代码文件和文件夹放在哪台机器
```

> **用例图是 UML 建模的起点。** 用例驱动整个开发过程：从用例图出发，可以推导出类图、顺序图、状态图等所有其他模型。理解用例图，是掌握 UML 建模的第一步。
