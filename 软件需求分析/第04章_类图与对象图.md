# 第04章 类图与对象图

## 1. 概念引入

### 1.1 什么是类图与对象图

类图（Class Diagram）和对象图（Object Diagram）是UML中的**结构模型**，用于描述系统的静态结构。

- **类图**：描述系统中类与类之间的关系，是对系统中各种对象类及其相互关系的静态结构描述。类图是面向对象建模中最常用的图。
- **对象图**：描述在某一时刻系统中一组对象及它们之间的联系，是类图的一个实例和快照。

### 1.2 类与对象的概念

**对象**是构成世界的一个独立单位，具有自己的静态特征和动态特征：
- **静态特征**：可以用某种数据来描述的属性
- **动态特征**：对象所表现的行为或对象所具有的功能

从一般意义上讲，对象是现实世界中一个实际存在的事物，它可以是**有形的**（比如一辆汽车），也可以是**无形的**（比如一项计划）。

**类**是具有相同属性和服务的一组对象的集合。类是对象的抽象，对象是类的实例。

**举例说明**（银行业务系统）：
- "张三的账户"（对象）：记录了张三的储蓄金额
- "李四的账户"（对象）：记录了李四的储蓄金额
- "客户的账户"（类）：记录了某个客户的储蓄金额

## 2. 核心概念

### 2.1 类的名字

类名有两种命名形式：
- **Simple name（简单名）**：就是简单的类的名字，如 `Account`
- **Path name（路径名）**：包括了包名的类的名字，如 `Banking::CheckingAccount`

### 2.2 类的属性

属性用来描述该类的对象所具有的**静态特征**：
- 一个类可以有一个或多个属性，或者根本没有属性
- 属性的值可以描述对象的状态
- 类的属性分为两种：一种属性值可以被其他对象存取；另一种属性值只能被本对象的操作所存取

**属性的语法格式**：

```
可见性 属性名 [: 类型] ['[' 多重性 ']'] [= 初始值] [{ 约束特性 }]
```

- **类型**：表示该属性的数据类型
- **多重性**：表示该属性值有一个或多个
- **约束特性**：描述了对属性取值的修改的限制，有3种预定义的属性可变性：
  - `changeable`（可变的）
  - `addOnly`（只可加）
  - `frozen`（冻结的）

**可见性**：

| 符号 | 名称 | 说明 |
|------|------|------|
| `+` | 公有属性（Public） | 能够被系统中其他任何操作查看和使用 |
| `-` | 私有属性（Private） | 仅在类内部可见，只有类内部的操作才能存取该属性 |
| `#` | 受保护属性（Protected） | 供类中的操作存取，也可被其子类使用 |

### 2.3 类的操作

操作用来描述该类的对象所具有的**动态特征**：
- 一个类可以有一个或多个操作，或者根本没有操作
- 操作描述对数据的具体处理方法，存取或改变属性值、执行某个动作都是操作
- 类的操作所提供的服务分为两种：一种是操作的结果引起了对象状态的改变；另一种是为服务的请求者提供返回结果

**操作的语法格式**：

```
可见性 操作名 (参数列表) [: 返回类型] [{ 约束特性 }]
```

- **参数列表**：操作被调用时接收传递过来的参数值的变量
- **返回类型**：指定了由操作返回的数据类型
- **约束特性**：说明该操作的一些有关信息，例如 `{query}` 表示该操作不会修改系统的状态

操作的可见性与属性相同：`+`（公有）、`-`（私有）、`#`（受保护）。

### 2.4 类的作用域

类的作用域分为两种：

| 作用域 | 说明 | 标识 |
|--------|------|------|
| **对象作用域** | 此类的每个对象都有一个自己的副本，彼此不共享属性值 | 无下划线 |
| **类作用域** | 只有一个副本，类的所有对象共享此副本（共享属性值） | 加下划线 |

**示例**：
- `mcount` 属性具有类作用域（下划线标识）
- `title` 属性具有对象作用域
- `getCount()` 操作具有类作用域（下划线标识）
- `getTitle()` 操作具有对象作用域

### 2.5 类的职责

类的职责指的是对该类的功能或服务的抽象概述。一个职责可能对应几个操作。

### 2.6 类之间的关系

#### 2.6.1 关联关系（Association）

关联关系表示一个类的实例与另一个类的实例在结构上的静态联系：
- 一个类的属性的数据类型是另一个类的定义
- 一个类的部分对象与另一个类的部分对象存在属性值上的联系
- **关联关系一旦建立，系统运行与否它都存在**

**代码示例**：
```java
public class A {
    public B theB;   // 属性的数据类型是另一个类B
    public A() { }
}

public class B {
    private int age;
    public B() { }
}
```

#### 2.6.2 依赖关系（Dependency）

依赖关系描述的是一个类的结构上的变化会影响到另一个类（代码级）：
- 一个类的方法的**参数**的数据类型是另一个类的定义
- 一个类的方法使用了另一个类的属性
- 一个类的方法调用了另一个类的方法
- 一个类的方法实例化了另一个类的对象

**代码示例**：
```java
public class Schedule {
    public int schecode;
    public add(Course c) { }  // 参数类型是Course类
    public void method() {
        Course c = new Course();   // 实例化Course对象
        this.schecode = c.code;    // 使用Course的属性
        c.find();                  // 调用Course的方法
    }
}
```

#### 2.6.3 关联关系与依赖关系的对比

| 对比项 | 关联关系 | 依赖关系 |
|--------|----------|----------|
| 稳定性 | 稳定的、静态的结构联系 | 临时的、动态的使用关系 |
| 存在形式 | 一个类的属性数据类型是另一个类 | 一个类的方法中使用另一个类 |
| 生命周期 | 系统运行与否都存在 | 仅在方法执行期间存在 |
| UML表示 | 实线 + 关联名/角色/多重性 | 虚线箭头 |

#### 2.6.4 泛化关系（Generalization）

泛化关系是一般类与特殊类之间的继承关系（代码级）：
- 一般类定义了共同的属性和方法
- 特殊类继承了一般类的属性和方法
- 特殊类还可以定义自己的属性和方法

**代码示例**：
```java
public class 公司人员 {
    public string 姓名;
    public string 身份证号;
}

public class 股东 extends 公司人员 {
    public int 股票数;
    public addstock() { }
}
```

#### 2.6.5 聚合（聚集）关系（Aggregation）

聚合是一种特殊形式的关联，表示类之间**整体与部分**的关系：
- 整体与部分**没有相同的生存期**
- 整体不存在了，部分仍能存在

**示例**：班级与学生的关系——班级不存在了，学生仍然存在。

#### 2.6.6 组成（组合）关系（Composition）

组合是一种特殊形式的关联，表示类之间**整体与部分**的关系：
- 整体与部分**有相同的生存期**
- 整体不存在了，部分也不能存在

**示例**：窗口与按钮的关系——窗口销毁了，按钮也不存在了。

#### 2.6.7 聚合与组合的对比

| 对比项 | 聚合关系 | 组合关系 |
|--------|----------|----------|
| 生存期 | 整体与部分没有相同生存期 | 整体与部分有相同生存期 |
| 部分独立性 | 整体不存在，部分仍存在 | 整体不存在，部分也不能存在 |
| UML表示 | 空心菱形 | 实心菱形 |
| 典型示例 | 班级-学生 | 窗口-按钮、订单-订单项 |

### 2.7 关联关系的相关概念

#### 2.7.1 关联名

可以给关联加上关联名来描述关联的作用，以便和其他关联关系相区别。

例如，`Company` 和 `Person` 类之间的关联如果不使用关联名，则可以有多种解释，Person 类可以表示公司的客户、雇员或所有者等。

#### 2.7.2 关联角色

关联关系两端的类的对象在对方类里的标识称为角色。

例如，Company 以 employer 的角色、Person 以 employee 的角色参与关联。

#### 2.7.3 多重性

多重性表示可以有多少个对象参与该关联。

常见多重性表示：
- `1`：恰好一个
- `0..1`：零或一个
- `*` 或 `0..*`：零或多个
- `1..*`：一或多个

**示例**：雇主（公司）可以雇佣多个雇员（`0..n`）；雇员只能被一家雇主雇佣（`1`）。

#### 2.7.4 关联类

如果在具有关联关系的类中，存在一个属性放在哪个类中都不合适的情况，就可以考虑使用关联类。

例如，`Contract` 类是一个关联类，其中有属性 salary，该属性既不是 Company 类的属性，也不是 Person 类的属性。

#### 2.7.5 关联约束

对于关联可以加上一些约束，以加强关联的含义。

例如，Account 类或者与 Person 类有关联，或者与 Corporation 类有关联，但不能同时与两者都有关联（用 `{xor}` 约束）。

#### 2.7.6 限定关联

在关联端紧靠源类图标处可以有限定符。限定符的作用就是在给定关联一端的一个对象和限定符值以后，可确定另一端的一个对象或对象集。

**示例**：
- 一个 person 可以在 bank 中有多个 account
- 给定了一个 account 值后，就可以对应一个 person 值
- 限定符可以理解为一种关键字，用关键字把所有的对象分开，有利于提高查询效率
- 限定关联常用于一对多或多对多的关联关系中，利用限定关联可以把模型中的**多重性从一对多变成一对一**
- **注意**：限定符是关联的属性，而不是类的属性

#### 2.7.7 自反关联

一个类与自身的关联。自反关联虽然只有一个被关联的类，但有两个关联端，每个关联端的角色不同。

**示例**："人"类的自返关联"结婚"。一个人与另一个人结婚，必然一个扮演丈夫角色，另一个扮演妻子角色。

#### 2.7.8 N元关联（N-ary Association）

3个以上类之间的关联。在同一时刻，每个类必须确定一个对象。

**示例**：一个运动员（player）在某一个年份（year）中在某一个运动队（team）中服役。

### 2.8 抽象类与接口

#### 2.8.1 抽象类

抽象类（abstract class）是不能直接产生实例的类：
- 抽象类可以有属性，但必然存在某些方法只是声明，没有具体实现
- 其他方法可以有实现
- 抽象类一般作为超类（或基类、父类）存在，用于描述其他类的公共属性和操作

#### 2.8.2 接口

接口（interface）定义为一个类对外可见的一组操作的规范：
- 接口包含操作但**不包含属性**
- 所有的操作只是声明，没有具体实现
- 不能对接口实例化

#### 2.8.3 实现关系

一个类提供了另一个类对外公布的操作规范的实现代码：
- 接口是一个特殊的类（无属性，有操作）
- 实现接口操作的类（**实现类**）与接口之间存在**实现关系**
- 调用接口操作的类（**使用类**）与接口之间存在**依赖关系**

#### 2.8.4 抽象类与接口的对比

| 对比项 | 抽象类 | 接口 |
|--------|--------|------|
| 方法实现 | 有些方法可以提供实现代码 | 所有方法都没有实现代码 |
| 属性 | 可以有属性 | 不能有属性 |
| 继承/实现 | 只能被继承 | 只能被实现 |
| 实例化 | 不能直接实例化 | 不能实例化 |
| 扩展性 | 提供实现代码的其他类算法不同，可扩展系统功能 | 同左 |

### 2.9 类的版型（Stereotype）

UML中主要的类版型有三种，引入这三种版型有助于分析人员和设计人员确定系统中的类。

#### 2.9.1 实体类（Entity Class）

- 用于保存需要放进持久存储体的信息
- 所谓持久存储体就是数据库、文件等可以永久存储数据的介质
- 实体类的对象是**永久性**的，它的生存时间长于会话生命周期

#### 2.9.2 边界类（Boundary Class）

- 位于系统与外界的交界处，是系统内的对象和系统外的参与者的联系媒介
- 外界的消息只有通过边界类的对象实例才能发送给系统
- **典型例子**：窗体（form）、对话框（dialog box）、报表（report）、表示通讯协议（如TCP/IP）的类、直接与外部设备交互的类、直接与外部系统交互的类等
- 通过用例图可以确定需要的边界类，每个 **actor/use case 对**至少要有一个边界类

#### 2.9.3 控制类（Control Class）

- 是负责其他类工作的类
- 根据业务规则，执行相应脚本流，以委托责任的形式向其他类发出消息，由其他类来实现具体的功能
- 控制类的一个主要用途是**协调边界类和实体类之间的交互**

#### 2.9.4 基于协作的类图

边界类、控制类、实体类三者协同工作，可以完成一个用例的功能：
- 根据用例描述的复杂性，可能需要多个控制类，也可能牵涉到多个实体类
- 在基于协作的类图上，将依赖关系改成链（关联关系的实例），再在链上附着消息，就可以得到协作图

### 2.10 对象图

#### 2.10.1 对象图的概念

对象图（object diagram）描述的是参与交互的各个对象在交互过程中某一时刻的状态：
- 对象图是系统的详细状态在某一时刻的快照
- 常用于表示复杂的类图的一个实例
- 对象是类的实例，对象之间的链是类之间的关联的实例
- 对象图实质上是具有关联关系的**类图的实例**

#### 2.10.2 类图与对象图的对比

| 对比项 | 类图 | 对象图 |
|--------|------|--------|
| 组成部分 | 类名、属性、操作（三部分） | 对象名、属性（两部分） |
| 名称栏 | 只包含类名 | 包含对象名和类名（如 `a1:Account`） |
| 属性栏 | 定义了所有属性的特征 | 定义了属性的**当前值** |
| 操作 | 列出了操作 | 不包含操作（同类对象操作相同） |
| 连接 | 关联连接（含关联名、角色、约束等） | 链连接（含名称、角色） |
| 抽象程度 | 通用情况的描述 | 特定情况的描述 |

#### 2.10.3 对象的状态与身份

对象是类的实例，创建一个对象时通常可以从两种情况来观察：
- **第一种**：将对象作为一个实体，它在某个时刻有明确的值
- **第二种**：作为一个身份持有者，不同时刻有不同的值

一个对象在系统的某一个时刻应当有其自身的状态。通常这个状态使用**属性的赋值**或分布式系统中的位置来描述。对象通过链和其他对象相联系。

## 3. 方法速通

### 3.1 识别类的方法

- **参考问题域中的术语**：从业务领域中提取名词
- **名词短语分析法**：从需求描述中提取名词短语
- **常识中的客观实体**：识别现实世界中的实体
- **CRC（类-责任-协作者）方法**：通过分析类的责任和协作者来识别类

**CRC示例**：
| 项目 | 内容 |
|------|------|
| 类名 | 订单 |
| 类的类型 | 可感知的物理实体 |
| 类的特征 | 有形的 |
| 责任 | 检查库存项、核查有效付款、发往交付地址 |
| 协作者 | 订单行、库存项、付款 |

### 3.2 识别属性的方法

- 参考系统的职责
- 定语用的词汇
- 常识中的客观实体的特征
- 对象需要保存的信息
- 对象需要区别的状态

### 3.3 识别操作的方法

- 参考系统的职责
- 常识中的客观实体的功能
- 对象需要保存的信息的处理方法
- 需要与其他对象发送的消息

### 3.4 判断类之间关系的方法

**判断关联关系**：
- 一个类的属性的数据类型是否是另一个类？
- 两个类的对象之间是否存在结构上的静态联系？

**判断依赖关系**：
- 一个类的方法的参数类型是否是另一个类？
- 一个类的方法是否使用了另一个类的属性？
- 一个类的方法是否调用了另一个类的方法？

**判断泛化关系**：
- 是否存在"是一个（is-a）"关系？
- 子类是否继承了父类的所有属性和方法？

**判断聚合关系**：
- 是否存在"有一个（has-a）"关系？
- 部分能否脱离整体独立存在？

**判断组合关系**：
- 部分是否与整体同生共死？
- 整体销毁后部分是否也随之销毁？

## 4. 典型示例

### 4.1 鸭子模拟游戏

**问题描述**：某游戏公司欲开发一款面向儿童的模拟游戏，模拟现实世界中各种鸭子的发声特征、飞行特征和外观特征。

| 鸭子种类 | 发声特征 | 飞行特征 | 外观特征 |
|----------|----------|----------|----------|
| 灰鸭（MallardDuck） | Quack | 用翅膀飞行（FlyWithWings） | 灰色羽毛 |
| 红头鸭（RedHeadDuck） | Quack | 用翅膀飞行（FlyWithWings） | 灰色羽毛红头 |
| 棉花鸭（CottonDuck） | 不发声（QuackNoWay） | 不能飞行（FlyNoWay） | 白色 |
| 橡皮鸭（RubberDuck） | 橡皮与空气摩擦声（Squeak） | 不能飞行（FlyNoWay） | 黑白橡皮色 |

**一般的面向对象方法**：
- 定义"鸭子"类（超类、父类），定义"外观特征"属性，定义"发声"操作、"飞行"操作
- 定义各种子类，继承父类，重置操作

**策略设计模式**：为了支持将来能够模拟更多种类鸭子的特征，可以采用**策略设计模式**，将发声和飞行行为抽取为接口。

### 4.2 银行业务系统

**账户的静态特征**：储户姓名、密码、银行账号、联系电话、储蓄金额
**账户的动态特征**：注销、冻结、解冻、查询余额

**交易清单的静态特征**：银行账号、交易类别、交易金额、交易时间
**交易清单的动态特征**：存款、取款、查询交易记录

### 4.3 图书馆系统

系统需求：
- 可以为读者提供借阅服务，读者包括学生、老师、研究人员
- 所有读者在借阅之前需要凭借书证
- 管理员将录入读者信息，并将系统登录信息通过邮件发送给读者
- 读者可以凭借书证借阅书籍
- 老师及研究人员可以借阅N本书，每本书的期限为M天
- 学生可以借阅K本书，每本书的期限为L天
- 若借阅书籍数量达到上限或有书籍超期未还，将不能借阅新的书籍
- 若书籍超期未还将产生罚款单
- 读者可以登录系统修改自身信息，查询馆藏书籍信息、借阅记录、使用到书提醒服务
- 管理员可以通过系统管理馆藏书籍

**涉及的类**：读者、学生、老师、研究人员、借书证、管理员、读者信息、书籍信息、借阅记录、罚款单、到书提醒、邮件

### 4.4 图形编辑器（组合模式）

图形编辑器具有基本图形（如直线、矩形），用户可以直接使用基本图形画图，也可以把基本图形组合在一起创建复杂图形。

**类设计**：
- `Graphics`：基本图形和组合图形的父类，声明了所有图形共同的操作（如 Draw），也声明了专用于组合图形管理子图形的操作（如 Add、Remove）
- `Line`、`Rectangle`：基本图形类
- `GroupGraphics`：组合图形类，与父类有组合关系

### 4.5 员工与地址的关系分析

**Employee 类**包含：empcode（职工代码）、name（名字）、address（地址）、birthday（生日）

### 5.2 人员角色设计

**问题**：一个人可以有一种或多种角色。例如一个研究生是学生，同时可以在学院做助教（TeachingAssistant），也可以在校园餐厅打工做收银员（Cashier）。

**设计方案分析**：
- **(a) 同一时刻只拥有一种角色**：人与角色之间是泛化关系，但不灵活
- **(b) 本质上可以归类为策略设计模式**：将角色抽象为接口
- **(c) 同一时刻拥有三种角色**：人与角色之间是关联关系（一对多）
- **(d) 同 (c)**

### 5.3 库存管理系统扩展设计

**原始需求**：某库存管理系统的类图。
**新需求**：
1. 对已经损坏（damaged）的货物的价格进行打折
2. 可以按货物的大小和颜色对货物进行查找

**四种方案分析**：

| 方案 | 做法 | 评价 |
|------|------|------|
| 方案(a) | 增加 InventoryProduct 的属性和方法 | **不妥**：产品目录类对应多种产品，添加 isDamaged() 会导致每种类型的产品要么全好要么全坏 |
| 方案(b) | 增加 PhysicalProduct 类，与 InventoryProduct 建立关联 | **正确**：具体产品对象可以判断是否损坏，可添加 size、color 属性（工厂方法设计模式的变形） |
| 方案(c) | 增加 Inventory 的属性和方法 | **不妥**：目录类是总纲目，中间隔了一层，不能对具体产品操作 |
| 方案(d) | 同时增加 InventoryProduct 和 Inventory 的属性和方法 | **不妥**：方案(a)和(c)的综合，更不合理 |

### 5.4 Transaction 创建责任

**问题**：在书店库存管理系统中，LineItem、Station、Payment、Sale 四个类中，由谁负责创建 Transaction 类最合理？

**分析**：
- **LineItem、Payment**：与 Transaction 是多对1的关系，会增加对象间的耦合度
- **Sale**：是 Transaction 的子类，通常不由子类负责创建父类
- **Station**：与 Transaction 之间是1对多的关系，Transaction 对象只能匹配一个 Station 对象

**结论**：由 **Station** 负责创建 Transaction 类最合理。

## 7. 总结复盘

### 7.1 本章核心知识点

| 知识点 | 关键内容 |
|--------|----------|
| 类的基本构成 | 类名、属性、操作、职责 |
| 属性/操作的语法 | 可见性、名称、类型、参数、返回值 |
| 作用域 | 对象作用域（不共享）vs 类作用域（共享，下划线标识） |
| 关联关系 | 静态结构联系，通过属性建立 |
| 依赖关系 | 临时使用关系，通过方法参数/调用建立 |
| 泛化关系 | 继承关系，is-a 关系 |
| 聚合关系 | 整体-部分，部分可独立存在（空心菱形） |
| 组合关系 | 整体-部分，部分不可独立存在（实心菱形） |
| 关联类 | 解决属性归属不明确的问题 |
| 限定关联 | 用限定符降低多重性 |
| 自反关联 | 类与自身的关联 |
| 抽象类 | 不能实例化，可有属性和部分实现 |
| 接口 | 不能实例化，无属性，方法全为声明 |
| 三种版型 | 实体类（持久存储）、边界类（系统边界）、控制类（协调逻辑） |
| 对象图 | 类图的实例快照，显示某一时刻的对象状态 |

### 7.2 常见错误与易混淆点

**错误1：混淆关联关系和依赖关系**
- 关联关系是结构性的、静态的、持久的
- 依赖关系是使用性的、动态的、临时的
- 判断方法：关联通过属性体现，依赖通过方法体/参数体现

**错误2：混淆聚合和组合**
- 聚合：部分可以独立于整体存在（如班级和学生）
- 组合：部分不能独立于整体存在（如订单和订单项）
- 判断方法：问"整体销毁后，部分还能存在吗？"

**错误3：滥用继承（泛化关系）**
- 不要把所有的"is-a"都用继承实现
- 当子类需要频繁切换行为时，考虑使用策略模式（组合优于继承）

**错误4：忽略多重性**
- 关联关系必须标注多重性
- 多重性直接影响数据库设计和代码实现

**错误5：在对象图中列出操作**
- 对象图中的对象不包含操作，因为同类对象的操作相同
- 对象图只显示对象名和属性的当前值

**错误6：混淆限定符归属**
- 限定符是**关联的属性**，而不是类的属性
- 限定符放在关联端靠近源类的位置

**错误7：将实体类的属性直接作为类名**
- 实体类的名称应该是业务领域的实体，而非属性值
- 例如应该用"学生"类，而不是"张三"类

### 7.3 两种类图的理解

本章要求理解两种类图：
1. **反映实体类之间关联关系的类图**：描述系统中业务实体之间的静态关系
2. **反映用例实现的基于协作的类图**：由边界类、控制类、实体类协作完成一个用例的功能

### 7.4 记忆口诀

- **关系判断口诀**：属性关联、方法依赖、继承泛化、整体聚合组合
- **聚合组合区分**：聚合"可以拆"，组合"拆不了"
- **三种版型**：实体存数据、边界接外界、控制来协调
- **对象图特征**：类图的实例、特定时刻的快照、只含对象名和属性值
