# 第07章 组件图与部署图

## 1. 概念引入

### 1.1 什么是物理模型？

在软件系统的分析与设计中，我们不仅要关注系统的**逻辑部分**（类图、对象图、用例图、顺序图等），还要关注系统的**物理部分**。

**物理模型**（也称**体系结构模型**或**实现模型**）描述的是系统在物理层面的组织方式，包括：

- **组件图**（Component Diagram）：描述系统中软件组件及其相互关系
- **部署图**（Deployment Diagram）：描述系统中硬件节点的拓扑结构以及软件在硬件上的部署方式

### 1.2 为什么需要组件图与部署图？

逻辑模型回答的是"系统做什么"，而物理模型回答的是"系统怎么实现"。具体来说：

- **组件图**：确定哪些类放在一个源代码文件中，由谁来编写，组件之间如何组装
- **部署图**：确定软件组件运行在哪些硬件节点上，节点之间如何通信

用一个形象的比喻：如果说类图是"设计图纸"，那么组件图就是"零件清单"，部署图就是"安装说明书"。

## 2. 核心概念

### 2.1 组件（Component）

#### 2.1.1 组件的定义

组件也称为**构件**，是系统中遵从一组接口且提供其实现的**物理的、可部署的、可替换的**部分。

组件的核心特征：

| 特征 | 说明 |
|------|------|
| 物理性 | 组件是实际存在的物理文件（如 .java、.cpp、.dll） |
| 可部署 | 组件可以被安装到某个节点上运行 |
| 可替换 | 组件可以通过接口被其他组件替换（"软插件"理念） |
| 可组装 | 组件内部包含实现元素（类、接口等），可与其他组件组装成系统 |

#### 2.1.2 组件的三种分类

| 类型 | 说明 | 举例 |
|------|------|------|
| **部署组件** | 运行系统需要配置的组件 | Java虚拟机、XML文件、JAR文件、DLL文件、数据库表 |
| **工作产品组件** | 用来产生部署组件的中间产物 | 源代码文件、数据文件 |
| **执行组件** | 系统执行后得到的组件 | EJB、动态Web页面、.exe文件、COM+对象、CORBA对象 |

#### 2.1.3 组件的具体类型

| 组件类型 | 说明 |
|----------|------|
| **数据库** | 存储数据的物理单元 |
| **包** | 提供公共视图的包 |
| **主程序** | 系统程序的根文件，指定系统的入口 |
| **子程序** | 单独处理的元素的包 |
| **子程序规范** | 源代码文件中的声明文件（.h文件） |
| **子程序体** | 源代码文件中的实现文件（.cpp文件） |
| **包规范** | 存放所有源代码文件中声明文件的包 |
| **包体** | 存放所有源代码文件中实现文件的包 |
| **任务规范** | 拥有独立控制线程的组件的规范 |
| **任务体** | 拥有独立控制线程的组件的实现体 |
| **标准组件** | 除特殊组件外的一切组件，包括自定义构造型的组件 |

### 2.2 组件之间的关系

#### 2.2.1 依赖关系（最主要）

组件之间的关系主要是**依赖关系**，包括编译、链接或执行时组件之间的依赖关系。

**依赖关系的推导规则**：

```
若：组件A中包含类X，组件B中包含类Y
且：在类图中，类X依赖于类Y
则：在组件图中，组件A依赖于组件B
```

#### 2.2.2 其他关系

除依赖关系外，组件之间也可以存在**关联关系**、**泛化关系**、**实现关系**等，这些关系的确定主要依赖于不同组件中包含的实现元素之间的关系。

### 2.3 组件和类的关系

组件和类的关系是**物理设计与逻辑设计**之间的桥梁：

- **类**是逻辑概念（"虚"的概念），是分析与设计阶段的产物
- **组件**是物理概念（"实"的概念），是实现阶段的产物

**关键理解**：

设计时将类图中的类**分配**到需要创建的物理组件上，以达到由逻辑设计向物理设计的过渡。分配时可以根据类在行为上的**联系程度**进行合理分配。

**举例说明**：

用 C++ 语言开发的系统来理解：一个 `.CPP` 文件可能包含几个类的定义。这个 `.CPP` 文件就是一个**组件**，而其中包含的几个**类**就是该组件包含的实现元素。

**一个组件包含一个类 vs 包含多个类**：

| 情况 | 组件关系 |
|------|----------|
| 一个组件只包含一个类 | 不同类之间的关系决定组件之间的关系 |
| 一个组件包含多个类 | 虽然这些类之间存在复杂关系，但组件本身是孤立的，不存在组件的自反关系 |

### 2.4 组件和接口的关系

- UML 中用组件表示将**类和接口**等逻辑元素打包而形成的物理模块
- 定义组件的目的之一是使用组件来**组装系统**，用另一个组件或改进过的组件**取代**一个组件
- 因此，组件必须为支持确定的**接口**而定义
- **接口**是一个组件提供给其他组件的一组操作
- 在组件**重用**和组件**替换**上，接口是一个很重要的概念

### 2.5 节点（Node）

#### 2.5.1 节点的定义

节点也称为**结点**，是存在于运行时的代表计算资源的物理元素。节点一般都具有一些内存，而且常常具有处理能力。节点上可以安置（多个）组件，即组件放在节点上运行。

#### 2.5.2 节点的类型

| 类型 | 说明 | 举例 |
|------|------|------|
| **处理机**（Processor） | 可以执行程序的硬件设备，具有计算能力 | UNIX主机、PC机、打印机、传感器 |
| **设备**（Device） | 无计算能力的硬件设备 | 调制解调器、终端 |

#### 2.5.3 节点之间的关系

节点通过**通信关联**相互连接，表示两个硬件之间的关联关系。它指出节点之间只存在着某种通信路径，并指出通过哪条通信路径可使这些节点交换对象或发送消息。

由于连接关系是关联，所以可以像类图那样，在关联上加**角色、多重性、约束、版型**等。

**示例**："计算机"节点和"显示设备"节点之间采用 RS-232 串行口连接，`《RS-232》` 就是一个关联名。

## 3. 方法速通

### 3.1 组件图的设计步骤

```
步骤1：建立类图模型
    ↓
步骤2：建立组件图模型
    ↓
步骤3：将类分配到组件中（根据类的行为联系程度）
    ↓
步骤4：确定组件之间的关系（依赖、关联等）
    ↓
步骤5：添加接口（支持组件重用和替换）
    ↓
步骤6：验证组件图与类图的一致性
```

### 3.2 部署图的设计步骤

```
步骤1：确定系统中的节点（处理机和设备）
    ↓
步骤2：确定节点之间的通信关联（通信协议）
    ↓
步骤3：将组件分配到节点上
    ↓
步骤4：标注节点上的组件
    ↓
步骤5：验证部署图与组件图的一致性
```

### 3.3 正向工程（Forward Engineering）

正向工程就是根据模型（类图和组件图）来产生**源代码框架**。

**操作流程**：

1. 把建模语言设为 Java（Tools → Options → Notation 中设置）
2. 在类图中创建类（public 类型的类、private 类型的类等）
3. 在组件图中添加一个组件（Java 中规定 public 类的名字必须和所在的文件名一致）
4. 右击组件 → Open Standard Specification → Realizes 标签 → 选择类 → Assign
5. 右击组件 → Java/J2EE → Generate Code
6. 选择 CLASSPATH（生成代码的目录）→ Assign → OK

### 3.4 逆向工程（Reverse Engineering）

逆向工程指的是从**源代码**逆向得到模型（类图和组件图）。

**操作流程**：

1. 选 Tools → Java/J2EE → Reverse Engineer...
2. 找到源代码文件，按 Add 按钮加入（可逐个加入或 Add All 一次性加入）
3. 按 Select All 按钮，再按 Reverse 按钮
4. Rose 开始进行逆向工程，得到组件和类
5. 将组件拖动到组件图中，将类拖动到类图中，自动显示关系

### 3.5 组件图的四大用途

| 用途 | 说明 |
|------|------|
| **对源代码文件之间的关系建模** | 分析源代码文件（组件）之间的依赖关系 |
| **对可执行文件之间的关系建模** | 分析 .exe、.dll 等可执行文件之间的关系 |
| **对物理数据库中具体对象之间的关系建模** | 分析数据库表等物理存储对象之间的关系 |
| **对自适应系统建模** | 分析可动态替换组件的系统 |

### 3.6 部署图的四大用途

| 用途 | 说明 |
|------|------|
| **对分布式系统的物理建模** | 分析分布式系统中各节点的拓扑结构 |
| **对外设配置的物理建模** | 分析打印机、传感器等外部设备的连接方式 |
| **对互联网的物理建模** | 分析 Web 服务器、应用服务器、数据库服务器等的部署 |
| **详细说明节点上运行的组件** | 标注每个节点上运行的具体组件 |

## 4. 典型示例

### 4.1 示例一：超市购买商品系统的组件图

**场景描述**：

顾客可以通过页面访问系统，具体的顾客服务处理过程依赖于顾客服务程序：

- 若顾客需要**查询订单信息**，则要通过**订单查询程序**获得服务
- 若**建立订购单**，则需要通过**订单处理程序**获得服务
- 在**提交订单**时，还需要建立付款记录，因此依赖于**支付处理程序**

这些程序都以组件形式构造，它们都依赖于**数据库管理程序**。

**组件依赖关系**：

```
[页面] → [顾客服务程序]
              ↓
     [订单查询程序] ← [订单处理程序] → [支付处理程序]
              ↓              ↓              ↓
         [数据库管理程序] ← [数据库管理程序] ← [数据库管理程序]
```

### 4.2 示例二：Java 数据库应用的组件图（DAO 模式）

**场景描述**：

采用 DAO（Data Access Object，数据访问对象）模式的简单 Java 数据库应用。

**组件关系**：

- `BookReviewApplication` 使用组件 `BookReviewPanel`，为应用提供 GUI
- `BookReviewApplication` 使用驻留在组件 `BookReviewData` 中的接口
- `BookReviewData` 由组件 `BookReviewMySQLDAO` 实现
- `BookReviewMySQLDAO` 是 DAO，提供对存放在关系数据库 MySQL 中的图书审查数据的访问

**关键设计思想**：

可以为不同的关系数据库提供不同的 DAO，只要它们实现了接口 `BookReviewData`。这体现了**接口**在组件重用和替换中的重要作用。

### 4.3 示例三：基于 Struts2 的个人信息管理系统

**组件识别方法**：

从目录结构中识别组件：

```
Ch13\WebRoot\login\
    index.jsp
    register.jsp
Ch13\src\edu\login\Action\
    LoginAction.java
    RegisterAction.java
```

**组件关系推导**：

- `index.jsp` 只负责从外部接收数据，里面说明了要依赖 `LoginAction.java`（`<s:form action="loginAction">`）
- `LoginAction.java` 中只定义了一个 `LoginAction` 类，该类继承了 `ActionSupport`，实现了 `ServletRequestAware` 接口

以此类推，可以获得所有的组件。根据组件中所包含的类之间的依赖关系，可以画出组件图。

### 4.4 示例四：EJB 组件的部署

**部署决策**：

- 客户使用什么硬件？→ Windows 2000 PC
- 服务器使用什么硬件？→ Windows 2000 PC
- EJB 容器使用什么软件？→ JBoss

然后将 EJB 的各组件（实体 Bean、会话 Bean 等）部署到相应的节点上。

## 5. 真题/实战

### 5.1 实战案例：网上书店系统

**需求**：为一个网上书店系统设计组件图和部署图。

**组件图设计**：

```
[用户界面组件]
       ↓
[用户管理组件] ←→ [订单管理组件] ←→ [库存管理组件]
       ↓                ↓                ↓
[用户DAO组件]    [订单DAO组件]    [库存DAO组件]
       ↓                ↓                ↓
       └──────── [数据库连接组件] ────────┘
                       ↓
               [MySQL数据库]
```

**部署图设计**：

```
[客户端浏览器] --HTTP--> [Web服务器:Tomcat]
                              |
                         [应用服务器]
                              |
                    [数据库服务器:MySQL]
```

### 5.2 实战案例：在线支付系统

**组件识别**：

| 组件名称 | 类型 | 说明 |
|----------|------|------|
| PaymentGateway | 执行组件 | 支付网关接口 |
| TransactionManager | 执行组件 | 事务管理器 |
| SecurityModule | 部署组件 | 加密/解密模块 |
| DatabaseAdapter | 工作产品组件 | 数据库适配器 |
| LogService | 部署组件 | 日志服务 |

**组件间依赖**：

```
[PaymentGateway] → [TransactionManager] → [SecurityModule]
                        ↓
                [DatabaseAdapter] → [LogService]
```

### 5.3 组件设计的常见模式

**分层架构模式**：

```
表现层组件（JSP、HTML）
    ↓
业务逻辑层组件（Action、Service）
    ↓
数据访问层组件（DAO）
    ↓
数据库组件（MySQL、Oracle）
```

**MVC 模式**：

```
View（JSP组件） → Controller（Action组件） → Model（Service组件）
                                                    ↓
                                              DAO组件 → 数据库组件
```

## 6. 巩固练习

### 练习题一：概念理解

**题目**：简述组件与类的关系，以及组件图与类图的区别。

**参考答案**：

- **关系**：类是逻辑概念，组件是物理概念。设计时将类分配到组件中，一个组件可以包含一个或多个类。
- **区别**：类图描述系统的逻辑结构（类、属性、方法、关系），组件图描述系统的物理结构（源代码文件、可执行文件、数据库等及其关系）。
- **过渡**：组件图是将逻辑设计转化为物理设计的桥梁。

### 练习题二：组件识别

**题目**：某系统有以下源代码文件，请识别组件并分析组件间的关系。

```
User.java - 定义了 User 类
Order.java - 定义了 Order 类
UserService.java - 定义了 UserService 类（依赖 User 类）
OrderService.java - 定义了 OrderService 类（依赖 Order 类和 User 类）
```

**参考答案**：

组件识别：
- 组件A：User.java（包含 User 类）
- 组件B：Order.java（包含 Order 类）
- 组件C：UserService.java（包含 UserService 类）
- 组件D：OrderService.java（包含 OrderService 类）

组件间关系：
- 组件C 依赖 组件A（UserService 依赖 User）
- 组件D 依赖 组件B（OrderService 依赖 Order）
- 组件D 依赖 组件A（OrderService 依赖 User）

### 练习题三：部署图设计

**题目**：为一个三层架构的 Web 应用设计部署图，要求包含客户端、Web 服务器、应用服务器和数据库服务器。

**参考答案**：

```
节点：
1. 客户端（PC机） - 运行浏览器组件
2. Web服务器（Apache） - 运行静态页面组件
3. 应用服务器（Tomcat） - 运行 Servlet/JSP 组件、EJB 组件
4. 数据库服务器（MySQL Server） - 运行数据库组件

通信关联：
- 客户端 --HTTP--> Web服务器
- Web服务器 --AJP--> 应用服务器
- 应用服务器 --JDBC--> 数据库服务器
```

### 练习题四：正向工程与逆向工程

**题目**：简述正向工程和逆向工程的区别及其应用场景。

**参考答案**：

| 对比项 | 正向工程 | 逆向工程 |
|--------|----------|----------|
| 方向 | 模型 → 源代码 | 源代码 → 模型 |
| 输入 | 类图、组件图 | 源代码文件 |
| 输出 | 源代码框架 | 类图、组件图 |
| 应用场景 | 新系统开发，从设计到编码 | 遗留系统理解，文档生成 |
| 工具支持 | Rational Rose、Enterprise Architect | Rational Rose、Enterprise Architect |

### 练习题五：综合设计

**题目**：某医院管理系统包含以下模块：挂号管理、门诊管理、药房管理、住院管理、财务管理。请为其设计组件图，要求：

1. 识别主要组件
2. 确定组件间的依赖关系
3. 考虑接口的设计

**参考答案**：

```
主要组件：
- RegistrationUI（挂号界面组件）
- OutpatientUI（门诊界面组件）
- PharmacyUI（药房界面组件）
- InpatientUI（住院界面组件）
- FinanceUI（财务界面组件）
- PatientService（患者服务组件）
- DoctorService（医生服务组件）
- MedicineService（药品服务组件）
- DataAccessLayer（数据访问层组件）
- Database（数据库组件）

接口设计：
- IPatientService：提供患者信息查询、修改接口
- IDoctorService：提供医生排班、诊断记录接口
- IMedicineService：提供药品库存、处方接口

依赖关系：
- 所有UI组件 → 各自的Service组件
- Service组件 → DataAccessLayer → Database
- RegistrationUI → PatientService
- OutpatientUI → PatientService, DoctorService
- PharmacyUI → MedicineService, PatientService
- InpatientUI → PatientService, DoctorService
- FinanceUI → PatientService
```

## 7. 总结复盘

### 7.1 核心知识点梳理

```
组件图与部署图
├── 组件图
│   ├── 组件的概念（物理的、可部署的、可替换的）
│   ├── 组件的类型（部署组件、工作产品组件、执行组件）
│   ├── 组件之间的关系（依赖关系为主）
│   ├── 组件和类的关系（逻辑→物理的过渡）
│   ├── 组件和接口的关系（支持重用和替换）
│   └── 组件图的用途（源代码、可执行文件、数据库、自适应系统）
├── 部署图
│   ├── 节点的概念（处理机、设备）
│   ├── 节点之间的关系（通信关联）
│   └── 部署图的用途（分布式系统、外设、互联网、组件部署）
└── 工具支持
    ├── 正向工程（模型→源代码）
    └── 逆向工程（源代码→模型）
```

### 7.2 常见错误与易混淆点

| 错误类型 | 正确理解 |
|----------|----------|
| 把组件等同于类 | 组件是物理概念（文件），类是逻辑概念（设计元素） |
| 忽略组件的可替换性 | 组件必须通过接口定义，支持替换 |
| 混淆节点类型 | 处理机有计算能力，设备没有计算能力 |
| 部署图只画硬件 | 部署图要同时标注节点和其上运行的组件 |
| 正向工程生成完整代码 | 正向工程只生成代码框架，需要手动添加实现逻辑 |
| 忽略组件图的用途 | 组件图不仅用于源代码建模，还可用于可执行文件、数据库等 |

### 7.3 考试重点提示

1. **组件的定义和特征**：物理的、可部署的、可替换的
2. **组件的三种分类**：部署组件、工作产品组件、执行组件
3. **组件间关系的推导**：基于组件中包含的类之间的关系
4. **组件和类的关系**：一个组件可以包含一个或多个类
5. **组件和接口的关系**：接口支持组件的重用和替换
6. **节点的类型**：处理机 vs 设备
7. **正向工程与逆向工程**：方向、输入输出、应用场景
8. **组件图和部署图的用途**：各自四种典型用途

### 7.4 一句话总结

> 组件图描述"软件由哪些零件组成"，部署图描述"这些零件安装在哪台机器上"。组件是类的物理载体，节点是组件的运行平台。
