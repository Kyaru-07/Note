# 第05章 顺序图与协作图

> **章节定位**：交互模型是UML动态行为建模的核心工具。顺序图与协作图从不同视角描述对象之间的消息交互，是将用例描述"落地"到类和对象上的关键一步。

---

## 1. 概念引入

### 1.1 为什么需要交互模型？

所有系统都可以从两个方面来描述：

| 方面 | 说明 | UML图 |
|------|------|-------|
| **静态结构** | 系统包含哪些对象类以及它们之间的关系 | 类图、对象图 |
| **动态行为** | 系统中的各个对象如何交互协作来实现系统功能，每个对象的状态变化规律 | 顺序图、协作图、状态图、活动图 |

- **顺序图**（Sequence Diagram，又称时序图）：描述对象之间**按时间顺序**进行的消息交互过程。
- **协作图**（Collaboration Diagram）：描述相互合作的对象间的**交互关系和链接关系**，强调对象的空间组织结构。

两者统称为**交互图**（Interaction Diagram），通常用来描述一个用例的行为，显示该用例中所涉及的对象和这些对象之间的消息传递情况。

### 1.2 交互模型的核心作用

交互模型可以帮助分析人员：
- 对照检查每个用例中所描述的用户需求
- 审查这些需求是否已经落实到能够完成这些功能的类中去实现
- 将用例行为分解到对象及其方法上

## 2. 核心概念

### 2.1 顺序图的基本元素

顺序图是一个二维图形：
- **横轴（对象维）**：表示参与交互的对象
- **纵轴（时间维）**：表示时间递增方向，自上而下

```
对象维（横轴）→
┌─────────┐  ┌─────────┐  ┌─────────┐
│ object1  │  │ object2  │  │ object3  │
│  :C1     │  │  :C2     │  │  :C3     │
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │  ┊          │  ┊          │  ┊
     │  ┊  oper()  │  ┊          │  ┊
     │  ╠═════════►│  ┊          │  ┊
     │  ┊          │  ┊          │  ┊
     │  ┊          │  ┊ oper1()  │  ┊
     │  ┊          │  ╠═════════►│  ┊
     │  ┊          │  ┊          │  ┊
     │  ┊          │  ┊ oper2()  │  ┊
     │  ┊          │  ╠═════════►│  ┊
     │  ┊          │  ┊          │  ┊
     ▼  ┊          ▼  ┊          ▼  ┊
  时间维（纵轴，时间递增 ↓）
```

#### 2.1.1 对象（Object）

- 同类图中的对象，是类的实例
- 表示形式：`对象名:类名`（如 `ob1:C1`），也可以只有类名 `:C1`
- 对象在顺序图顶端的位置表示其**存在状态**

#### 2.1.2 生命线（Lifeline）

- 从对象图标向下延伸的一条**虚线**
- 表示对象存在的**生命期**
- 虚线的**末端有"X"符号**：表示在本次交互活动的结束时刻，该对象会被**销毁**
- 虚线的**末端一直延伸下去**：表示在本次交互活动的结束时刻，该对象会**继续存在**

#### 2.1.3 控制焦点（Focus of Control）/ 激活期（Activation）

- 生命线上的一个**矩形条**
- 表示对象**执行一个动作的时间段**
- 本质含义：对象正在执行某个方法
- 可以通过重叠的小矩形条表示**嵌套调用**

#### 2.1.4 对象的创建与销毁

| 图形位置 | 含义 |
|---------|------|
| 顶端的对象 | 在本次交互活动的**开始时刻已经存在** |
| 箭头指向的对象 | 在本次交互活动的**过程中创建** |
| 虚线末端有"X" | 在本次交互活动的结束时刻会被**销毁** |
| 虚线末端延伸 | 在本次交互活动的结束时刻会**继续存在** |

### 2.2 消息（Message）

消息是对象间的一次通信，表示一个对象向另一个对象发送信息或请求执行某个操作。

#### 2.2.1 消息的语法格式

```
[predecessor] [guard-condition] [sequence-expression] [return-value:=] message-name ([argument-list])
```

| 组成部分 | 说明 |
|---------|------|
| `predecessor` | 必须先发生的消息的列表 |
| `guard-condition` | 消息执行前必须满足的**警戒条件** |
| `sequence-expression` | 消息顺序项的列表 |
| `return-value` | 消息的**返回值** |
| `message-name` | **消息名**（即方法名） |
| `argument-list` | 消息的**参数列表** |

#### 2.2.2 消息顺序项的语法格式

```
[integer | name] [recurrence]
```

| 组成部分 | 说明 |
|---------|------|
| `integer` | 消息序号 |
| `name` | 并发的控制线程名 |
| `recurrence` | 表示消息是条件执行或循环执行 |

各消息顺序项之间用英文句点"."连接。

#### 2.2.3 消息语法示例

```
2:display(x,y)           ← 简单消息
1.3.1:p:=find(specs)     ← 嵌套消息，消息带返回值
[x<0]4:invert(x,color)   ← 警戒条件消息或条件发送消息
4.2[x>y]:invert(x,color) ← 条件发送消息
3.1*:update()            ← 循环发送消息
A3,B4/C2:copy(a,b)       ← 线程间同步
1.1a,1.1b/1.2:continue() ← 同时发送的并发消息作为先发消息序列
```

### 2.3 消息的种类

UML定义了6种消息类型：

#### 2.3.1 调用消息（Procedure Call）

- 消息的发送者把控制传递给消息的接收者，然后**停止活动**
- **等待**消息接收者放弃或返回控制
- 可以用来表示**同步**的意义
- 一定有一个配对的返回消息（可省略不画）

```
:Sender ──────► :Receiver
            call()
```

#### 2.3.2 异步消息（Asynchronous）

- 消息的发送者通过消息把信号传递给消息的接收者，然后**继续自己的活动**
- **不等待**接收者返回消息或控制
- 异步消息的接收者和发送者是**并发工作**的

```
:Sender ──────► :Receiver
          async()
```

#### 2.3.3 返回消息（Return）

- 表示前面发出的消息要返回的消息值
- 对于**调用消息**：一定有返回消息，但为了简洁可以不画
- 对于**异步消息**：如果是过程调用，返回消息隐含可不画；如果是非过程调用，有返回消息必须明确表示

```
:Receiver ──────► :Sender
            return
```

#### 2.3.4 反身消息（Self Message）

- 消息的发送者和接收者是**同一个对象**
- 消息的箭头从对象发出又回到自身

```
         ┌──────┐
         │  │   │
         │  ▼   │
         └──────┘
:CustomerActor
```

#### 2.3.5 阻止消息（Prevention）

- 消息发送者发出消息给接收者
- 如果接收者**无法立即接收**，则发送者**放弃**这个消息

#### 2.3.6 超时消息（Timeout）

- 消息发送者发出消息给接收者并**按指定时间等待**
- 如果接收者无法在**指定时间内**接收消息，则发送者放弃这个消息

### 2.4 调用消息与异步消息的对比

这是考试重点，需要深入理解：

| 对比维度 | 调用消息 | 异步消息 |
|---------|---------|---------|
| 控制流 | 完成之前**需要中断** | 完成之前**不需要中断** |
| 发送者行为 | 停止活动，等待返回 | 继续自己的活动 |
| 返回消息 | 一定有（可省略画出） | 不一定有 |
| 并发性 | 同步 | 并发 |

**实例分析（学生成绩管理系统）：**

- **调用消息示例**：在账号和密码的对错判断完成之前，对象":Web页面"的第一个控制焦点表达的活动必须被中断。因此，"用户验证()"消息是**调用消息**。
- **异步消息示例**：在进行分数登记的时候，同时会写入日志文件，即分数登记操作不需要中断。因此，"写入日志文件()"消息是**异步消息**。

### 2.5 协作图的基本元素

协作图用于描述相互合作的对象间的交互关系和链接关系。

#### 2.5.1 协作图与顺序图的元素对比

| 元素 | 顺序图 | 协作图 |
|------|--------|--------|
| 对象 | 同类图中的对象，是类的实例 | 同类图中的对象，是类的实例 |
| 连接/生命线 | 生命线（虚线） | 对象之间的**连接关系**（实线） |
| 消息 | 对象间的一次通信 | 对象间的一次通信（附着在连接上） |
| 时间顺序 | 有明确的时间顺序 | 无时间顺序，用**消息序号**表示 |
| 对象生命周期 | 虚线末端标记 | 对象名后标以 `{new}` 或 `{destroy}` |

#### 2.5.2 协作图中的对象生命周期

- 对象名之后标以 `{new}` 约束：表示**创建对象**
- 对象名之后标以 `{destroy}` 约束：表示**销毁对象**

#### 2.5.3 协作图的路径

协作图中的连接可以带有**路径信息**，表示对象间的导航关系。

### 2.6 边界类、控制类、实体类

在系统设计阶段，需要将一般意义的类细分出三种版型：

| 类型 | 职责 | 示例 |
|------|------|------|
| **边界类**（Boundary） | 处理系统与参与者之间的交互 | JSP页面、窗口界面 |
| **控制类**（Control） | 协调对象之间的交互，封装业务逻辑 | Action类、Bean类 |
| **实体类**（Entity） | 存储和管理数据 | 数据库表对应的类 |

## 3. 方法速通

### 3.1 顺序图的建模步骤

```
步骤1：根据用例描述，找出基本事件流和可选事件流
  ↓
步骤2：根据结构模型中确定的类，确定它在用例描述中的职责
        一个职责对应该对象执行一个动作
  ↓
步骤3：事件流的先后次序决定了消息的发送次序
        一个消息导致接收消息的对象执行一个动作
  ↓
步骤4：必要的话，表达出需要创建的对象和需要撤销的对象
  ↓
步骤5：对于循环发送或分支发送的消息，使用消息顺序项进行表达
        或者使用交互架构表达
  ↓
系统分析阶段：可以只使用一般意义下的类
系统设计阶段：可以细分出边界类、控制类、实体类
```

### 3.2 协作图的建模步骤

```
方法一：同顺序图的建模步骤，但需要在连接上附着消息

方法二：从顺序图直接变换过来
  - 去掉时间轴
  - 将生命线替换为对象间的连接
  - 保留消息序号

方法三：根据类图画出对应的对象图，在链上附着消息
  - 将类图中的虚线（依赖关系）换成实线（连接关系）
  - 在连接上附着消息
```

### 3.3 条件消息与循环消息的表达方法

#### 条件消息的4种表达方式

```
① 在消息描述符中加上警戒条件
   [x<0]4:invert(x,color)

② 在消息顺序表达式中加上条件子句
   4.2[x>y]:invert(x,color)

③ 使用交互架构（Interaction Fragment）
   alt / opt 片段

④ 绘制多个顺序图（每个条件分支一个图）
```

#### 循环消息的2种表达方式

```
① 在消息顺序表达式中加上循环子句
   3.1*:update()

② 使用交互架构
   loop 片段
```

### 3.4 顺序图中的交互架构

交互架构（Interaction Fragment）包含分成若干片段的一个顺序图的某一区域，用边框包围这个区域，构成一个局部整体。

常用交互架构操作符：
- `alt`：条件分支（多选一）
- `opt`：可选执行（单条件）
- `loop`：循环执行
- `par`：并行执行
- `break`：中断

### 3.5 嵌套调用

在顺序图的控制焦点上通过绘制**重叠的小矩形条**可以表示嵌套调用。因为是嵌套调用，所以必须是**调用消息**。

三种嵌套形式：
- **(a) 自反递归**：对象调用自身的方法
- **(b) 间接递归**：A调用B，B又调用A
- **(c) 普通嵌套**：A调用B，B调用C

### 3.6 协作图中的消息顺序与嵌套

- **发送顺序**：消息1发送后，可以发送消息2。消息1和消息2的完成相互独立。
- **嵌套顺序**：消息1发送后，可以发送消息1.1。在消息1.1完成之前，消息1无法完成。

### 3.7 顺序图与协作图的对比

| 对比维度 | 顺序图 | 协作图 |
|---------|--------|--------|
| 侧重点 | 消息的**时间顺序** | 对象的**空间位置关系** |
| 生命线/控制焦点 | 有 | 无 |
| 路径 | 无 | 有 |
| 消息顺序号 | 可选 | **必须** |
| 生命线分叉 | 可以表示 | 不能表示 |
| 多对象/主动对象 | 不能直接表示 | 可以表示 |
| 适用场景 | 强调交互的时间顺序 | 强调对象间的结构关系 |

### 3.8 顺序图的两种形式

| 形式 | 说明 | 特点 |
|------|------|------|
| **一般形式** | 描述一个场景中所有可能的选择 | 可以包含条件、分支和循环 |
| **实例形式** | 详细描述一个特定的场景，说明一次可能的交互 | **没有任何条件、分支和循环** |

### 3.9 协作图的两种形式

| 形式 | 说明 | 元素 |
|------|------|------|
| **描述符形式** | 描述类元角色以及它们之间的关系 | 类元角色、关联角色 |
| **实例形式** | 描述类元实例（对象）以及它们之间的关系 | 对象、链 |

## 4. 典型示例

### 4.1 示例一：建设银行ATM机系统

**用例描述（取款）：**
1. 要求用户输入取款金额
2. 判断该储户的账户余额是否足额
3. 若足额，则输出等额的人民币现金，修改账户余额
4. 若不足额，则本次取款操作失败

**系统分析阶段的顺序图：**

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ :ATM界面  │    │ :ATM控制  │    │ :账户    │    │ :现金    │
└────┬─────┘    └────┬─────┘    └────┬─────┘    └────┬─────┘
     │               │               │               │
     │  输入取款金额   │               │               │
     │──────────────►│               │               │
     │               │  查询余额()    │               │
     │               │──────────────►│               │
     │               │  余额          │               │
     │               │◄──────────────│               │
     │               │               │               │
     │               │  [余额>=金额]   │               │
     │               │  修改余额()    │               │
     │               │──────────────►│               │
     │               │  输出现金()    │               │
     │               │──────────────────────────────►│
     │               │               │               │
     │  [余额<金额]   │               │               │
     │  提示余额不足   │               │               │
     │◄──────────────│               │               │
```

**系统设计阶段**：将一般类细分为边界类、控制类、实体类。

### 4.2 示例二：销售统计管理系统

**协作图描述：**

在一个统计销售的管理系统中，需要实现对每个销售员的订单和预算进行统计工作。

```
┌───────────────┐
│ :销售统计窗口   │
└───────┬───────┘
        │ 1:show()
        │ (从别处发来)
        │
┌───────▼───────┐     2:create()     ┌───────────────┐
│ :销售统计窗口   │──────────────────►│ :统计汇总      │
└───────────────┘                    └───────┬───────┘
                                             │
                          3.1*:getBudget()   │
                    ┌────────────────────────┤
                    │                        │
            ┌───────▼───────┐        ┌───────▼───────┐
            │ :销售员1       │        │ :销售员2       │
            └───────┬───────┘        └───────────────┘
                    │
          3.1.1:getOrderAmount()
                    │
            ┌───────▼───────┐
            │ :订单          │
            └───────────────┘
```

**交互行为说明：**
1. 从别处发来第1个消息 `show`，要求查看销售统计的情况
2. 销售统计窗口对象发出第2个消息 `create`，创建了一个统计汇总对象
3. 统计汇总对象循环地发出第3个消息 `getBudget`，分别从不同的销售员对象那里获得订单的总和与销售预算
4. 销售员对象给订单对象发消息 `getOrderAmount`，获得他自己的订单总和

## 5. 真题/实战

### 5.1 实战案例一：个人信息管理系统 - 修改个人信息

**用例描述（基本事件流）：**

1. 用户可以修改除**登录名**和**登录密码**之外的其他个人信息
2. 系统检查**日期格式**和**电子邮箱**地址格式是否正确。如果不正确，则要求用户重新输入出生日期或用户邮箱
3. 如果必填项目都有内容，且用户提交修改的个人信息，则系统修改该用户的**个人信息**。否则，系统不修改
4. 系统退回到个人信息管理主界面

**实体类之间的关系：**

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│  User    │    │  Person  │    │  Address │
│──────────│    │──────────│    │──────────│
│ userName │    │ name     │    │ street   │
```

**题目5.1**：下面关于控制焦点的概念，描述正确的是哪些？

- A. 生命线上的一个矩形条
- B. 对象的生存期
- C. 对象的方法的执行

- C错误：控制焦点不仅限于方法执行，而是对象执行一个动作的时间段

**题目5.2**：在顺序图建模中，消息指向对象，表达出以下哪些含义？

- A. 消息激活一个对象
- B. 消息创建一个对象
- C. 消息执行对象的构造方法
- D. 消息开启一个对象的生命周期

<details>
<summary>查看答案</summary>

**正确答案：A、B、C、D**

消息指向对象可以有多种含义：
- 激活一个已存在的对象

### 练习三：分支与从属流

**题目5.3**：在顺序图中，从同一点发出多条消息，下面哪些描述是对的？

- A. "分支"消息指向不同的对象
- B. "分支"消息指向同一个对象
- C. "从属流"消息指向不同的对象
- D. "从属流"消息指向同一个对象

<details>
<summary>查看答案</summary>

**正确答案：A、D**

- **分支**：从同一点发出多条消息，指向**不同的对象**（又分为条件分支和并行分支）

### 练习四：消息语法解析

**题目5.4**：在消息 `1.2.1[x>3]:p:=find(specs)` 中：
- `1.2.1` 代表什么？
- `[x>3]` 代表什么？
- `p` 代表什么？
- `find` 代表什么？
- `specs` 代表什么？

<details>
<summary>查看答案</summary>

- `1.2.1` — 消息序号，表示这是第1个消息的第2个子消息的第1个子子消息（嵌套调用）
- `[x>3]` — 警戒条件（guard condition），只有当 x>3 时才发送此消息
- `p` — 返回值，消息执行结果赋给变量 p
- `find` — 消息名（方法名）
- `specs` — 参数列表，传递给 find 方法的参数

</details>

### 练习五：调用消息与异步消息

**题目5.5**：下面哪些描述是正确的？

- A. 发出异步消息的对象，会停止前期的活动
- B. 发出调用消息的对象，会继续前期的活动
- C. 异步消息必有返回消息，且需要表达出来
- D. 调用消息必有返回消息，且无需表达出来

<details>
<summary>查看答案</summary>

**正确答案：D**

- A错误：发出**调用消息**的对象才会停止活动，异步消息的发送者继续自己的活动
- B错误：发出**调用消息**的对象会停止活动等待返回
- C错误：异步消息不一定有返回消息；如果有且是非过程调用，才需要明确表示
- D正确：调用消息是过程调用，一定有配对的返回消息，但为了简洁可以不画出来

</details>

### 7.2 常见错误与易混淆点

#### 错误1：混淆控制焦点与生命线

- **生命线**：从对象向下延伸的**虚线**，表示对象存在的整个生命期
- **控制焦点**：生命线上的**矩形条**，表示对象正在执行某个动作的时间段
- 一个对象在一个交互中可以有**多个**控制焦点

#### 错误2：混淆调用消息与异步消息的发送者行为

```
正确理解：
  调用消息 → 发送者 停止 等待（同步）
  异步消息 → 发送者 继续 活动（并发）

常见错误说法：
  ✗ "发出异步消息的对象会停止活动"
  ✗ "发出调用消息的对象会继续活动"
```

#### 错误3：混淆顺序图与协作图中对象生命周期的表达方式

```
顺序图：
  - 箭头指向的对象 → 交互过程中创建
  - 虚线末端有"X" → 对象被销毁

协作图：
  - 对象名后 {new} → 创建对象
  - 对象名后 {destroy} → 销毁对象
```

#### 错误4：忽略返回消息的绘制规则

```
调用消息：一定有返回消息（可省略不画）
异步消息：不一定有返回消息
  - 如果是过程调用 → 返回消息隐含
  - 如果是非过程调用 → 有返回消息必须画出
```

#### 错误5：混淆一般形式与实例形式

```
一般形式：可以包含条件、分支、循环（描述所有可能的选择）
实例形式：没有任何条件、分支、循环（描述一次特定的交互）
```

#### 错误6：协作图中忘记消息序号

协作图**必须**使用消息序号来描述消息的发送顺序和嵌套顺序，因为协作图没有时间轴。

#### 错误7：混淆"分支"与"从属流"

```
分支：从同一点发出多条消息，指向 不同的对象
从属流：从同一点发出多条消息，指向 同一个对象 的不同生命线
```
