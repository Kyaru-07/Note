# 第09章 数据建模

---

## 1. 概念引入：什么是数据建模，为什么它重要

### 1.1 开篇问题

在软件需求分析阶段，我们通过类图、用例图等建立了系统的**对象模型**。但对象模型只存在于内存和代码中，最终系统的数据需要**持久化存储**——这就是数据库。

核心矛盾：**面向对象的分析模型**与**关系数据库的表结构**之间存在本质差异。如何将类、属性、关系转换为表、字段、外键？这就是数据建模要解决的问题。

### 1.2 数据建模的定义

> **数据建模（Data Modeling）** 是将现实世界中的信息需求抽象为数据库结构的过程，是连接"需求分析"与"数据库实现"的桥梁。

### 1.3 为什么数据建模重要

- **数据是系统的核心资产**。数据库设计的好坏直接影响系统的性能、可维护性和扩展性。
- **对象模型与数据模型的鸿沟**。面向对象世界中的"类、继承、关联"需要映射为关系世界中的"表、外键、范式"。
- **E-R图的局限性**。传统的E-R（实体-关系）图仅仅着眼于数据，**不能对行为建模**。例如，无法对触发器（trigger）或存储过程（stored procedure）建模。
- **UML类图的优势**。UML类图可以看作E-R图的扩充——不仅着眼于数据，也能对行为建模。用类描述数据库表，用类的操作来描述触发器和存储过程。

### 1.4 学习目标

本章的核心学习目标：

1. 理解数据建模的意义
2. 理解并掌握**对象模型和数据模型的相互转换**
3. 理解**关联关系的多重性**在数据模型中的映射
4. 理解**关联、组合、泛化**等关系在数据模型中的映射

## 2. 核心概念：数据库设计的全貌

### 2.1 数据库设计的基本过程

数据库设计分为三个阶段：

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   概念设计    │ -> │   逻辑设计    │ -> │   物理设计    │
│ Conceptual   │    │   Logical    │    │  Physical    │
└──────────────┘    └──────────────┘    └──────────────┘
```

#### (1) 概念设计（Conceptual Design）

- 把用户的信息要求统一到一个**整体逻辑结构**中
- 此结构能表达用户的要求，且**独立于任何DBMS软件和硬件**
- 典型成果：**E-R图**（实体-关系图）
- 产出：信息结构

#### (2) 逻辑设计（Logical Design）

- 把概念设计阶段得到的结果转换为与选用的**DBMS所支持的数据模型相符合**的逻辑结构
- 对于关系数据库而言，逻辑设计的结果是一组**关系模式的定义**（表、记录、字段）
- 产出：逻辑数据库结构

#### (3) 物理设计（Physical Design）

- 对给定的逻辑数据模型，选取一个**最适合应用要求的物理结构**
- 要考虑：数据库的存储记录格式、存储记录安排、存取方法等
- 选择具体的DBMS：MS SQL Server、IBM DB2、Oracle、ACCESS等
- 产出：物理数据库结构

#### 完整流程图

```
处理需求 ──> 需求分析 ──> 概念设计 ──> 逻辑设计 ──> 物理设计
信息需求      需求说明书    信息结构     逻辑数据库结构   DBMS能处理的
                                                          应用程序说明书
                                                          DBMS、硬件
                                                          物理数据库结构
```

### 2.2 关键概念

数据库设计中需要掌握的核心概念：

| 概念 | 英文 | 说明 |
|------|------|------|
| **模式** | Schema | 数据库中全体数据的逻辑结构和特征的描述 |
| **主键** | Primary Key | 唯一标识表中每一行的字段或字段组合 |
| **外键** | Foreign Key | 一个表中引用另一个表主键的字段，实现表间关联 |
| **域** | Domain | 属性的取值范围和类型 |
| **关系** | Relationship | 表与表之间的关联 |
| **约束** | Constraint | 对数据的限制规则 |
| **索引** | Index | 提高检索速度的数据结构 |
| **触发器** | Trigger | 特定事件发生时自动执行的存储过程 |
| **存储过程** | Stored Procedure | 预编译的SQL语句集合 |
| **视图** | View | 虚拟表，基于查询结果定义 |
| **引用完整性** | Referential Integrity | 外键必须引用已存在的主键值 |
| **范式** | Normal Form | 数据库规范化程度（1NF、2NF、3NF等） |

### 2.3 数据库设计时用到的UML版型（Stereotype）

UML中用版型来表示数据库设计中的各种元素：

| 版型 | 对应的UML元素 | 数据库概念 |
|------|---------------|------------|
| `<<database>>` | Component | 数据库 |
| `<<Table>>` | Class | 表 |
| `<<Domain>>` | Class | 用户自定义数据类型 |
| `<<Schema>>` | Package | 模式（所有表及表间关系的集合） |
| `<<Index>>` | Operation | 索引 |
| `<<PK>>` | Operation | 主键 |
| `<<FK>>` | Operation | 外键 |
| `<<Unique>>` | Operation | 唯一性约束 |
| `<<Check>>` | Operation | 检查约束 |
| `<<Trigger>>` | Operation | 触发器 |
| `<<SP>>` | Operation | 存储过程 |
| `<<View>>` | Class | 视图 |
| `<<Identifying>>` | 关联/聚集 | 表与表之间确定性关系 |
| `<<Non-Identifying>>` | 关联 | 表与表之间非确定性关系 |

## 3. 方法速通：数据库设计的具体步骤

### 3.1 Rose中数据库设计的完整步骤

以Rose 2003为例，数据库设计包含以下8个步骤：

```
步骤1: 创建数据库对象 (<<Database>>)
    ↓
步骤2: 创建模式 (Schema)
    ↓
步骤3: 创建域包和域 (Domain Package / Domain)
    ↓
步骤4: 创建数据模型图 (Data Model Diagram)
    ↓
步骤5: 创建表和列 (Table / Column)
    ↓
步骤6: 创建关系 (Relationship)
    ↓
步骤7: 规范化 (Normalizing)
    ↓
步骤8: 优化和实现 (Optimizing / Implementing)
```

#### 步骤1：创建数据库对象

- 在组件视图（Component View）中创建一个组件，版型为 `<<Database>>`
- 缺省目标数据库为 ANSI SQL92，也可设为其他数据库（SQL Server 2000、Oracle 8.x、IBM DB2等）

#### 步骤2：创建模式（Schema）

- 在逻辑视图（Logical View）中创建 Schema，并选定目标数据库
- 对于关系数据库而言，模式可理解为**所有表及表间关系的集合**

#### 步骤3：创建域包和域

- 在逻辑视图中先创建 Domain Package
- 再创建 Domain——域可看作是**定制的数据类型**
- 可以给每个 Domain 加检查语句
- 类似于 VARCHAR2、NUMBER 等数据类型，但是用户自定义的

#### 步骤4：创建数据模型图

- 在 Schema 下创建 Data Model Diagram
- 表、视图等可以放在数据模型图中

#### 步骤5：创建表和列

- 在 Data Model Diagram 中创建 Table
- 在表上创建 Column，包括列名和列的属性

#### 步骤6：创建关系（Relationship）

表与表之间有两种关系：

| 关系类型 | 版型 | 含义 | 外键处理 |
|----------|------|------|----------|
| **非确定性关系** | `<<Non-Identifying>>` | 子表可以**离开父表单独存在** | 外键**不成为**子表主键的一部分 |
| **确定性关系** | `<<Identifying>>` | 子表**不能离开父表**而单独存在 | 外键**成为**子表主键的一部分 |

**示例说明**：
- `Order` 与 `Customer` 之间是**确定性关系**——没有 Customer，Order 就不会存在
- `Customer` 与 `Employee` 之间是**非确定性关系**——Customer 可以独立于 Employee 存在

在两种关系中，子表中都**增加外键**以支持关系。

#### 步骤7：规范化

- 创建数据模型后，要将模型规范化，如转换为 **3NF**（第三范式）
- 目的：减少记录之间的数据冗余

#### 步骤8：优化和实现

- 创建索引、视图、存储过程
- 可进行非规范化（denormalization）以提高性能
- 利用 Rose 生成 DDL，或直接在数据库中建表

### 3.2 非确定性关系的进一步分类

非确定性关系根据父表一端的多重性进一步分为：

| 类型 | 父表一端多重性 | 含义 |
|------|----------------|------|
| **强制的非确定性关系** (mandatory) | `1..n` | 父表记录必须存在 |
| **可选的非确定性关系** (optional) | `0..1` 或 `0..n` | 父表记录可以不存在 |

### 3.3 三种关系生成的SQL代码区别

#### (1) Identifying Relationship（确定性关系）

```sql
CREATE TABLE Table1 (
    column1 VARCHAR2(20) NOT NULL,
    column2 NUMBER(5),
    CONSTRAINT PK_Table10 PRIMARY KEY (column1)
);

CREATE TABLE Table2 (
    column3 VARCHAR2(10) NOT NULL,
    column4 VARCHAR2(20),
    column5 VARCHAR2(10) UNIQUE,
    column1 VARCHAR2(20) NOT NULL,
    CONSTRAINT PK_Table21 PRIMARY KEY (column1, column3)  -- 外键成为主键的一部分！
);

ALTER TABLE Table2 ADD (
    CONSTRAINT FK_Table28
    FOREIGN KEY (column1) REFERENCES Table1 (column1)
);
```

**关键**：`column1` 既是外键，又是主键的一部分（`PRIMARY KEY (column1, column3)`），且 `NOT NULL`。

#### (2) Mandatory Non-Identifying（强制非确定性关系）

```sql
CREATE TABLE Table2 (
    column3 VARCHAR2(10) NOT NULL,
    column4 VARCHAR2(20),
    column5 VARCHAR2(10) UNIQUE,
    column1 VARCHAR2(20) NOT NULL,  -- 外键不允许为空
    CONSTRAINT PK_Table21 PRIMARY KEY (column3)  -- 外键不参与主键
);

ALTER TABLE Table2 ADD (
    CONSTRAINT FK_Table28
    FOREIGN KEY (column1) REFERENCES Table1 (column1)
);
```

**关键**：`column1` 是外键，`NOT NULL`，但**不参与主键**。

#### (3) Optional Non-Identifying（可选非确定性关系）

```sql
CREATE TABLE Table2 (
    column3 VARCHAR2(10) NOT NULL,
    column4 VARCHAR2(20),
    column5 VARCHAR2(10) UNIQUE,
    column1 VARCHAR2(20),  -- 外键允许为空！
    CONSTRAINT PK_Table21 PRIMARY KEY (column3)
);

ALTER TABLE Table2 ADD (
    CONSTRAINT FK_Table28
    FOREIGN KEY (column1) REFERENCES Table1 (column1)
);
```

**关键**：`column1` 是外键，**允许为 NULL**，不参与主键。

### 3.4 类到表的转换规则

从对象模型到数据模型的基本变换规则：

1. **类转换为表**，类与类之间的关系转换为表与表之间的关系，或者也转换为表
2. **类的属性**转换为表的**字段**，类的**操作**转换为**触发器和存储过程**
3. 一个类可以转换为**一个以上的表**（当类间有一对多的关系时）
4. 一个表也可以对应**多个类**（减少数据冗余）
5. 关系（一对一、一对多、多对多及三项关系）的转换可能有多种情况，但一般转换为一个表，也可以在表间定义相应的外键
6. **持久性（Persistent）**标记：类必须设置成持久性，否则转换时不会生成对应的表

### 3.5 对象模型与数据模型的相互转换

#### 对象模型 -> 数据模型（正向工程）

具体步骤：
1. 在组件视图下创建数据库对象
2. 在逻辑视图下创建包（如 Demo 包），在包中创建类（如 Flight、FlightAttendant）
3. 类必须设置成**持久性（Persistent）**
4. 建立类之间的关系（如普通单向关联），设置多重性
5. 右键单击包 -> Data Modeler -> Transform to Data Model
6. 设置模式名字、目标数据库、表名前缀、是否对外键生成索引
7. 生成模式（如 S_1），其中包含表（如 T_Flight、T_FlightAttendant、T_0）

#### 数据模型 -> 对象模型（逆向工程）

具体步骤：
1. 对模式（即包）进行操作——Rose会把**一个模式中的所有表及其关系**转换为对象模型
2. 右键单击模式 -> Data Modeler -> Transform to Object Model
3. 设置包名字、类名前缀、是否根据主键生成类的对应属性
4. 生成包（如 OM_S_1），其中包含类（如 OM_T_Flight、OM_T_FlightAttendant）

## 4. 典型示例：泛化关系的映射

### 4.1 泛化关系的映射方法

泛化关系（继承）在数据模型中有**三种映射策略**：

#### 方法一：父类和子类各映射为一个表

每个表有公共的主码（ID-based Key），子表通过外键引用父表。

**示例**：Employee（父类）-> Pilot（子类）、FlightAttendant（子类）

```
┌─────────────────────────┐
│    Employee Table       │
│─────────────────────────│
│ employeeID (PK)         │
│ name                   │
│ taxpayerNumber          │
│ employeeType            │
└─────────────────────────┘
         ↑
    ┌────┴────┐
    ↓         ↓
┌──────────┐ ┌──────────────────┐
│Pilot Table│ │FlightAttendant   │
│──────────│ │Table             │
│pilotID(PK)│ │────────────────│
│flightRating│ │flightAttendantID│
└──────────┘ │(PK)             │
              └──────────────────┘
```

**优点**：结构清晰，无数据冗余
**缺点**：查询时需要多表连接（JOIN），性能较低

#### 方法二：不映射父类，父类属性加到子类中

```
┌──────────────────┐  ┌──────────────────────┐
│  Pilot Table     │  │ FlightAttendant Table │
│──────────────────│  │──────────────────────│
│ pilotID (PK)     │  │ flightAttendantID(PK)│
│ name             │  │ name                 │
│ taxpayerNumber   │  │ taxpayerNumber       │
│ flightRating     │  │                      │
└──────────────────┘  └──────────────────────┘
```

**优点**：查询简单，无需JOIN
**缺点**：父类属性在每个子表中**重复存储**，数据冗余

#### 方法三：不映射子类，所有子类属性加入父类

```
┌────────────────────────────┐
│  Employee Table            │
│────────────────────────────│
│ employeeID (PK)            │
│ name                       │
│ taxpayerNumber             │
│ pilotID                    │
│ flightRating               │
│ flightAttendantID          │
│ employeeType               │
└────────────────────────────┘
```

**优点**：单表查询，最简单
**缺点**：大量NULL值（如果一个人是Pilot，则flightAttendantID为NULL），浪费存储空间

### 4.2 三种方法的对比

| 方法 | 冗余 | NULL值 | 查询复杂度 | 适用场景 |
|------|------|--------|------------|----------|
| 父子各一个表 | 无 | 无 | 高（需JOIN） | 子类属性差异大、查询需要区分类型 |
| 父类属性加到子类 | 有 | 无 | 低 | 子类数量少、多数查询只涉及单一子类 |
| 子类属性加到父类 | 无 | 有 | 最低 | 子类属性差异小、多数查询涉及所有类型 |

### 4.3 多重继承的处理

- 对多重继承的超类和子类**分别转换为表**
- 对多次多重继承的泛化关系也转换为一个表
- 对转换后的表进行**冗余控制调整**，使其达到合理的关系范式

## 5. 真题/实战：多重性在数据模型中的映射

### 5.1 关联映射的分类

从对象模型到数据模型，关联的映射分为：

- **1:0..*** 关联的映射（一对多）
- **1:0..1** 关联的映射（一对一）
- **0..*:0..*** 关联的映射（多对多）

### 5.2 一对多（1:0..*）关联的映射

**场景**：一个部门有多个员工（Department 1 --- 0..* Employee）

**映射方法**：在"多"端（Employee）表中增加外键，引用"一"端（Department）表的主键。

```
┌─────────────────┐       ┌─────────────────┐
│ Department Table │       │  Employee Table  │
│─────────────────│       │─────────────────│
│ deptID (PK)     │  ←────│ deptID (FK)     │
│ deptName        │       │ empID (PK)      │
│ ...             │       │ empName         │
└─────────────────┘       │ ...             │
                          └─────────────────┘
```

**SQL示例**：

```sql
CREATE TABLE Department (
    deptID VARCHAR2(10) NOT NULL,
    deptName VARCHAR2(50),
    CONSTRAINT PK_Department PRIMARY KEY (deptID)
);

CREATE TABLE Employee (
    empID VARCHAR2(10) NOT NULL,
    empName VARCHAR2(50),
    deptID VARCHAR2(10) NOT NULL,
    CONSTRAINT PK_Employee PRIMARY KEY (empID),
    CONSTRAINT FK_Employee_Dept
        FOREIGN KEY (deptID) REFERENCES Department (deptID)
);
```

### 5.3 一对一（1:0..1）关联的映射

**场景**：一个经理管理一个部门（Manager 1 --- 0..1 Department）

**映射方法**（任选其一）：
1. 在任一端增加外键引用另一端
2. 合并为一个表

### 5.4 多对多（0..*:0..*）关联的映射

**场景**：学生选课（Student 0..* --- 0..* Course）

**映射方法**：创建**关联表（中间表）**，包含两端的外键，联合做主键。

```
┌──────────────┐    ┌──────────────────┐    ┌──────────────┐
│ Student Table │    │  SC (选课表)      │    │ Course Table  │
│──────────────│    │──────────────────│    │──────────────│
│ stuID (PK)   │←───│ stuID (FK/PK)   │───→│ courseID(PK) │
│ stuName      │    │ courseID (FK/PK) │    │ courseName   │
└──────────────┘    │ score            │    │ credit       │
                    └──────────────────┘    └──────────────┘
```

**SQL示例**：

```sql
CREATE TABLE Student (
    stuID VARCHAR2(10) NOT NULL,
    stuName VARCHAR2(50),
    CONSTRAINT PK_Student PRIMARY KEY (stuID)
);

CREATE TABLE Course (
    courseID VARCHAR2(10) NOT NULL,
    courseName VARCHAR2(100),
    credit NUMBER(2),
    CONSTRAINT PK_Course PRIMARY KEY (courseID)
);

CREATE TABLE SC (
    stuID VARCHAR2(10) NOT NULL,
    courseID VARCHAR2(10) NOT NULL,
    score NUMBER(5,2),
    CONSTRAINT PK_SC PRIMARY KEY (stuID, courseID),
    CONSTRAINT FK_SC_Student
        FOREIGN KEY (stuID) REFERENCES Student (stuID),
    CONSTRAINT FK_SC_Course
        FOREIGN KEY (courseID) REFERENCES Course (courseID)
);
```

### 5.5 主键的定义方法

定义主键可以采用两种方法：

1. **ID-based Key（代理键）**：使用无业务含义的自增ID作为主键
2. **Candidate Key（候选键/自然键）**：使用有业务含义的字段组合作为主键

## 6. 巩固练习

### 练习1：概念辨析

**问题**：请说明确定性关系（Identifying）和非确定性关系（Non-Identifying）的区别，并各举一个例子。

**参考答案**：

| 特征 | 确定性关系 | 非确定性关系 |
|------|------------|--------------|
| 子表能否独立存在 | 不能离开父表 | 可以离开父表 |
| 外键是否参与主键 | 是（成为子表主键的一部分） | 否 |
| 版型 | `<<Identifying>>` | `<<Non-Identifying>>` |
| 示例 | Order 与 Customer | Customer 与 Employee |

### 练习2：关系映射

**问题**：有一个网上书店系统，包含以下实体和关系：
- **Book**（书）：bookID, title, price, author
- **Author**（作者）：authorID, name, bio
- **Publisher**（出版社）：publisherID, name, address

关系：
- 一本书有一个出版社，一个出版社出版多本书
- 一本书可以有多个作者，一个作者可以写多本书

请设计对应的数据库表结构（含主键、外键）。

**参考答案**：

```sql
-- 出版社表
CREATE TABLE Publisher (
    publisherID VARCHAR2(10) NOT NULL,
    name VARCHAR2(100),
    address VARCHAR2(200),
    CONSTRAINT PK_Publisher PRIMARY KEY (publisherID)
);

-- 作者表
CREATE TABLE Author (
    authorID VARCHAR2(10) NOT NULL,
    name VARCHAR2(50),
    bio CLOB,
    CONSTRAINT PK_Author PRIMARY KEY (authorID)
);

-- 书表（一对多：出版社 -> 书）
CREATE TABLE Book (
    bookID VARCHAR2(10) NOT NULL,
    title VARCHAR2(200),
    price NUMBER(8,2),
    publisherID VARCHAR2(10) NOT NULL,
    CONSTRAINT PK_Book PRIMARY KEY (bookID),
    CONSTRAINT FK_Book_Publisher
        FOREIGN KEY (publisherID) REFERENCES Publisher (publisherID)
);

-- 书-作者关联表（多对多）
CREATE TABLE BookAuthor (
    bookID VARCHAR2(10) NOT NULL,
    authorID VARCHAR2(10) NOT NULL,
    CONSTRAINT PK_BookAuthor PRIMARY KEY (bookID, authorID),
    CONSTRAINT FK_BA_Book
        FOREIGN KEY (bookID) REFERENCES Book (bookID),
    CONSTRAINT FK_BA_Author
        FOREIGN KEY (authorID) REFERENCES Author (authorID)
```

### 练习3：泛化映射策略选择

**问题**：在一个车辆管理系统中，有如下类层次：

```
Vehicle (vehicleID, licensePlate, color)
  ├── Car (seats, trunkCapacity)
  ├── Truck (loadCapacity, axleCount)
  └── Motorcycle (hasSidecar)
```

请分别用三种泛化映射方法设计表结构，并分析各方法的优缺点。

**参考答案**：

**方法一：父子各一个表**

```sql
CREATE TABLE Vehicle (
    vehicleID VARCHAR2(10) NOT NULL,
    licensePlate VARCHAR2(20),
    color VARCHAR2(20),
    CONSTRAINT PK_Vehicle PRIMARY KEY (vehicleID)
);

CREATE TABLE Car (
    vehicleID VARCHAR2(10) NOT NULL,
    seats NUMBER(2),
    trunkCapacity NUMBER(5),
    CONSTRAINT PK_Car PRIMARY KEY (vehicleID),
    CONSTRAINT FK_Car_Vehicle
        FOREIGN KEY (vehicleID) REFERENCES Vehicle (vehicleID)
);

-- Truck、Motorcycle 类似...
```

**方法二：不映射父类**

```sql
CREATE TABLE Car (
    vehicleID VARCHAR2(10) NOT NULL,
    licensePlate VARCHAR2(20),
    color VARCHAR2(20),
    seats NUMBER(2),
    trunkCapacity NUMBER(5),
    CONSTRAINT PK_Car PRIMARY KEY (vehicleID)
);

-- Truck、Motorcycle 类似，每个表都重复存储 Vehicle 的属性
```

**方法三：不映射子类**

```sql
CREATE TABLE Vehicle (
    vehicleID VARCHAR2(10) NOT NULL,

    color VARCHAR2(20),
    vehicleType VARCHAR2(20),
    seats NUMBER(2),
    trunkCapacity NUMBER(5),
    loadCapacity NUMBER(8),
    axleCount NUMBER(2),
    hasSidecar CHAR(1),
    CONSTRAINT PK_Vehicle PRIMARY KEY (vehicleID)
);
-- 所有字段在一张表中，非该类型车辆的字段为 NULL
```

### 练习4：判断题

判断以下说法的正误：

1. **"一个类最多只能转换为一个表。"**
   - **错误**。一个类可以转换为一个以上的表，当类间有一对多关系时。

2. **"在确定性关系中，外键成为子表主键的一部分。"**
   - **正确**。

3. **"类必须设置成持久性（Persistent），否则转换时不会生成对应的表。"**
   - **正确**。

4. **"E-R图可以对触发器和存储过程建模。"**
   - **错误**。E-R图仅仅着眼于数据，不能对行为建模。UML类图才能通过操作来描述触发器和存储过程。

5. **"多对多关系必须通过创建关联表（中间表）来实现。"**
   - **正确**。

### 练习5：SQL代码分析

**问题**：阅读以下SQL代码，判断两个表之间的关系类型。

```sql
CREATE TABLE Table1 (
    column1 VARCHAR2(20) NOT NULL,
    column2 NUMBER(5),
    CONSTRAINT PK_Table1 PRIMARY KEY (column1)
);

CREATE TABLE Table2 (
    column3 VARCHAR2(10) NOT NULL,
    column4 VARCHAR2(20),
    column1 VARCHAR2(20) NOT NULL,
    CONSTRAINT PK_Table2 PRIMARY KEY (column1, column3)
);

ALTER TABLE Table2 ADD (
    CONSTRAINT FK_Table2
    FOREIGN KEY (column1) REFERENCES Table1 (column1)
);
```

**参考答案**：这是**确定性关系（Identifying Relationship）**。

判断依据：
1. `column1` 在 Table2 中**既是外键，又是主键的一部分**（`PRIMARY KEY (column1, column3)`）
2. `column1` 在 Table2 中为 **NOT NULL**

## 7. 总结复盘

### 7.1 本章核心知识点

```
数据建模
├── 数据库设计三阶段
│   ├── 概念设计（E-R图，独立于DBMS）
│   ├── 逻辑设计（关系模式：表、记录、字段）
│   └── 物理设计（存储结构、索引、具体DBMS选择）
├── UML版型体系
│   ├── <<Database>>、<<Schema>>、<<Table>>
│   ├── <<PK>>、<<FK>>、<<Index>>、<<View>>
│   └── <<Identifying>>、<<Non-Identifying>>
├── 关系映射
│   ├── 一对多："多"端加外键
│   ├── 一对一：任一端加外键或合并
│   └── 多对多：创建关联表
├── 泛化映射（三种策略）
│   ├── 父子各一个表（无冗余，需JOIN）
│   ├── 父类属性加到子类（有冗余，查询简单）
│   └── 子类属性加到父类（有NULL，最简单）
└── 对象模型 <-> 数据模型双向转换
    ├── 正向：类 -> 表，属性 -> 字段，操作 -> 触发器/存储过程
    └── 逆向：表 -> 类，外键 -> 关联
```

### 7.2 常见易错点

1. **混淆确定性和非确定性关系**：记住——确定性关系中，外键**参与主键**；非确定性关系中，外键**不参与主键**。

2. **忘记设置持久性**：Rose中类必须标记为 `Persistent`，否则不会生成对应的表。

3. **多对多关系处理不当**：多对多关系**必须**创建中间表，不能直接在两个表中互相加外键。

4. **泛化映射策略选择**：没有"最好"的方法，要根据具体场景（查询频率、数据量、冗余容忍度）选择。

5. **E-R图与UML类图的区分**：E-R图不能建模行为（触发器、存储过程），UML类图可以。

6. **数据模型与对象模型的转换是双向的**：正向工程（对象 -> 数据）和逆向工程（数据 -> 对象）都是支持的。

7. **规范化与反规范化的权衡**：规范化减少冗余但增加JOIN，反规范化提高查询性能但增加冗余。实际项目中常需要在两者之间取舍。

### 7.3 考试重点

- 确定性关系 vs 非确定性关系的**SQL代码差异**（外键是否NOT NULL、是否参与主键）
- 多对多关系的**中间表设计**
- 泛化关系的**三种映射方法**及其优缺点
- 对象模型与数据模型的**对应关系**（类->表，属性->字段，操作->触发器/存储过程）
- 数据库设计的**三个阶段**及各阶段的产出物
