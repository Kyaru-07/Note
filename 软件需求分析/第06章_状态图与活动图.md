# 第06章 状态图与活动图

---

## 1. 概念引入：行为模型——描述系统"怎么动"

### 1.1 从静态模型到动态模型

前面几章我们学习了用例图、类图、对象图等**静态模型**，它们描述了系统"有什么"。但系统是活的——对象会在不同状态之间切换，业务流程会按步骤流转。这就需要**行为模型**来描述系统"怎么动"。

行为模型包括三大类：

| 模型类型 | 对应图 | 描述对象 | 涉及范围 |
|---------|--------|---------|---------|
| **状态模型** | 状态图 | 一个对象的生命期内的状态及状态变迁 | 单个对象 |
| **活动模型** | 活动图 | 一个活动到另一个活动的控制流、工作流程 | 单个或多个对象 |
| **交互模型** | 顺序图、协作图 | 对象之间通过消息传递协作完成功能 | 多个对象 |

### 1.2 为什么需要状态图和活动图？

**开篇案例：借书证的生命周期**

考虑一个图书馆系统中的"借书证"对象：

- 当办理好借书证并开通后，借书证处于**可借书状态**
- 当读者需要借书，且"已借书本数"未达上限时，仍处于**可借书状态**
- 当读者需要借书，且"已借书本数"已达上限时，借书证变为**不可借书状态**
- 无论哪个状态，只要读者还书，就回到**可借书状态**
- 无论哪个状态，只要挂失或有效期过期，就进入**注销状态**

这个案例说明：对象在生命周期中会经历不同的**状态**，状态之间的转换由**事件**触发。用自然语言描述容易遗漏和模糊，而**状态图**可以清晰、完整地表达这些行为。

同理，当我们描述"借书"这个业务流程时（录入条码 → 判断能否借书 → 生成借阅记录 → 修改可借本数），需要一种图来表达活动之间的先后顺序和分支，这就是**活动图**。

### 1.3 状态图与活动图的核心区别

```
状态图：关注一个对象"处于什么状态"  → 状态驱动
活动图：关注一个流程"执行什么活动"  → 流程驱动
```

- **状态图**用于复杂状态的对象（状态多、状态间转换复杂）
- **活动图**用于复杂行为的对象（活动多、流程复杂）

> **关键判断**：状态图应该用于具有**复杂状态**的对象，而不必用于具有**复杂行为**的对象。对于有复杂行为的对象，使用活动图会更加合适。

## 2. 核心概念：状态图的要素

### 2.1 状态（State）

**定义**：状态是指在对象的生命期中的某个条件或状况。

- 状态具有一定的时间**稳定性**，在此期间对象将满足某些条件、执行某些活动或等待某些事件
- 可以认为"状态"对应某些**变量或属性取了特定值**

**状态的组成**：

```
┌─────────────────────┐
│      状态名          │  ← 状态的标识符
├─────────────────────┤
│    状态变量          │  ← 进行计数的内部变量
│    入口动作/         │  ← 转入/转出状态时必须执行的动作
│    出口动作          │
│    内部转移          │  ← 不导致状态发生变化的动作
└─────────────────────┘
```

### 2.2 转移（Transition）

**定义**：转移是两个状态之间的一种关系，表示在指定事件发生后，在特定警戒条件满足时，对象执行指定的动作，并从起始状态进入目标状态。

**转移的语法格式**：

```
event-signature [guard-condition] / action
```

| 组成部分 | 含义 | 示例 |
|---------|------|------|
| event-signature | 事件特征标记 | `put(c)` |
| guard-condition | 警戒条件 | `[c!='>']` |
| action | 要执行的动作 | `token.append(c)` |

**完整示例**：`put(c) [c!='>'] / token.append(c)`

**转移由五个部分组成**：起始状态、目标状态、触发事件、警戒条件、转移动作。

**转移的分叉规则**：对于一个给定的状态，最终只能产生一个转移，因此从相同的状态出来的、事件相同的几个转移之间的条件应该是**互斥的**。

### 2.3 事件（Event）

**定义**：事件是对一个可观察的事情的类型描述。

事件产生的原因包括：调用、满足条件的状态的出现、到达时间点、经历某一时间段、发送信号等。

#### 2.3.1 四种事件类型

| 事件类型 | 说明 | 示例 |
|---------|------|------|
| **调用事件** | 对操作的调用 | 订单状态图中调用`cancel()`方法 |
| **变化事件** | 布尔表达式变量发生变化 | 锅炉温度超过阈值 |
| **时间事件** | 满足某一时间表达式 | 信号灯每隔30秒变换 |
| **信号事件** | 由一个对象异步发送、另一对象接收的已命名实体 | 遥控器发出的信号 |

#### 2.3.2 调用事件 vs 信号事件（重要考点）

| 对比维度 | 调用事件 | 信号事件 |
|---------|---------|---------|
| **本质** | 对应对象的一个方法的执行 | 对应一个变量有新赋值 |
| **触发者** | 可以由状态图描述的对象发出，也可以由其它对象发出 | 只能由其它对象发出 |
| **效果** | 直接执行方法 | 通常导致接收到信号的对象的一个方法的执行 |
| **涉及对象** | 两个对象 | 三个对象（发送对象、接收对象、信号对象） |

#### 2.3.3 信号的特征

- 信号是一个已命名的**类**
- 发送一个信号，等同于**创建一个信号类的实例**，并将这个信号的实例传送给接收对象
- 信号也可以有自己的属性，所有信号共享隐式的操作
- 信号之间可以建立**泛化（继承）关系**

**信号事件的优势**：如果将各种类型的信号作为一个抽象类的若干子类，则要增加新的信号时，只需要新增信号类的子类，接收对象不用修改。这体现了**开闭原则**。

**遥控器示例**：
- "遥控器"对象发出`Off`事件，夹带传递了"遥控器信号"的"关机"对象
- "电视机"对象接收到`Off`事件后，解读出"关机"对象，调用自己的`Off`方法，断电关机，进入"Off"状态

### 2.4 组合状态与子状态

**组合状态**：一个状态内部存在一个子状态的变化过程。

**子状态**：包含在某一个状态内部的状态。

**示例——汽车行驶的状态**：

```
汽车
├── 静止状态
└── 运动状态（组合状态）
    ├── 加速（子状态）
    ├── 均速（子状态）
    └── 减速（子状态）
```

### 2.5 历史状态

**定义**：记录了组合状态被转出时的活跃子状态是哪个。

| 符号 | 名称 | 含义 |
|------|------|------|
| H | 浅历史状态 | 只记住最外层组合状态的历史 |
| H* | 深历史状态 | 可记住任何深度的组合状态的历史 |

**注意事项**：如果一个组合状态到达了其**终态**，则会丢失历史状态中的信息，就好像还没有进入过这个组合状态一样。

## 3. 核心概念：活动图的要素

### 3.1 活动（Activity）

**定义**：活动表示的是某流程中的任务的执行，它可以表示某算法过程中语句的执行。

- 一个活动是一个对象的某个方法中的部分（或全部）语句的执行
- 可以对应成顺序图中的控制焦点（激活期）的矩形条

**动作状态 vs 活动状态**：

| 类型 | 特点 | 说明 |
|------|------|------|
| **动作状态** | 原子的，不可分解 | 最小的执行单元 |
| **活动状态** | 非原子的，可分解 | 可以进一步细化 |

### 3.2 分支（Branch）

对于同一个触发事件，可以根据不同的**警戒条件**转向不同的活动，每个可能的转移是一个分支。

```
         ┌─ [条件A] → 活动X
触发事件 →├─ [条件B] → 活动Y
         └─ [条件C] → 活动Z
```

### 3.3 分叉（Fork）与汇合（Join）

| 元素 | 含义 |
|------|------|
| **分叉** | 一个控制流被两个或多个控制流代替，这些控制流**并发**进行 |
| **汇合** | 两个或多个控制流被一个控制流代替（同步等待） |

```
        ┌→ 活动A →┐
分叉 →  │          │ → 汇合
        └→ 活动B →┘
```

### 3.4 泳道（Swimlane）

**定义**：泳道是活动图中的区域划分，根据每个活动的**职责**对所有活动进行划分，每个泳道代表一个**责任区**。

**关键理解**：
- 泳道和类**并不是一一对应**的关系
- 泳道关心的是其所代表的**职责**，一个泳道可能由一个类实现，也可能由多个类实现
- 一个活动，在顺序图中对应几个矩形条，就可以将这几个矩形条的对象集作为一个泳道

**泳道的使用场景**：

| 场景 | 是否带泳道 | 用途 |
|------|-----------|------|
| 需求分析阶段 | 不带泳道 | 刻画一个用例的用例描述 |
| 系统分析及设计阶段 | 带泳道 | 刻画用例描述中的步骤如何落实到对象的头上 |

### 3.5 对象流

**定义**：对象可以作为活动的输入或输出，对象流是一种特殊的控制流，表示活动和对象之间的关系。

- 一个活动**创建**对象（作为活动的输出）
- 一个活动**使用**对象（作为活动的输入）
- 活动可以**修改**对象状态

> 活动图中画出对象流，就是强调这个对象在这个活动中的重要性。

### 3.6 信号（活动图中的）

同状态图中的"信号"概念，信号是表示两个对象之间**异步通信**的方式。

| 元素 | 含义 |
|------|------|
| **发送信号** | 活动过程中发送事件，触发另一活动流程 |
| **接受信号** | 活动过程中接收事件，接收到信号的活动流程开始执行 |

## 4. 方法速通：状态图与活动图的建模步骤

### 4.1 状态图的建模步骤

**第一步：识别需要绘制状态图的对象**
- 状态图应该用于具有**复杂状态**的对象
- 不必用于具有复杂行为的对象（用活动图更合适）

**第二步：识别状态空间**
- 根据对象**生命周期的不同阶段**来划分状态
- 根据对象的**属性值的不同**来划分状态
- 识别每个候选状态是否有动作

**第三步：识别状态间的转移**
- 状态之间通过转移连接在一起
- 寻找一些明显的事件（调用事件、变化事件、时间事件、信号事件）
- 注意信号的特征：一个信号类的实例（实体），该信号类具有泛化（继承）关系图

**第四步：检查与完善**
- 确保所有状态都被覆盖
- 确保转移条件互斥
- 检查是否有遗漏的事件

### 4.2 活动图的建模步骤

**第一步：明确建模范围**
- 是单个用例？
- 还是一个用例的一部分？
- 还是一个包含多个用例的业务流程？
- 还是一个类的单个方法？

**第二步：识别活动**
- 将相关性强的主要步骤放入到一个活动中
- 无论是对用例建模，还是对方法的算法建模

**第三步：识别分支**
- 如果一个活动有多个变迁，则使用分支
- 必须对每个变迁加以标识（警戒条件）

**第四步：识别并行**
- 当两个活动间没有直接的联系，而且它们都必须在第三个活动开始前结束，那它们是可以**并行运行**的
- 使用分叉和汇合表示并发

**第五步：分配泳道**（可选）
- 根据活动的职责划分泳道
- 将活动分配到对应的泳道中

## 5. 典型示例：从课件中提取的经典案例

### 5.1 状态图案例：借书证

**场景描述**：
- 办理并开通借书证 → 可借书状态
- 借书且未达上限 → 可借书状态
- 借书且已达上限 → 不可借书状态
- 还书 → 可借书状态
- 挂失或有效期过期 → 注销状态

**状态图结构**：

```
[初始] → 可借书状态
可借书状态 → 不可借书状态：借书 [已借书本数=上限]
可借书状态 → 可借书状态：借书 [已借书本数<上限]
不可借书状态 → 可借书状态：还书
可借书状态 → 注销状态：挂失 / 有效期过期
不可借书状态 → 注销状态：挂失 / 有效期过期
注销状态 → [终态]
```

**对比分析**：
- **借书证（实体类）**：需要借助于一个属性值（已借书本数）的判断
- **借书证（控制类）**：不需要借助于一个属性值的判断
- 状态图中的事件都是**调用事件**，可以对应成"方法"或"操作"，或对应成顺序图中的"调用消息"

### 5.2 状态图案例：图书

**场景描述**：
1. 新书购买回来后，没有编码之前，不能入库。编码之后，方可入库。
2. 入库后的图书，才能被借出。
3. 被借出的图书，不能被再借出。
4. 还书后，图书才能再借出。
5. 图书过时了或破损了，必须下架，被清除的图书不能被借出。

**状态图结构**：

```
[初始] → 未编码
未编码 → 已入库：编码
已入库 → 已借出：借出
已借出 → 已入库：还书
已入库 → 已下架：下架
已借出 → 已下架：下架
```

### 5.3 状态图案例：手机

**场景描述**：
1. 没有使用手机打电话时，手机处于**闲置状态**
2. 点击电话图标时，手机处于**就绪状态**
3. 输入对方电话号码，点击通话图标后，手机处于**拨号状态**
4. 对方接听，则手机处于**通话状态**
5. 若在5秒钟之内自己主动挂断，则返回**就绪状态**
6. 若在15秒钟之内对方没有接听操作，则一直处于**拨号状态**
7. 若超过15秒钟，或者对方在15秒之内挂断，则处于**就绪状态**
8. 通话结束后，手机处于**就绪状态**
9. 关闭打电话应用后，手机处于**闲置状态**

**状态图结构**：

```
[初始] → 闲置状态
闲置状态 → 就绪状态：点击电话图标
就绪状态 → 闲置状态：关闭应用
就绪状态 → 拨号状态：输入号码并点击通话
拨号状态 → 就绪状态：5秒内主动挂断 / 超过15秒 / 对方15秒内挂断
拨号状态 → 通话状态：对方接听
通话状态 → 就绪状态：通话结束
```

### 5.4 状态图案例：汽车（组合状态）

**场景描述**：
1. 没有发动前，汽车处于静止状态
2. 发动后，汽车处于运动状态
3. 汽车在行驶过程中，可以加速、均速、减速

**状态图结构**：

```
[初始] → 静止状态
静止状态 → 运动状态：发动
运动状态（组合状态）：
  [初始] → 均速
  均速 → 加速：加速
  加速 → 均速：松油门
  均速 → 减速：减速
  减速 → 均速：松刹车
运动状态 → 静止状态：熄火
```

**分析**：
- 总共有**4个状态**：静止、运动（组合状态）、加速、均速、减速
- **运动状态**是组合状态
- 加速、均速、减速是**子状态**
- 子状态之间是**并发**关系（可以相互转换）

### 5.5 活动图案例：借书（带泳道）

**场景描述**：
1. 录入图书流通条码和借书证号
2. 判断该借书证能否借书
3. 若不能借书，则结束本用例
4. 若能借书，则生成借阅记录，并修改借书证可借本数

**带泳道的活动图**：

```
┌──────────┬──────────┬──────────┐
│  借阅者   │  系统     │  数据库   │
├──────────┼──────────┼──────────┤
│ 录入条码  │          │          │
│ 和借书证号│          │          │
├──────────┼──────────┼──────────┤
│          │ 判断能否  │          │
│          │ 借书      │          │
├──────────┼──────────┼──────────┤
│          │ [不能借]  │          │
│          │ → 结束    │          │
│          │ [能借]    │          │
│          │ 生成借阅  │          │
│          │ 记录      │          │
├──────────┼──────────┼──────────┤
│          │          │ 保存借阅  │
│          │          │ 记录      │
├──────────┼──────────┼──────────┤
│          │ 修改可借  │          │
│          │ 本数      │          │
├──────────┼──────────┼──────────┤
│          │          │ 更新借书证│
│          │          │ 信息      │
└──────────┴──────────┴──────────┘
```

### 5.6 活动图案例：ATM存款

**场景描述（建设银行ATM机系统）**：
1. 用户放入需要存款的现金
2. 系统验证真伪并统计金额
3. 系统提示存款金额
4. 若用户确认金额及存款，则系统修改账户余额

**需求分析阶段的活动图**（不带泳道）：

```
[初始] → 放入现金
放入现金 → 验证真伪并统计金额
验证真伪并统计金额 → 提示存款金额
提示存款金额 → [用户确认] 修改账户余额
提示存款金额 → [用户取消] 取消存款
修改账户余额 → [终态]
取消存款 → [终态]
```

**系统设计阶段的活动图**（带泳道）：

```
┌──────────┬──────────┬──────────┐
│   用户    │   ATM    │  银行系统  │
├──────────┼──────────┼──────────┤
│ 放入现金  │          │          │
├──────────┼──────────┼──────────┤
│          │ 验证真伪  │          │
│          │ 统计金额  │          │
├──────────┼──────────┼──────────┤
│          │ 提示存款  │          │
│          │ 金额      │          │
├──────────┼──────────┼──────────┤
│ 确认存款  │          │          │
├──────────┼──────────┼──────────┤
│          │ 发送存款  │ 修改账户  │
│          │ 请求      │ 余额      │
├──────────┼──────────┼──────────┤
│          │ 显示存款  │          │
│          │ 成功      │          │
└──────────┴──────────┴──────────┘
```

### 5.7 状态图案例：防盗报警系统（综合案例）

**系统组成**：控制盒、安全传感器、火警传感器、报警盒

**状态图分析**：

1. 无条件转移到状态`SystemInactive`（系统未激活）
2. `activate`事件发生时，从`SystemInactive`转移到组合状态`SystemActive`
3. 进入`SystemActive`，无条件转移到组合状态中的初态
4. 从初态无条件转移到状态`InitializingSystem`
5. 进入`InitializingSystem`，有一个分叉，两个子状态`InitializingFireSensors`和`InitializingSecuritySensors`开始**并发执行**
6. 当这两个子状态都结束时，才能从`InitializingSystem`中退出（汇合同步）
7. 退出后，无条件转移到状态`Monitoring`
8. 进入`Monitoring`，有一个分叉，两个子状态`MonitoringFireSensors`和`MonitoringSecuritySensors`开始并发执行
9. 当事件`fire`发生时，转移到`SoundingFireAlarm`，发出火警报警鸣声
10. 当事件`intruder`发生时，转移到`SoundingIntruderAlarm`，发出非法侵入报警鸣声，持续5分钟
11. 如果在报警非法侵入的同时又发生火警，则立即转移到`SoundingFireAlarm`（**火警优先级高于非法侵入**）
12. 当事件`sensorError`发生时，转移到`Error`状态，然后无条件转移到终态
13. 当事件`deactivate`发生时，从`SystemActive`转移到`SystemInactive`

## 6. 真题/实战：状态与活动的深层理解

### 6.1 状态与方法、用例的联系

**核心关系**：

1. 当一个**方法**被执行时，使得对象的属性值发生变化，或者对象在生命周期中处于不同的阶段，即对象进入到不同的**状态**
2. 对象处于不同的**状态**，会暗示后续要执行的不同的**方法**有哪些
3. 一个用例的执行对应一个**顺序图**，顺序图中的消息可以作为对象状态改变的事件集
4. 通常，需要多个**顺序图**，来融合地描述一个对象的**完整状态图**

```
用例 → 顺序图 → 消息 → 事件 → 状态转移 → 状态图
```

### 6.2 活动与方法、用例的联系

**核心关系**：

1. 当一个**活动**被执行时，对应于对象的一个**方法**中的若干个（或者全部的）语句的执行
2. 可能需要多个（也可以是一个）**活动**来完成一个**方法**
3. 一个**活动**只能属于一个**方法**
4. 一个**活动**是一个用例的几个流程步骤的集成
5. 通常，一个用例描述对应于一个**活动图**，因此一个**用例**对应于若干个**活动**

```
用例 → 活动图 → 活动 → 方法中的语句
```

### 6.3 状态 vs 活动（重要考点）

| 对比维度 | 状态 | 活动 |
|---------|------|------|
| **本质** | 一个对象所处的境况 | 一段程序代码的执行 |
| **产生** | 通常是执行了一个（或多个）活动后的**结局** | 对应于若干个步骤的集成 |
| **关系** | 不同的状态会导致不同的功能（对应于若干个活动）的执行 | 活动的执行可能导致状态的变化 |

### 8.2 常见错误与陷阱

**错误1：混淆状态图和活动图的使用场景**
- ❌ 错误：用状态图描述复杂的业务流程
- ✅ 正确：状态图用于复杂**状态**的对象，活动图用于复杂**行为**的流程

**错误2：混淆调用事件和信号事件**
- ❌ 错误：认为调用事件和信号事件是一样的
- ✅ 正确：调用事件对应方法执行，信号事件对应变量赋值；调用事件可以由自身对象发出，信号事件只能由其他对象发出

**错误3：泳道与类一一对应**
- ❌ 错误：一个泳道对应一个类
- ✅ 正确：泳道关心的是**职责**，一个泳道可能由一个类实现，也可能由多个类实现

**错误4：忘记转移条件互斥**
- ❌ 错误：从同一状态出来的、事件相同的转移，条件有重叠
- ✅ 正确：这些条件必须**互斥**，因为最终只能产生一个转移

**错误5：混淆动作状态和活动状态**
- ❌ 错误：认为所有状态都是可分解的
- ✅ 正确：动作状态是**原子的、不可分解**的；活动状态是**非原子的、可分解**的

**错误6：历史状态信息丢失**
- ❌ 错误：认为历史状态永远保存
- ✅ 正确：如果组合状态到达了**终态**，则会丢失历史状态中的信息

**错误7：忽略需求分析与设计阶段的区别**
- ❌ 错误：在需求分析阶段就画带泳道的活动图
- ✅ 正确：需求分析阶段画**不带泳道**的活动图（描述用例），设计阶段画**带泳道**的活动图（落实到对象）

### 8.3 考试要点速记

```
状态图三要素：状态 + 转移 + 事件
转移五部分：起始状态 + 目标状态 + 触发事件 + 警戒条件 + 转移动作
事件四类型：调用事件 + 变化事件 + 时间事件 + 信号事件
活动图关系：转移 + 分支 + 分叉 + 汇合
泳道定义：按职责划分的责任区
对象流：活动和对象之间的关系
```

### 8.4 状态图 vs 活动图对比总结

| 对比维度 | 状态图 | 活动图 |
|---------|--------|--------|
| **关注点** | 对象的状态变化 | 活动的执行流程 |
| **适用对象** | 复杂状态的对象 | 复杂行为的对象 |
| **基本元素** | 状态、转移、事件 | 活动、分支、分叉、汇合 |
| **泳道** | 不使用 | 可选使用 |
| **并发** | 组合状态中的并发 | 分叉/汇合表示并发 |
| **典型应用** | 订单生命周期、设备状态 | 业务流程、算法描述 |

### 8.5 快速判断题

**判断以下说法是否正确**：

1. 状态图可以描述多个对象的状态变化。（❌，只能描述一个对象）
2. 活动图中的泳道和类是一一对应的。（❌，泳道关心的是职责）
3. 信号事件只能由其他对象发出。（✅）
4. 调用事件只能由自身对象发出。（❌，可以由自身或其他对象发出）
5. 动作状态是原子的、不可分解的。（✅）
6. 历史状态在组合状态到达终态后仍然保存。（❌，会丢失）
7. 分叉表示多个控制流并发执行。（✅）
8. 汇合表示等待多个控制流同步结束。（✅）
9. 需求分析阶段应使用带泳道的活动图。（❌，应使用不带泳道的）
10. 转移的警戒条件可以重叠。（❌，必须互斥）

---

> **本章核心一句话**：状态图描述对象"是什么状态"，活动图描述流程"做什么活动"；状态是活动的结局，活动是状态的驱动力。
