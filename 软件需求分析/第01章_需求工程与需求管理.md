# 第01章 需求工程与需求管理

---

## 1. 概念引入：什么是需求工程，为什么它重要

### 1.1 开篇案例：工资管理系统的需求困境

某公司因业务发展，员工从300人增长到5000人，原有的人工工资管理系统已不适应新形势。公司总经理李云约见技术开发部主管王奇，提出开发新系统的需求。

然而，这个看似简单的沟通却暴露了诸多问题：

- **需求获取并非想象中容易**。不同层次的用户（公司老总、财务主管、会计审核、出纳等）对新系统有不同的期望。
- **高层管理者并非实际使用者**。提出业务需求的高管可能并不直接使用系统。
- **用户难以完整描述需求**。只有实际用户才能描述系统必须达到的目标，但他们又无法指出所有具体的功能需求。
- **用户开始时无法准确表达需求**。软件开发人员需要做大量深入细致的调查研究，反复与用户交流，才能真正理解需求。

**核心启示**：用户说"我们不就是用户吗？哪有那么多时间讨论细节？"——这恰恰是需求工程要解决的根本矛盾。

### 1.2 需求工程的定义

> **需求工程**是指对系统应该提供的服务和所受到的约束进行理解、分析、建立文档、检验以及跟踪管理的全过程。

它把所有与需求直接相关的活动统称为需求工程，包括两大过程域：
- **需求开发**：通过调查与分析，获取用户需求并定义产品需求。
- **需求管理**：维护需求约定的完整性、准确性，控制需求变更。

### 1.3 为什么需求工程如此重要

**Frederick Brooks**在1987年经典文章《No Silver Bullet》中指出：

> "开发软件系统最困难的部分就是准确说明开发什么。最困难的概念性工作是编写出详细的需求，包括所有面向用户、面向机器和其它软件系统的接口。此工作一旦做错，将会给系统带来极大的损害，并且以后对它修改也极为困难。"

**河流比喻**：需求是产品的根源，需求工作的优劣对产品影响最大。就像一条河流，如果源头被污染了，那么整条河流也就被污染了。

**Garbage in, garbage out**——输入的是垃圾，输出的也只能是垃圾。

## 2. 核心公式/概念：需求的定义、层次与分类

### 2.1 需求的定义

#### IEEE标准定义

1. 用户为解决某个问题或达到某个目标而需具备的**条件或能力**。
2. 系统或系统组件为符合合同、标准、规范或其他正式文档而必须满足的**条件或必须具备的能力**。
3. 上述第一项或第二项中定义的条件和能力的**文档表达**。

> 宽泛地讲，需求来源于用户的一些"需要"，这些"需要"被分析、确认后形成完整的文档，该文档详细地说明了产品**要做什么**。它描述了系统的行为、特性或属性，是在开发过程中对系统的约束。

#### 专家定义

| 专家 | 定义要点 |
|------|----------|
| **Davis** | 从软件外部可见的、软件所具有的、满足于用户的特点、功能及属性等的集合 |
| **Sommerville** | 问题信息和系统行为、特性、设计和实现约束的描述的集合 |

**软件需求**是指用户对目标软件系统在功能、性能、设计约束等方面的期望。

### 2.2 需求的层次模型

需求按抽象程度从高到低分为三个层次：

```
┌─────────────────────────────────────┐
│           业务需求                    │  ← 最高层
│  (Business Requirement)              │
├─────────────────────────────────────┤
│           用户需求                    │  ← 中间层
│  (User Requirement)                  │
├─────────────────────────────────────┤
│           系统需求                    │  ← 最底层
│  (System Requirement)                │
│  ┌───────────┬───────────┐          │
│  │ 功能需求   │ 非功能需求  │          │
│  └───────────┴───────────┘          │
└─────────────────────────────────────┘
```

#### (1) 业务需求（Business Requirement）

- 反映组织机构或客户对系统、产品的**概括性目标要求**
- 在**项目视图与范围文档**中予以说明
- 是系统建立的**战略出发点**，表现为高层次的目标（Objective）
- 描述了组织**为什么要开发系统**
- 参与各方必须对高层次的解决方案达成一致，建立共同的前景（Vision）

#### (2) 用户需求（User Requirement）

- 描述用户使用系统而完成的**任务集合**
- 在**用户案例（Use Case）文档**或方案脚本中予以说明
- 执行实际工作的用户对系统所能完成的具体任务的期望
- 描述了系统能够帮助用户**做些什么**

#### (3) 系统需求（System Requirement）

- **功能需求**（Functional Requirement）：定义开发人员必须实现的软件功能，源于用户需求
- **非功能需求**（Non-functional Requirement）：定义系统属性或约束，如可靠性、响应时间等
- 系统需求可以直接映射为系统行为，描述开发人员需要**实现什么**

#### 需求层次实例：酒店预订系统

| 层次 | 示例内容 |
|------|----------|
| **业务需求** | 网站上提供酒店促销价格的预订，以吸引对价格敏感的用户 |
| **用户需求** | 总结出几种可以在网站上销售的促销方式；系统能够录入促销价格；网站可以显示并预订 |
| **系统需求** | 促销价格录入、促销价格查询、订单处理、异常情况处理（提前离店/延住）、界面显示 |

#### 课堂测验

> 在需求层次模型中，按照"从高到低"的次序，正确的次序是？
> - A. 用户需求，业务需求，系统需求
> - B. **业务需求，用户需求，系统需求** ✓
> - C. 用户需求，系统需求，业务需求
> - D. 业务需求，系统需求，用户需求

### 2.3 需求的分类

#### (1) 功能需求（Functional Requirements）

- 对系统应该提供的服务、功能以及系统在特定条件下行为的描述
- 与软件系统的类型、使用系统的用户等相关
- 用户的功能需求可以高层次说明，但**软件系统的功能需求必须详细、可操作**

**图书馆系统功能需求示例**：
- **基本数据维护**：提供使用者录入、修改并维护基本数据（读者信息）的途径
- **基本业务功能**：新书编目入库、读者借还书登记、书籍预留、书籍遗失登记
- **数据管理与查询**：图书信息及作者信息的统一管理维护，各类信息查询

#### (2) 非功能需求（Non-functional Requirements）

- 定义系统的**属性或约束**
- 可能比功能需求更难达到
- 应定义可客观评价的指标，以便通过测试验证
- 无法满足可能导致系统**无法使用**

**图书馆系统非功能需求示例**：
- **安全性需求**：对各项功能进行分级、分权限操作
- **效率需求**：各类查询操作在X秒内完成，每次借还书登记在Y秒内完成
- **可靠性需求**：还书登记出错率为1/10000，系统故障率为1/500h

**非功能需求获取的关键问题**：
- **易用性**：用户技能水平？熟悉什么界面标准？
- **可靠性**：可用性指标？可接受系统重启吗？数据丢失影响？
- **性能**：反应速度？时间要求？并发用户数？
- **可支持性**：预计扩展？谁来维护？移植计划？

#### (3) 领域需求（Domain Requirements）

- 由软件系统的**应用领域**决定
- 描述系统具有领域特征的功能或特性
- 包括新的功能需求、已有功能的约束、特殊的计算
- 无法满足可能导致系统**无法正常工作**

**图书馆系统领域需求示例**：
- 图书编目要求按照**中国图书馆分类法**进行
- 由于版权限制，某些文献资料只能在图书馆规定的阅览室阅读，不能出借

### 2.4 干系人（Stakeholders）

> 干系人是指直接或间接从产品中获得利益的个人或组织。

- **第一类**：提出要求、支付款项、具体描述业务、使用软件产品的**项目风险承担者**
- **第二类**：获得软件产品所产生的**结果**的人

## 3. 方法速通：需求获取、分析与验证技术

### 3.1 需求工程的过程全景

```
需求工程
├── 需求开发
│   ├── 需求获取 → 产生《用户需求说明书》
│   ├── 需求分析 → 消除错误，刻画细节
│   └── 需求定义 → 产生《软件需求规格说明书》(SRS)
└── 需求管理
    ├── 需求确认 → 双方达成共识，建立需求基线
    ├── 需求跟踪 → 建立需求跟踪矩阵
    └── 需求变更控制 → 变更申请-审批-更改-重新确认
```

### 3.2 需求获取技术

需求获取是需求工程的**主体**，非常困难，主要原因：

| 困难点 | 说明 |
|--------|------|
| 缺乏领域知识 | 开发人员不了解业务领域 |
| 存在默认知识 | 用户认为"理所当然"的不会主动说 |
| 存在多个知识源 | 不同用户可能有矛盾的需求 |
| 双方误解需求 | 技术术语与业务术语的鸿沟 |
| 客户可能的偏见 | 用户可能有先入为主的观念 |

#### 需求获取方法

1. **建立联合分析小组**：由领域专家、分析员、用户代表组成
2. **用户访谈**：
   - 结构化面谈：按预设问题逐一提问
   - 非结构化面谈：自由交谈，灵活深入
3. **问卷调查表**：适用于大规模用户群
4. **观察用户工作流程**：实地了解用户的实际操作
5. **原型法**：构建系统原型让用户试用反馈

### 3.3 需求分析方法

需求分析包括将高层需求分解成具体细节、建立需求模型、评估可行性和协商需求优先级。

#### (1) 结构化分析（Structured Analysis, SA）

**基本思想**："分解"和"细化"

- **分解**：将大问题分解成若干小问题，分别解决
- **细化**：先考虑最本质的属性，暂把细节略去，逐层添加细节

**结构化需求模型**：

| 模型类型 | 工具 | 说明 |
|----------|------|------|
| 功能模型 | 数据流图（DFD） | 描述数据如何在系统中流动 |
| | 加工规格说明（PSPEC） | 描述每个加工的详细逻辑 |
| 数据模型 | 数据字典（DD） | 定义数据项的含义 |
| | 实体关系图（ERD） | 描述数据实体之间的关系 |
| 行为模型 | 状态转换图（STD） | 描述系统状态的变化 |
| | 控制规格说明（CSPEC） | 描述控制逻辑 |

#### (2) 面向对象分析（Object Oriented Analysis, OOA）

- 基于**用例**的面向对象需求分析
- **Ivar Jacobson**提出了Use Case的概念及可视化表示方法
- Use Case被广泛应用到面向对象的系统分析中
- 基于用例的需求方法已成为面向对象分析方法的**主流**

**面向对象需求模型**：
- 用例模型（用例图、用例规约、补充规约、术语表）

### 3.4 需求文档

#### 用户需求说明书 vs 软件需求规格说明书

| 对比项 | 用户需求说明书 | 软件需求规格说明书（SRS） |
|--------|---------------|-------------------------|
| 表达方式 | 自然语言（和应用域术语） | 计算机语言和图形符号 |
| 详细程度 | 比较粗略 | 详细精确 |
| 主要读者 | 用户、管理人员 | 开发人员、测试人员 |
| 作用 | 沟通需求 | 系统设计的直接依据 |

#### 软件需求规格说明书（SRS）

> 精确地阐述了一个软件系统必须提供的功能和性能以及它所要考虑的限制条件。不仅是系统测试和用户文档的基础，也是所有项目规划、设计和编码的基础。

**描述方法**：
- 自然语言
- 图形化模型
- 形式化规格说明

#### 编写需求规格的原则

1. 保持语句和段落的**简短**
2. 编写具有正确语法、拼写和标点的**完整句子**
3. 使用术语与词汇表所定义的应该**一致**
4. 避免模糊的、主观的术语，减少**不确定性**
5. 避免使用**比较性词汇**

### 3.5 需求验证

需求验证是指开发方和客户方共同对《软件需求规格说明书》进行评审，双方对需求达成共识后作出承诺以建立**需求基线**。

#### 验证内容

| 检查类型 | 说明 |
|----------|------|
| 有效性检查 | 功能需求是否符合用户所提出的需求 |
| 一致性检查 | 系统功能描述和约束是否一致 |
| 完备性检查 | 是否包含所有用户的需求和约束 |
| 可检验性检查 | 能否设计出验证方法，确定检查标准 |

#### 需求评审的重要性

在不同阶段发现缺陷的修复成本：

| 发现阶段 | 相对修复成本 |
|----------|-------------|
| 需求阶段 | **3-6** |
| 功能测试 | 15-40 |
| 验收测试 | 30-70 |
| 发布之后 | **40-1000** |

**结论**：越早发现需求问题，修复成本越低！

#### 评审方法

- **分层次评审**：目标性评审 → 功能性评审 → 操作性评审
- **分阶段评审**：在需求开发的不同阶段进行多次评审

#### 评审进入标准

1. 文档符合模板规范
2. 已做过拼写检查和语法检查
3. 已检查文档版面安排
4. 审查员已获得所需文档
5. 文档中打印了行号
6. 所有未解决问题标记为TBD
7. 包含术语词汇表

#### 评审退出标准

1. 已明确阐述审查员提出的所有问题
2. 已正确修改文档
3. 修订过的文档已做拼写和语法检查
4. TBD问题已全部解决或已记录解决过程
5. 文档已登记入配置管理系统

## 4. 典型示例：从案例中理解需求工程

### 4.1 案例一：工资管理系统的需求演化

**背景**：某公司员工从300人增长到5000人，需要开发新的工资管理系统。

**需求层次演进**：

| 阶段 | 内容 | 问题 |
|------|------|------|
| 总经理李云的初始描述 | "允许员工以无纸化方式登记时间卡信息，自动生成工资单" | 仅是**高层次业务需求** |
| 王奇的反馈 | "这些是高层次的业务需求，我们还需要足够的详细信息" | 需要功能、性能、用户要求等详细信息 |
| 李云的质疑 | "你不能让你的手下人自己确定各种需求吗？" | 忽视了用户参与的重要性 |

**启示**：需求获取是一个迭代的过程，需要开发人员与用户反复沟通。

### 4.2 案例二：大学图书馆管理系统

**系统描述**：一个大学图书馆系统，能够为学生和教工提供查询、借阅图书以及文献资料的服务。

#### 功能需求

```
基本数据维护功能
├── 录入、修改、维护基本数据
└── 读者信息的修改和更新

基本业务功能
├── 新书编目、入库
├── 读者借、还书籍的登记功能
├── 书籍预留功能
└── 书籍遗失登记功能

数据管理与查询功能
├── 图书信息及作者信息的统一管理维护
└── 各类信息查询（书籍、借书、还书、预留等）
```

#### 非功能需求

- **安全性**：对各项功能进行分级、分权限操作
- **效率**：查询操作在规定时间内完成
- **可靠性**：还书登记出错率1/10000，系统故障率1/500h

#### 领域需求

- 图书编目按照**中国图书馆分类法**进行
- 受版权限制，某些文献只能在阅览室阅读，不能出借

### 4.3 案例三：酒店促销预订系统

**业务需求**：网站上提供酒店促销价格的预订，以吸引对价格敏感的用户。

**用户需求分解**：
- 总结出几种可以在网站上销售的促销方式（和酒店订协议）
- 系统能够录入这些促销价格
- 网站可以显示并预订

**系统需求分解**：
- 促销价格录入
- 促销价格查询
- 订单处理
- 异常情况处理（提前离店、延住）
- 界面显示

## 5. 真题/实战：优秀需求的特性与需求管理

### 5.1 优秀需求的七大特性

```
优秀需求特性
├── 完整性
├── 正确性
├── 无歧义性
├── 可行性
├── 有优先级
├── 必要性
└── 可验证性
```

#### (1) 完整性

每一项需求必须将所要实现的功能描述清楚，使开发人员获得设计和实现这些功能所需的**所有必要信息**。

必须从业务角度组织需求项，让用户验证主题域、业务事件、业务活动、业务步骤、困难与障碍点等是否完整。

#### (2) 正确性

每一项需求必须准确地陈述其要开发的功能。判断依据是用户或高层的系统需求规格说明。

#### (3) 无歧义性

对所有需求说明的读者都只能有一个**明确统一的解释**。

避免歧义的有效方法：
- 对需求文档的正规审查
- 编写测试用例
- 开发原型
- 设计特定的方案脚本

#### (4) 可行性

每一项需求必须在已知系统和环境的权能和限制范围内**可以实施**。

**反例**："在一年内的所有日子里，每天工作24小时，系统的有效性必须达到100%"——从当前技术来讲是不可行的。

#### (5) 有优先级

为了更好地管理项目，需要区分需求的优先级别。可以从三个角度进行划分：
- **业务角度**（最重要）：根据业务的价值和频度进行评价
- 技术开发角度
- 项目管理角度

#### (6) 必要性

每一项需求都应该把客户真正所需要的和最终系统所需遵从的标准记录下来。

#### (7) 可验证性

检查每项需求是否能通过设计测试用例或其他验证方法，来确定产品是否确实按需求实现了。

### 5.2 需求管理的核心活动

#### (1) 版本控制

**软件版本**包含两种含义：
1. 为满足不同用户的不同使用要求，适用于不同运行环境或不同平台的**系列产品**
2. 软件产品投入使用后，经过运行提出变更要求，需要做较大的修正或纠错，增强功能或提高性能

**版本控制**是对配置项的不同版本进行标志和跟踪的过程。其目的在于：
- 提供有效的追踪手段
- 保证需要时可回到旧版本
- 避免文件丢失、修改丢失和相互覆盖
- 通过访问控制避免未经授权的访问和修改
- 有效保护企业软件资产和知识产权
- 实现团队并行开发，提高开发效率

#### (2) 需求跟踪

**需求的标识与属性**：

| 属性 | 说明 |
|------|------|
| 创建需求的时间 | 记录需求产生的时间点 |
| 需求的版本号 | 跟踪需求的变更历史 |
| 创建需求的作者 | 明确需求的来源 |
| 负责认可该需求的人员 | 明确审批责任人 |
| 需求状态 | 已建议/已批准/已实现/已删除 |
| 需求的原因或根据 | 信息的出处 |
| 需求涉及的子系统 | 明确影响范围 |
| 需求涉及的产品版本号 | 关联产品版本 |

**需求状态流转**：

```
已建议 → 已批准 → 已实现
                  ↘ 已删除
```

- **已建议**：该需求已被有权提出需求的人建议
- **已批准**：该需求已被分析，估计了对项目余下部分的影响，已确定版本号，开发团队已同意实现
- **已实现**：使用所选择的方法已验证了实现的需求，该需求被认为完成
- **已删除**：计划的需求已被删除，并包含原因说明和作出删除决定的人员

**双向跟踪**：

- **正向跟踪**：以用户需求为切入点，检查《用户需求说明书》或《需求规格说明书》中的每个需求是否都能在后继工作产品中找到对应点
- **逆向跟踪**：检查设计文档、代码、测试用例等工作产品是否都能在《需求规格说明书》中找到出处

#### (3) 需求变更控制

需求的变更是不可避免的，有效控制需求变化对项目成功至关重要。

**变更控制流程**：

```
变更申请 → 变更评估 → 变更审批 → 变更实施 → 重新确认
```

**变更控制策略**：
1. **项目启动阶段**：变更预防
2. **项目实施阶段**：需求变更管理
3. **项目收尾阶段**：总结经验教训

## 6. 巩固练习

### 练习一：需求层次识别

**题目**：某电商公司希望"通过提供个性化推荐功能，提高用户购买转化率20%"。请判断这属于哪个层次的需求，并说明理由。

**参考答案**：这属于**业务需求**层次。原因：它描述的是组织的战略目标（提高转化率），是一个高层次的目标，而非具体的用户任务或系统功能。

### 练习二：需求分类判断

**题目**：对于一个在线考试系统，请分别举出一个功能需求、非功能需求和领域需求的例子。

**参考答案**：
- **功能需求**：系统能够自动批改客观题并生成成绩报告
- **非功能需求**：系统应支持1000人同时在线考试，页面响应时间不超过3秒
- **领域需求**：考试成绩的评定标准需符合教育部相关规定

### 练习三：优秀需求特性分析

**题目**：请分析以下需求描述的问题："系统应该很快响应用户的请求。"

**参考答案**：该需求违反了以下优秀需求特性：
1. **无歧义性**："很快"是模糊术语，不同人理解不同
2. **可验证性**：无法设计测试用例来验证"很快"是否达标
3. **完整性**：未说明具体的响应时间指标

**改进后**："系统在正常负载下，对用户查询请求的响应时间应不超过2秒。"

### 练习四：需求变更场景分析

**题目**：在项目开发中期，客户提出要增加一个"社交分享"功能。作为需求工程师，你会如何处理？

**参考答案**：
1. **记录变更请求**：正式记录客户的需求变更申请
2. **影响评估**：评估该功能对项目进度、成本、技术架构的影响
3. **优先级协商**：与客户协商该功能的优先级，是否可以延后实现
4. **变更审批**：提交变更控制委员会（CCB）审批
5. **更新文档**：如批准变更，更新需求规格说明书和相关设计文档
6. **通知相关方**：通知开发团队、测试团队等相关人员

### 练习五：双向跟踪练习

**题目**：请说明正向跟踪和逆向跟踪的区别，并各举一个应用场景。

**参考答案**：
- **正向跟踪**：从需求出发，检查后续工作产品是否实现了该需求。例如：检查SRS中的"用户登录"功能是否在设计文档中有对应的设计方案。
- **逆向跟踪**：从工作产品出发，检查其是否有需求依据。例如：检查代码中的"密码加密"功能是否在SRS中有对应的需求描述。

## 7. 总结复盘

### 7.1 本章核心知识点

```
需求工程与需求管理
├── 需求的定义
│   ├── IEEE标准定义
│   ├── Davis定义
│   └── Sommerville定义
├── 需求的层次
│   ├── 业务需求（最高层）
│   ├── 用户需求（中间层）
│   └── 系统需求（最低层）
│       ├── 功能需求
│       ├── 非功能需求
│       └── 领域需求
├── 需求工程过程
│   ├── 需求获取（访谈、问卷、原型等）
│   ├── 需求分析（结构化分析、面向对象分析）
│   ├── 需求定义（SRS编写）
│   └── 需求验证（评审、测试用例）
├── 优秀需求特性
│   └── 完整、正确、无歧义、可行、有优先级、必要、可验证
└── 需求管理
    ├── 版本控制
    ├── 需求跟踪（正向/逆向）
    └── 需求变更控制
```

### 7.2 常见错误与陷阱

| 错误类型 | 具体表现 | 正确做法 |
|----------|----------|----------|
| **需求来源单一** | 只听取高管意见，忽视实际用户 | 多层次用户参与，建立联合分析小组 |
| **需求描述模糊** | "系统应该好用"、"响应要快" | 量化指标，具体可测量 |
| **忽视非功能需求** | 只关注功能，不考虑性能、安全 | 系统性地收集非功能需求 |
| **需求文档不更新** | 需求变更后文档未同步更新 | 建立版本控制和变更管理机制 |
| **缺少需求评审** | 跳过评审直接进入开发 | 分层次、分阶段进行需求评审 |
| **过度承诺** | 接受不合理的时间和功能要求 | 基于现实评估，协商优先级 |
| **忽视领域需求** | 不了解行业规范和法律法规 | 学习领域知识，咨询领域专家 |

### 7.3 考试重点提醒

1. **需求层次模型**：业务需求、用户需求、系统需求的关系与区别（高频考点）
2. **需求分类**：功能需求、非功能需求、领域需求的识别与举例
3. **优秀需求特性**：七大特性的理解与应用
4. **需求工程过程**：需求获取、分析、定义、验证的流程
5. **需求管理**：版本控制、需求跟踪（正向/逆向）、变更控制流程
6. **需求评审**：评审的进入标准、退出标准、评审方法
7. **用户需求说明书与SRS的区别**：表达方式、详细程度、主要读者

### 7.4 一句话总结

> 需求工程是软件开发的**源头工程**。做好需求工程，就是为项目成功奠定基础。记住：**需求是产品的根源，源头被污染，整条河流都会被污染。**
