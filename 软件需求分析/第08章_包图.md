# 第08章 包图

## 1. 概念引入

### 1.1 什么是包图

在面向对象软件开发中，一个大型系统可能包含成百上千个类。如何组织和管理这些类，是控制软件复杂性的重要手段。

- 在**结构化方法**中，考虑的是如何对功能进行分解
- 在**面向对象方法**中，考虑的是如何把**相关的类放在一起**，而不再是对系统的功能进行分解

**包图（Package Diagram）** 就是用来描述包与包之间关系的UML图。包（Package）是UML中用于将模型元素组织成组的通用机制，它相当于一个"容器"，可以用来存储和管理类、接口、用例等模型元素。

### 1.2 为什么需要包图

软件开发时常见的问题是：**如何把一个大系统分解为多个较小系统？** 包图提供了一种有效的组织方式：

- **控制复杂性**：将大系统分解为多个可管理的小单元
- **支持团队开发**：不同的包可以分配给不同的开发团队
- **促进软件重用**：设计良好的包可以作为可重用的单元
- **便于配置管理**：以包为单位进行版本管理和发布

### 1.3 包与目录的关系

包可以对应成文件系统中的**文件夹（目录）**：

- 文件夹中存储源代码文档
- 当系统运行时，源代码文档会被装入内存运行
- 此时，**组件被实例化**，但**不能认为包被实例化**
- 体系结构建模中的"包图"解释了组件如何被安排到文件夹中

## 2. 核心概念

### 2.1 包中的元素

包在开发大型软件系统时是一个非常重要的机制，它把关系密切的模型元素组织在一起。

**关键特征**：
- 包中的模型元素不仅仅限于类，可以是**任何UML模型元素**
- 一个模型元素**只能被一个包所拥有**（唯一归属原则）
- 包就像一个"容器"，对模型元素进行分组管理

### 2.2 包的命名

包的命名有两种方式：

| 命名方式 | 说明 | 示例 |
|----------|------|------|
| **简单包名** | 只包含包本身的名称 | `Vision` |
| **路径包名** | 包含外围包的名称，用 `::` 分隔 | `Sensors::Vision` |

在路径包名 `Sensors::Vision` 中，`Sensors` 是 `Vision` 包的外围包，即 `Vision` 包嵌套在 `Sensors` 包中。

**命名约束**：在一个包中，同种元素必须有不同的名字。例如，一个包中两个JSP文档必须具有不同的文件名（组件名）。

### 2.3 包中元素的可见性

通过可见性，控制包中元素被访问的权限。包中元素的可见性分为三种：

| 可见性 | 符号 | 语义 | 说明 |
|--------|------|------|------|
| **公有访问** | `+` (public) | 任何导入该包的包都可以引用 | 对外完全开放 |
| **保护访问** | `#` (protected) | 只有该包的子包可以引用 | 仅对家族内部开放 |
| **私有访问** | `-` (private) | 只有该包内部的元素可以引用 | 仅包内部可用 |

**示例**：在AWT包中：
- `Window` 的可见性为 **public**：任何导入AWT包的包都可以引用Window
- `Form` 的可见性为 **protected**：只有AWT包的子包才可以引用Form
- `EventHandler` 的可见性为 **private**：只有AWT包中的元素才可以引用EventHandler

可见性的语义同类的可见性语义一样，本质上都是控制访问权限。

### 2.4 包的层次性

包可以拥有其他包（称为**子包**）作为自己的内部元素，子包又可以拥有自己的子包。因此，包可以**嵌套**，形成一个层次结构。

```
Sensors（顶层包）
├── Vision（子包）
│   ├── Camera
│   └── ImageProcessor
├── Audio（子包）
│   ├── Microphone
│   └── SoundAnalyzer
└── Motion（子包）
```

层次结构的好处：
- 反映系统的组织结构
- 支持自顶向下的设计方法
- 便于理解和管理大型系统

### 2.5 包之间的关系

因为包中包含了模型元素（例如类），而模型元素之间存在着关系，因此折射出包之间也存在相同的关系。包之间的关系主要有两种：

#### 2.5.1 依赖关系

对于由对象类组成的包，如果两个包中的对象类之间存在着依赖关系，那么这两个包之间就有依赖关系。

**重要特性**：包之间的依赖关系**没有传递性**。

```
包A --《import》--> 包B --《import》--> 包C
（这不意味着包A自动依赖包C）
```

常见的依赖关系版型：
- `《import》`：一个包导入另一个包中的公共元素
- `《access》`：一个包访问另一个包中的元素

#### 2.5.2 泛化关系

包之间也可以有泛化关系：
- **子包继承父包**中可见性为 `public` 和 `protected` 的元素
- 包之间的泛化关系描述了系统的**接口**

```
父包（General Package）
    ↑ （泛化/继承）
子包（Special Package）
```

## 3. 方法速通

### 3.1 包的设计原则

在考虑如何对类进行分组并放入不同的包时：
- 主要是根据类之间的**依赖关系和泛化关系**进行分组
- 类之间的**关联关系**看作是依赖关系（因为关联关系可以归类为特殊的依赖关系）
- 在建立包时，应把**概念上和语义上相近**的模型元素放入一个包中

**核心目标**：好的设计要求体现**高内聚、低耦合**的特性。

### 3.2 四大设计原则

#### 原则一：重用等价原则（REP, Reuse Equivalence Principle）

> 把类放入包中时，应考虑把包作为**可重用的单元**。

- 可重用的类存在升级版本，而且会比较快速地更新版本
- 把这些类放在一个包中，方便对这个包的各个版本的管理
- **核心思想**：包是重用的基本单位

#### 原则二：共同闭包原则（CCP, Common Closure Principle）

> 把那些**需要同时改变的类**放在一个包中。

满足以下任一条件的类应放在同一包中：
- 一个类的改变要求另一个类做相应的改变
- 删除了一个类后，另一个类变成多余的
- 两个类之间有大量的消息发送

**核心思想**：一起变化的类放在一起，减少修改时需要打开的包数量。

#### 原则三：共同重用原则（CRP, Common Reuse Principle）

> **不会一起使用的类不要放在同一包中。**

- 一个包中包含的多个类之间如果关系不密切，改变其中的一个类不会引起别的类的改变
- 那么把这些类放在同一个包中会对用户的使用造成不便

**核心思想**：只打包一起使用的类，避免不必要地强制依赖。

#### 原则四：非循环依赖原则（ADP, Acyclic Dependencies Principle）

> 包之间的依赖关系**不要形成循环**。

**循环依赖的问题**：
```
包A → 包B → 包C → 包A  （形成循环！）
```
- 使得系统中的每个包都无法独立构成一个可重用的单元
- 严重妨碍软件的重用性和可扩展性
- 导致系统耦合度的增强

**打破循环依赖的方法**：
- 将导致循环的类提取到一个新的包中
- 使用接口或抽象类解耦
- 重新审视包的划分是否合理

### 3.3 REP、CCP、CRP 三者的关系

**重要提示**：重用等价原则（REP）、共同闭包原则（CCP）、共同重用原则（CRP）这三个原则事实上是**相互排斥的**！

| 原则 | 关注点 | 包的大小倾向 |
|------|--------|-------------|
| REP | 可重用性 | 包较大（包含更多可重用的类） |
| CCP | 可维护性 | 包较大（包含更多一起变化的类） |
| CRP | 避免不必要依赖 | 包较小（只包含紧密相关的类） |

设计时需要在三者之间找到平衡点。

### 3.4 高内聚和低耦合的实现方法

以下三种方法可以做到高内聚和低耦合：

1. **最小化包之间的依赖关系** —— 减少包之间的连线
2. **最小化每个包的公共元素和受保护元素的数目** —— 减少对外暴露的接口
3. **最大化每个包的私有元素的数目** —— 将实现细节隐藏在包内部

### 3.5 包图设计步骤

```
步骤1：识别类之间的关系（依赖、泛化、关联）
    ↓
步骤2：根据设计原则（CCP、CRP、REP）对类进行分组
    ↓
步骤3：确定包之间的依赖关系
    ↓
步骤4：检查是否存在循环依赖
    ↓
步骤5：优化包结构（高内聚、低耦合）
    ↓
步骤6：确定元素的可见性（public/protected/private）
```

## 4. 典型示例

### 4.1 AWT包的可见性示例

```
┌─────────────────────────────┐
│           AWT               │
│                             │
│  + Window (public)          │  ← 任何导入AWT的包都可引用
│  # Form (protected)         │  ← 只有AWT的子包可引用
│  - EventHandler (private)   │  ← 只有AWT内部可引用
│                             │
└─────────────────────────────┘
```

**含义解读**：
- 外部包导入AWT后，只能看到 `Window`
- AWT的子包可以看到 `Window` 和 `Form`
- 只有AWT包自身才能访问 `EventHandler`

### 4.2 包的层次结构示例

以"个人信息管理系统"为例：

```
个人信息管理系统（顶层包）
├── 界面层（UI Package）
│   ├── 主界面
│   ├── 联系人管理界面
│   └── 日程管理界面
├── 业务逻辑层（Business Package）
│   ├── 联系人管理
│   ├── 日程管理
│   └── 备忘录管理
└── 数据访问层（Data Package）
    ├── 数据库访问
    └── 文件访问
```

### 4.3 包的依赖关系示例

以"个人信息管理系统"为例：

```
┌─────────┐    import    ┌─────────┐    import    ┌─────────┐
│  界面层   │ ──────────→ │ 业务逻辑层│ ──────────→ │ 数据访问层│
└─────────┘              └─────────┘              └─────────┘
```

- 界面层依赖业务逻辑层（调用业务功能）
- 业务逻辑层依赖数据访问层（读写数据）
- 依赖关系是**单向的**，没有形成循环

### 4.4 路径包名示例

```
Sensors（外围包）
├── Vision（子包） → 完整路径：Sensors::Vision
│   ├── Camera
│   └── ImageProcessor
└── Audio（子包） → 完整路径：Sensors::Audio
```

## 5. 真题/实战

### 5.1 实战：电商系统包图设计

**需求**：为一个电商系统设计包图，包括用户管理、商品管理、订单管理、支付管理等模块。

**步骤1：识别主要的类**

| 模块 | 主要类 |
|------|--------|
| 用户管理 | User, Role, Permission, LoginService |
| 商品管理 | Product, Category, Inventory, ProductService |
| 订单管理 | Order, OrderItem, Cart, OrderService |
| 支付管理 | Payment, PaymentMethod, PaymentService |

**步骤2：按共同闭包原则分组**

```
┌──────────────┐    ┌──────────────┐
│  用户管理包    │    │  商品管理包    │
│  - User      │    │  - Product   │
│  - Role      │    │  - Category  │
│  - Permission│    │  - Inventory │
│  + LoginSvc  │    │  + ProductSvc│
└──────┬───────┘    └──────┬───────┘
       │                   │
       │    import         │    import
       ↓                   ↓
┌──────────────┐    ┌──────────────┐
│  订单管理包    │    │  支付管理包    │
│  - Order     │    │  - Payment   │
│  - OrderItem │    │  - PayMethod │
│  - Cart      │    │  + PaymentSvc│
│  + OrderSvc  │    └──────────────┘
└──────────────┘
```

**步骤3：检查循环依赖**
- 界面层 → 业务层 → 数据层：单向依赖，无循环
- 订单管理依赖用户管理和商品管理：单向，无循环
- 支付管理依赖订单管理：单向，无循环

**步骤4：确定可见性**
- Service类设为 `public`（对外提供接口）
- 实体类和内部实现设为 `private`（封装实现细节）

### 5.2 实战：图书馆管理系统包图设计

**场景**：设计一个图书馆管理系统的包结构。

**包划分**：
```
LibrarySystem
├── UI（用户界面包）
│   + LoginForm
│   + BookSearchForm
│   + BorrowForm
├── Business（业务逻辑包）
│   + BookManager
│   + ReaderManager
│   + BorrowManager
├── Data（数据访问包）
│   - BookDAO
│   - ReaderDAO
│   - BorrowDAO
└── Common（公共工具包）
    + DatabaseUtil
    + DateUtil
```

**依赖关系**：
```
UI ──import──→ Business ──import──→ Data
                     ──import──→ Common
```

### 5.3 常见建模错误

**错误1：包之间形成循环依赖**
```
❌ 错误：
订单包 ──→ 支付包
订单包 ←── 支付包  （循环！）

✅ 正确：
提取公共接口包，打破循环
订单包 ──→ 支付接口包 ←── 支付包
```

**错误2：包中元素可见性设置不当**
```
❌ 错误：所有元素都设为public
→ 外部可以随意访问内部实现，破坏封装性

✅ 正确：只暴露必要的接口
→ 内部实现设为private，对外接口设为public
```

**错误3：包的粒度不合适**
```
❌ 错误：一个包中只放一个类
→ 包失去了分组的意义

❌ 错误：一个包中放了上百个不相关的类
→ 违反了共同重用原则
```

## 6. 巩固练习

### 练习1：概念理解

**题目**：请简述包中元素的三种可见性及其含义。

**参考答案**：
- **public（+）**：任何导入该包的包都可以引用该元素
- **protected（#）**：只有该包的子包可以引用该元素
- **private（-）**：只有该包内部的元素可以引用该元素

### 练习2：原则辨析

**题目**：以下场景分别违反了哪个设计原则？

1. 包A中的类X修改后，需要同步修改包B中的类Y，但它们不在同一个包中
2. 包C中同时包含了"用户管理类"和"报表生成类"，两者几乎没有交互
3. 包D和包E互相依赖，无法独立重用

**参考答案**：
1. 违反了**共同闭包原则（CCP）**——需要同时改变的类应放在同一包中
2. 违反了**共同重用原则（CRP）**——不会一起使用的类不要放在同一包中
3. 违反了**非循环依赖原则（ADP）**——包之间不应形成循环依赖

### 练习3：设计实践

**题目**：为一个在线学习平台设计包图，包含以下功能模块：课程管理、学生管理、考试管理、成绩管理。要求：
1. 画出包的层次结构
2. 标注包之间的依赖关系
3. 确定各包中元素的可见性
4. 检查是否存在循环依赖

**参考答案要点**：

```
OnlineLearningPlatform
├── CoursePackage（课程管理）
│   + Course, + CourseService
│   - CourseDAO
├── StudentPackage（学生管理）
│   + Student, + StudentService
│   - StudentDAO
├── ExamPackage（考试管理）
│   + Exam, + ExamService
│   - ExamDAO
└── ScorePackage（成绩管理）
    + Score, + ScoreService
    - ScoreDAO
```

依赖关系：
- ExamPackage → CoursePackage（考试关联课程）
- ScorePackage → StudentPackage（成绩关联学生）
- ScorePackage → ExamPackage（成绩关联考试）

无循环依赖，设计合理。

### 练习4：判断题

判断以下说法是否正确，并说明理由：

1. 包之间的依赖关系具有传递性（ ）
2. 一个模型元素可以同时属于多个包（ ）
3. 子包可以继承父包中所有可见性的元素（ ）
4. REP、CCP、CRP三个原则可以同时满足（ ）
5. 包被实例化后会在内存中运行（ ）

**参考答案**：
1. **错误** —— 包之间的依赖关系没有传递性
2. **错误** —— 一个模型元素只能被一个包所拥有
3. **错误** —— 子包只能继承父包中public和protected的元素
4. **错误** —— REP、CCP、CRP三个原则事实上是相互排斥的
5. **错误** —— 组件可以被实例化，但包不能被实例化

### 练习5：简答题

**题目**：请解释高内聚、低耦合在包图设计中的具体体现。

**参考答案**：
- **高内聚**：一个包中的类应该紧密相关，具有相似的职责或需要同时变化
- **低耦合**：包之间的依赖关系应尽可能少，每个包应尽量独立
- **实现方法**：
  1. 最小化包之间的依赖关系
  2. 最小化每个包的公共元素和受保护元素的数目
  3. 最大化每个包的私有元素的数目

## 7. 总结复盘

### 7.1 本章核心知识点

| 知识点 | 关键内容 |
|--------|----------|
| 包的概念 | 将相关模型元素组织在一起的容器机制 |
| 包中的元素 | 可以是任何UML模型元素，一个元素只能属于一个包 |
| 包的命名 | 简单包名和路径包名（用 `::` 分隔） |
| 元素可见性 | public（+）、protected（#）、private（-） |
| 包的层次性 | 包可以嵌套形成层次结构 |
| 依赖关系 | 包之间通过类的依赖关系折射出来，无传递性 |
| 泛化关系 | 子包继承父包的public和protected元素 |
| 重用等价原则（REP） | 包作为可重用的单元 |
| 共同闭包原则（CCP） | 需要同时改变的类放在同一包中 |
| 共同重用原则（CRP） | 不会一起使用的类不要放在同一包中 |
| 非循环依赖原则（ADP） | 包之间不要形成循环依赖 |
| 高内聚低耦合 | 最小化依赖、最小化公共元素、最大化私有元素 |

### 7.2 常见错误与易混淆点

**错误1：认为包之间的依赖关系有传递性**
- 包A依赖包B，包B依赖包C，不意味着包A自动依赖包C
- 这与类之间的依赖关系不同

**错误2：混淆包的实例化**
- 组件（源代码文档）可以被实例化（装入内存运行）
- 包不能被实例化，它只是一个逻辑分组

**错误3：一个元素放在多个包中**
- 一个模型元素只能被一个包所拥有
- 这是包作为"容器"的基本特性

**错误4：忽略三原则的冲突**
- REP、CCP、CRP三个原则是相互排斥的
- 设计时需要根据实际情况在三者之间找到平衡

**错误5：子包可以访问父包所有元素**
- 子包只能继承父包中可见性为public和protected的元素
- private元素对外部（包括子包）不可见

**错误6：可见性设置过于开放**
- 不要将所有元素都设为public
- 应该遵循最小暴露原则，只将必要的接口设为public

**错误7：忽略循环依赖的危害**
- 循环依赖会导致包无法独立重用
- 严重妨碍软件的可扩展性
- 必须在设计阶段就识别并打破循环

### 7.3 包图的用途总结

1. **数据模型和对象模型之间的转换**：以包为单位进行
2. **Web建模**：包可以表示虚拟目录，目录下的所有Web元素都在包中
3. **控制单元**：方便团队开发和配置管理
4. **组件管理**：组件放在包中，同包一起布置在硬件结点上

### 7.4 记忆口诀

- **可见性三种**：公有对外全开放，保护只给子包用，私有仅限包内部
- **设计四原则**：重用等价打基础，共同闭包一起变，共同重用别硬塞，非循环依赖最关键
- **三原则关系**：REP、CCP、CRP，三者相互排斥，设计找平衡
- **内聚耦合**：高内聚靠分组，低耦合靠可见性控制
- **包与类的区别**：类可实例化，包只做分组；元素属一包，依赖无传递
