# 第11章 Rational统一过程

---

## 1. 概念引入：什么是 RUP，为什么它重要

### 1.1 开篇问题：软件开发过程的选择

一个团队接到一个中大型软件项目，面临关键抉择：是按照瀑布模型从头到尾线性推进，还是采用迭代方式逐步交付？如果需求在开发过程中频繁变更，瀑布模型的"一次性把事情做对"的假设就会被打破。

**Rational 统一过程（Rational Unified Process，简称 RUP）** 正是为了解决这类问题而诞生的——它提供了一种**用例驱动、以体系结构为中心、迭代和增量开发**的软件开发过程框架。

### 1.2 软件开发过程的定义

> **软件开发过程（SDP, Software Development Process）**，又称软件工程过程（Software Engineering Process），是一个将用户的需求转化为软件系统所需要的活动的集合。

用一句话概括：**A Process defines who is doing what, when, and how to reach a certain goal.**（一个过程定义了谁在做什么、何时做、如何做，以达到某个目标。）

常见的软件开发过程包括：
- 瀑布式软件开发方法
- 快速原型法
- 螺旋式开发方法
- 喷泉式开发方法
- 净室软件开发过程
- 个体软件过程 / 小组软件过程
- 极限编程（XP）
- **RUP**

### 1.3 统一过程与 UML 的关系

> **统一过程（Unified Process）** 是一个软件的开发过程，也是一个通用的过程框架，使用统一的建模语言 UML 来制定软件系统的所有蓝图。

关键区别：
- **UML** 是一种建模语言，独立于过程，可以用于不同的开发过程
- **RUP** 是一个完整的开发过程框架，UML 是其组成部分
- 如果不了解建模技术如何适应于过程，那么建模技术也就没有任何实际意义

## 2. 核心概念：RUP 的四要素与三大特点

### 2.1 RUP 的发展历史

RUP 的诞生与一个人的职业生涯密切相关——**Ivar Jacobson**。

```
时间线：
1967 ── Jacobson 在 Ericsson 公司就职，提出爱立信（Ericsson）方法
         └─ 基于构件的开发，将系统拆分为子系统/构件
         └─ 包含 RUP 中的很多概念：需求视图、静态视图、构架描述、
            动态视图、顺序图、协作图、状态图

1987 ── Jacobson 成立 Objectory 公司
         └─ 提出"对象工厂过程"，明确了用例的使用

1995 ── Objectory 公司被 Rational 公司收购
         └─ Jacobson 到 Rational 公司
         └─ 方法改称为 Rational Objectory Process
         └─ 提出"4+1"构架视图

1997 ── Rational 公司提出"Rational 对象工厂过程"
         └─ 继承过程模型和用例核心概念
         └─ 增加阶段和受控的迭代方法
         └─ 建立构架的精确定义

1998 ── Rational 公司正式发布"Rational 统一过程"（RUP）
```

**引入 RUP 的代表性企业：**

| 行业 | 企业 |
|------|------|
| 电信业 | Ericsson, Alcatel, MCI |
| 交通、航空、国防 | Lockheed-Martin, British Aerospace |
| 制造业 | Xerox, Volvo, Intel |
| 金融业 | Visa, Merrill Lynch, Schwab |
| 系统集成业 | Ernst & Young, Oracle, Deloitte & Touche |

### 2.2 RUP 的三大突出特点

RUP 最突出的特点在于提出了**"用例驱动、以体系结构为中心、迭代和增量开发"**的开发过程。

#### 特点一：用例驱动（Use-Case Driven）

用例始终贯穿在软件开发的整个过程中：
- **需求阶段**：通过用例捕获功能需求
- **分析和设计阶段**：顺序图、协作图、活动图都与用例有关
- **实现阶段**：所有用例汇总的实体类建立类图
- **测试阶段**：测试用例基于用例来设计

> 用例把分析、设计、实现、测试等所有过程"绑"到一起，是整个开发过程的粘合剂。

#### 特点二：以体系结构为中心（Architecture-Centric）

**体系结构**描述的是系统在其所处环境中最高层次的模仿，侧重于描述系统的重要结构部件（如类、组件和子系统），以及这些利用接口交互的各个部件的组织或结构。

体系结构层次的设计问题包括：
- 系统的总体组织和全局控制
- 通讯协议、同步
- 数据存取
- 给设计元素分配特定功能
- 设计元素的组织
- 物理分布
- 系统的伸缩性和性能

**"4+1"视图模型**——不同角色关注不同视图：

```
                    ┌──────────────┐
                    │   用例视图    │ ← 分析人员、测试人员（系统行为）
                    │  (Use Case)  │
                    └──────┬───────┘
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
┌───────┴──────┐  ┌────────┴───────┐  ┌───────┴──────┐
│   逻辑视图    │  │   进程视图      │  │   实现视图    │
│  (Logical)   │  │  (Process)     │  │(Implementation)│
│ 最终用户      │  │ 系统集成人员    │  │ 程序员        │
│ 系统功能      │  │ 性能、可伸缩性  │  │ 配置、装配    │
└──────────────┘  └────────────────┘  └──────────────┘
                           │
                    ┌──────┴───────┐
                    │   部署视图    │ ← 系统工程师（发布、安装、拓扑）
                    │ (Deployment) │
                    └──────────────┘
```

#### 特点三：迭代和增量开发（Iterative and Incremental Development）

- **迭代**：指工作流中的步骤
- **增量**：指系统中增加的部分
- 每次迭代都会增加或者明确一些目标系统的性质

迭代开发的优势：
1. 允许通过不断细化来加深对问题的理解
2. 可以降低项目的风险系数
3. 容易容纳需求的变更
4. 每个迭代以可执行版本告终，有利于鼓舞开发团队士气

### 2.3 RUP 的四个核心概念（四要素）

RUP 围绕四个核心概念展开，分别回答四个关键问题：

| 核心概念 | 英文 | 回答的问题 | 含义 |
|----------|------|-----------|------|
| **角色（Role）** | Role | **Who** — 谁来做 | 描述某个人或一个小组的行为与职责 |
| **活动（Activity）** | Activity | **How** — 怎么做 | 一个有明确目的的独立工作单元，定义角色执行的行为 |
| **制品（Artifact）** | Artifact | **What** — 做什么 | 活动生成、创建或修改的一段信息 |
| **工作流（Workflow）** | Workflow | **When** — 何时做 | 描述一个有意义的连续的活动序列 |

#### 角色（Role）—— Who

RUP 预先定义了很多角色，并对每个角色的工作和职责做了详尽说明：

| 角色 | 英文名 | 职责 |
|------|--------|------|
| 体系结构师 | Architect | 系统整体架构设计 |
| 设计人员 | Designer | 详细设计 |
| 实现人员 | Implementer | 编码实现 |
| 测试员 | Tester | 软件测试 |
| 配置管理人员 | Configuration Manager | 配置管理 |

#### 活动（Activity）—— How

RUP 中常见的活动举例：
- 项目经理计划一个迭代过程
- 系统分析员寻找用例和参与者
- 测试人员执行性能测试

#### 制品（Artifact）—— What

RUP 中常见的制品举例：
- 系统设计模型
- 项目计划文档
- 项目程序源代码

#### 工作流（Workflow）—— When

工作流描述了一个有意义的连续的活动序列，每个工作流产生一些有价值的产品，并显示了角色之间的关系。RUP 中强调三方面：

1. **核心工作流**：给出开发过程全面的工作流
2. **工作流细节**：将每个核心工作流进行分解，描述相关活动的输入、输出制品
3. **迭代计划**：从某一典型迭代过程的角度，更加详细地描述过程

## 3. 方法速通：RUP 的生命周期与实施步骤

### 3.1 RUP 的二维开发模型

RUP 的开发过程使用一种**二维结构**来表示：

```
          初始    细化    构造    移交
          阶段    阶段    阶段    阶段
          ┌───┐  ┌───┐  ┌───┐  ┌───┐
业务建模   │   │  │   │  │   │  │   │   ← 横轴：核心工作流
需求      │   │  │   │  │   │  │   │      （开发过程的成分）
分析设计   │   │  │   │  │   │  │   │
实现      │   │  │   │  │   │  │   │
测试      │   │  │   │  │   │  │   │
部署      │   │  │   │  │   │  │   │
配置管理   │   │  │   │  │   │  │   │
项目管理   │   │  │   │  │   │  │   │
环境      │   │  │   │  │   │  │   │
          └───┘  └───┘  └───┘  └───┘
           ↑      ↑      ↑      ↑
         里程碑  里程碑  里程碑  里程碑   ← 纵轴：时间维度
                                         （周期、阶段、迭代、里程碑）
```

- **横轴**代表时间，显示软件开发过程的生命周期安排（周期、阶段、迭代和里程碑）
- **纵轴**代表过程成分，确定开发过程包含的具体工作内容（九个核心工作流）

### 3.2 九个核心工作流详解

RUP 规定了 **9 个核心工作流**，其中 **6 个属于核心过程工作流**，**3 个属于支持工作流**。

#### 核心过程工作流（6 个）

**（1）业务建模（Business Modeling）**
- **目标**：理解待开发系统所在的机构及其商业运作，确保所有参与人员有共同认识，评估待开发系统对机构的影响
- **角色**：业务流程分析人员，业务设计员
- **制品**：业务词汇表，业务规则，业务用例模型，业务对象模型

**（2）需求（Requirements）**
- **目标**：定义系统功能及用户界面，使客户知道系统的功能，使开发人员理解系统的需求，为项目预算及计划提供基础
- **角色**：系统分析员，需求分析员，软件架构师，用例解释人员，用户界面设计员，需求评审员
- **制品**：软件需求规格说明书，用例模型

**（3）分析与设计（Analysis & Design）**
- **目标**：把需求分析的结果转化为分析与设计模型
- **角色**：软件架构师，设计人员，用户界面设计人员，数据库设计人员，封装设计人员
- **制品**：分析模型，设计模型，部署模型，数据模型，用户界面原型

**（4）实现（Implementation）**
- **目标**：把设计模型转换为实现结果，对代码做单元测试，将不同实现人员开发的模块集成为可执行系统
- **角色**：软件架构师，软件开发人员，软件测试人员，系统集成人员
- **制品**：源程序代码，单元测试报告，系统版本

**（5）测试（Test）**
- **目标**：检查各子系统的交互与集成，验证所有需求是否均被正确实现，对发现的软件质量缺陷进行归档，提出改进建议
- **角色**：测试设计人员，测试人员，测试分析人员
- **制品**：测试计划，测试结果，测试分析报告

**（6）部署（Deployment）**
- **目标**：打包、分发、安装软件，升级旧系统；培训用户及销售人员，提供技术支持；制定并实施 beta 测试
- **角色**：部署人员，课程开发人员，技术支持人员
- **制品**：运行环境打包，培训教材，用户技术手册

#### 支持工作流（3 个）

**（7）配置与变更管理（Configuration & Change Management）**
- **目标**：控制项目中的制品的变更，维护制品的完整性和一致性
- **角色**：项目经理，配置经理，变更控制经理
- **制品**：软件开发计划，配置管理计划，管理变更请求

**（8）项目管理（Project Management）**
- **目标**：为软件开发项目提供计划、人员分配、执行、监控等方面的指导，为风险管理提供框架
- **角色**：项目经理，项目评审员
- **制品**：软件开发计划，迭代计划，迭代评估，风险管理计划，产品验收计划，质量保证计划

**（9）环境（Environment）**
- **目标**：为软件开发机构提供软件开发环境，即提供过程管理和工具的支持
- **角色**：工具专家，系统管理员，流程工程师
- **制品**：开发环境包，开发案例，项目专用模版

### 3.3 四个阶段（Phase）

RUP 把软件开发生命周期划分为多个**周期（Cycle）**，每个 Cycle 生成一个产品的新的版本。每个 Cycle 都依次由四个连续的阶段组成：

```
┌─────────────┬─────────────┬─────────────┬─────────────┐
│  初始阶段    │  细化阶段    │  构造阶段    │  移交阶段    │
│ (Inception) │(Elaboration)│(Construction)│(Transition) │
│             │             │             │             │
│ 定义产品视图 │ 确定体系结构 │ 构造产品     │ 提交产品     │
│ 确定系统范围 │ 制定工作计划 │ 演进需求     │ 给用户使用   │
└──────┬──────┴──────┬──────┴──────┬──────┴──────┬──────┘
       │里程碑       │里程碑       │里程碑       │里程碑
```

| 阶段 | 英文名 | 核心任务 |
|------|--------|---------|
| 初始阶段 | Inception | 定义最终产品视图、业务模型并确定系统范围 |
| 细化阶段 | Elaboration | 设计及确定系统的体系结构，制定工作计划及资源要求 |
| 构造阶段 | Construction | 构造产品并继续演进需求、体系结构、计划直至产品提交 |
| 移交阶段 | Transition | 把产品提交给用户使用 |

**里程碑（Milestone）**：在每个阶段结束前都应有一个里程碑来评估该阶段的工作。如果未能通过里程碑的评估，决策者应做出决定是取消该项目，还是继续做该阶段的工作。

**迭代（Iteration）**：每一个阶段都由一个或多个连续的迭代组成。每一个迭代都是一个完整的开发过程，项目经理需要根据当前迭代所处的阶段以及上次迭代的结果，适当地对核心工作流中的行为进行裁剪。

### 3.4 RUP 的模型体系

首先建立**用例模型**，再以用例模型为核心，构造一系列的模型：

| 模型 | 作用 |
|------|------|
| **用例模型** | 定义所有的用例及用例之间的关系，以及用例与用户之间的关系 |
| **分析模型** | 定义问题域涉及的类及其属性和关系，更详细地提炼用例，将系统行为初步分配给一组对象 |
| **设计模型** | 将系统的静态结构定义为子系统、类和接口，并定义由子系统、类和接口之间的协作来实现的用例 |
| **配置模型** | 定义计算机的物理结点和组件到这些结点的映射 |
| **实现模型** | 定义组件和类到组件的映射 |
| **测试模型** | 定义用于验证用例的测试用例 |

### 3.5 RUP 的文档体系

RUP 开发过程所获得的文档资料包括**技术文档**和**管理文档**两大类：

**技术文档：**

| 信息集 | 包含内容 |
|--------|---------|
| 需求分析信息集 | 软件需求规格说明书，需求补充说明，业务用例图 |
| 设计信息集 | 软件设计规格说明书，图形界面，类图，对象图，包图，顺序图，协作图，状态图，活动图 |
| 实现信息集 | 源程序清单，动态链接库说明，用户使用手册，组件图 |
| 配置信息集 | 部署图及详细说明 |

**管理文档：**

风险分析说明、软件开发进度计划、配置计划、测试计划、测试方案与步骤。

### 3.6 RUP 的剪裁（Tailoring）五步法

RUP 是一个通用的过程框架，具体应用时需要做裁剪。RUP 就像一个**元过程（meta-process）**，通过裁剪可以得到适合不同开发机构和项目需要的具体开发过程实例。

**裁剪五步法：**

```
步骤1：确定本项目的软件过程需要哪些工作流
       └─ 九大核心工作流并不总是需要的
       └─ 例：嵌入式软件项目一般不需要业务建模工作流

步骤2：确定每个工作流产出哪些制品
       └─ 规定工作流应产出哪些类型的文档

步骤3：确定阶段间演进
       └─ 以风险控制为原则
       └─ 决定每个阶段执行哪几个工作流、执行到什么程度、产出哪些制品

步骤4：确定阶段内的迭代计划
       └─ 决定是否采用迭代开发
       └─ 如果采用，每次迭代开发的内容有哪些

步骤5：规划工作流内部结构  ← 裁剪的难点
       └─ 决定本软件过程要设立哪些角色
       └─ 规划工作流内部的结构（通常用活动图给出）
```

**裁剪的本质**是软件过程开发的再工程：
1. 对 RUP 进行逆向工程，得到《RUP 开发需求》《RUP 设计方案》等文档
2. 考虑具体软件项目，得到项目软件过程需求
3. 比较两个需求，借鉴 RUP 设计方案，进行正向工程，得到项目软件开发过程

## 4. 典型示例：RUP 六大最佳实践

### 4.1 六个最佳开发经验概览

RUP 总结了六个最佳开发经验（Best Practices）：

```
┌─────────────────────────────────────────────────────┐
│                RUP 六大最佳实践                        │
├─────────────────────────────────────────────────────┤
│  1. 迭代式开发     (Develop software iteratively)    │
│  2. 管理需求       (Manage requirements)             │
│  3. 使用基于构件的体系结构                             │
│     (Use component-based architectures)             │
│  4. 可视化软件建模 (Visually model software)          │
│  5. 验证软件质量   (Verify software quality)          │
│  6. 控制软件变更   (Control changes to software)      │
└─────────────────────────────────────────────────────┘
```

### 4.2 最佳实践详解

#### 实践一：迭代式开发

**问题**：软件系统非常复杂，很难按照"定义整个问题 → 设计整个系统 → 实现软件 → 测试产品"的顺序线性进行（瀑布式开发的缺点）。

**优势**：
- 迭代允许通过不断地细化来加深对问题的理解
- 迭代可以降低项目的风险系数
- 迭代容易容纳需求的变更
- 每个迭代过程以可执行版本告终，有利于鼓舞开发团队的士气

#### 实践二：管理需求

RUP 描述了如何提取、组织和文档化系统的功能和约束条件。有效需求管理的好处：
- 更好地控制复杂的项目
- 改善软件质量和客户满意度
- 降低项目的开发费用和时间（修正需求中错误需要很大的开销）
- 改善团队之间的交流
- Use Case 和 Scenario 的使用被证明是捕获功能性需求的卓越方法

#### 实践三：使用基于构件的体系结构

- 设计时以体系结构为中心
- 早期的迭代开发主要是产生并验证软件体系结构
- 在随后的迭代开发中，逐渐演化为最终系统
- 设计灵活、适应变化、易于理解、促进软件复用的弹性架构

#### 实践四：可视化软件建模

- UML 是工业标准的统一建模语言
- 作为一种图形化语言，UML 是对系统模型进行可视化建模的基础
- 提供了表示系统模型及其中各类元素的标准方法
- 保证利用图形化模型进行交流时理解的一致性，不会产生歧义

#### 实践五：验证软件质量

- 软件质量低下是影响软件使用的最重要的因素
- 质量评估被内建于过程所有的活动，包括全体成员
- 使用客观的度量和标准，不是事后型或单独小组进行的分离活动
- 将开发过程划分成各个可以独立评审的小过程，在整个流程中贯穿质量概念

#### 实践六：控制软件变更

- 软件开发中变更是不可避免的
- 确定每个修改是可接受的、能被跟踪的
- 通过控制制品（模型、代码、文档等），隔离来自其他工作空间的变更
- 为每个开发人员建立安全的工作空间

## 5. 真题/实战：RUP 的实际应用场景

### 5.1 RUP 在电信行业的应用

**背景**：Ericsson 公司是 RUP 的发源地。在大型电信系统开发中，系统由数百个子系统和构件组成。

**RUP 的应用方式**：
- 通过**业务建模**工作流理解电信业务流程
- 用**用例**捕获通话、计费、网络管理等功能需求
- 在**细化阶段**确定分布式系统的体系结构
- 通过**迭代开发**逐步交付各子系统

### 5.2 RUP 裁剪实例：小型 Web 项目

假设一个 5 人团队开发一个企业内部管理系统，对 RUP 进行裁剪：

| 裁剪决策 | 说明 |
|----------|------|
| **工作流选择** | 保留需求、分析与设计、实现、测试、项目管理；省略业务建模（内部系统，业务流程清晰）、部署（轻量级部署）、环境（团队已有开发环境） |
| **制品精简** | 需求阶段仅输出用例模型和简版需求规格说明书；设计阶段输出核心类图和顺序图 |
| **阶段演进** | 初始阶段 1 次迭代，细化阶段 1 次迭代，构造阶段 2 次迭代，移交阶段 1 次迭代 |
| **角色合并** | 架构师兼任设计人员，项目经理兼任配置管理人员 |

### 5.3 "4+1"视图模型应用实例

以一个**在线购物系统**为例，说明不同角色关注的视图：

| 角色 | 关注视图 | 具体关注点 |
|------|----------|-----------|
| 系统分析师 | 用例视图 | 购物车管理、订单处理、支付等用例的行为 |
| 最终用户 | 逻辑视图 | 商品浏览、下单、查询订单等功能 |
| 程序员 | 实现视图 | 前端组件、后端服务模块的配置和装配 |
| 系统集成人员 | 进程视图 | 并发用户处理、订单高峰期的吞吐率 |
| 系统工程师 | 部署视图 | Web 服务器、数据库服务器的部署拓扑 |

## 6. 巩固练习

### 练习一：概念辨析

**题目**：判断以下说法是否正确，并说明理由。

1. RUP 是一种建模语言，用于绘制 UML 图。
2. RUP 的三大特点是：用例驱动、以体系结构为中心、迭代和增量开发。
3. RUP 的四个阶段是：需求、设计、编码、测试。
4. RUP 的裁剪意味着删除不需要的工作流和角色。
5. 迭代和增量是同一个概念。

**参考答案**：

1. **错误**。RUP 是一个软件开发过程框架，UML 才是建模语言。UML 是 RUP 的组成部分，但二者不等同。
2. **正确**。这是 RUP 最突出的三大特点。
3. **错误**。RUP 的四个阶段是初始阶段（Inception）、细化阶段（Elaboration）、构造阶段（Construction）、移交阶段（Transition）。需求、设计、编码、测试属于工作流而非阶段。
4. **不完全正确**。RUP 的裁剪是一个系统化的过程（五步法），不仅涉及工作流和角色的取舍，还包括确定制品、阶段间演进、迭代计划、工作流内部结构等。
5. **错误**。迭代是指工作流中的步骤，增量是指系统中增加的部分。二者相关但不同。

### 练习二：填空题

1. RUP 的四个核心概念是：____、____、____、____，分别回答了____、____、____、____四个问题。
2. RUP 规定了____个核心工作流，其中____个属于核心过程工作流，____个属于支持工作流。
3. RUP 的六个最佳实践是：____、____、____、____、____、____。
4. RUP 的二维结构中，横轴代表____，纵轴代表____。
5. RUP 的发展历史与____密切相关，他在____年成立了 Objectory 公司。

**参考答案**：

1. 角色（Role）、活动（Activity）、制品（Artifact）、工作流（Workflow）；Who、How、What、When
2. 9；6；3
3. 迭代式开发、管理需求、使用基于构件的体系结构、可视化软件建模、验证软件质量、控制软件变更
4. 时间（生命周期）；过程成分（核心工作流）
5. Ivar Jacobson；1987

### 练习三：简答题

1. 简述 RUP 四个阶段的核心任务和每个阶段的里程碑意义。
2. 试比较 RUP 与瀑布模型的主要区别。
3. 为什么说"用例驱动"是 RUP 的核心特征？用例如何贯穿整个开发过程？

### 练习四：综合分析题

**题目**：某公司计划开发一个银行核心业务系统，团队有 20 人，项目周期为 18 个月。请运用 RUP 的知识，回答以下问题：

1. 该项目是否需要"业务建模"工作流？为什么？
2. 如何规划四个阶段的时间分配（大致比例）？
3. 在"细化阶段"，主要应该完成哪些工作？产出哪些制品？
4. 如何利用"4+1"视图模型来组织系统设计？

**参考要点**：

1. **需要**。银行核心业务系统涉及复杂的业务流程（开户、转账、贷款、结算等），需要通过业务建模来确保所有参与人员对业务有共同认识。
2. 参考比例：初始阶段约 10%（~2个月），细化阶段约 30%（~5个月），构造阶段约 50%（~9个月），移交阶段约 10%（~2个月）。
3. 细化阶段主要工作：设计并确定系统体系结构、识别高风险用例并进行详细分析、制定迭代计划和资源需求。主要制品：分析模型、设计模型、部署模型、风险列表、迭代计划。
4. 用例视图：银行业务用例（存取款、转账、贷款审批等）；逻辑视图：账户类、交易类、客户类等核心领域模型；进程视图：并发交易处理、事务管理；实现视图：各服务模块和组件的组织；部署视图：应用服务器、数据库集群的物理部署。

### 7.1 核心知识图谱

        ┌────────────┬───────────┼───────────┬────────────┐
        │            │           │           │            │
    发展历史      三大特点     四个核心概念   生命周期      最佳实践

    ├─Objectory  ├─体系结构   ├─活动(How)  │ ├─初始      ├─管理需求
    │ 中心       ├─制品(What) │ ├─细化     ├─构件体系结构
    ├─Rational   └─迭代增量   └─工作流     │ ├─构造     ├─可视化建模

      发布RUP                             └─裁剪五步法

### 7.2 核心要点速记

**一句话记住 RUP**：RUP 是一个**用例驱动、以体系结构为中心、迭代和增量开发**的软件开发过程框架，由四个阶段（初始、细化、构造、移交）和九个核心工作流组成。

**速记口诀**：
- 三大特点："用例驱动、体系结构、迭代增量"
- 四个阶段："初（初始）细（细化）构（构造）移（移交）"
- 四个核心概念："人（Role）做（Activity）事（Artifact）时（Workflow）" → "谁做、怎么做、做什么、何时做"
- 六大最佳实践："迭需构模质变"（迭代、需求、构件、建模、质量、变更）

### 7.3 常见错误与易混淆点

| 常见错误 | 正确理解 |
|----------|----------|
| 把 RUP 等同于 UML | UML 是建模语言，RUP 是开发过程框架；UML 是 RUP 的组成部分 |
| 把"阶段"和"工作流"混为一谈 | 阶段是时间维度（初/细/构/移），工作流是内容维度（业务建模/需求/...） |
| 认为 RUP 必须严格按模板执行 | RUP 是通用框架，必须根据项目情况进行裁剪 |
| 认为迭代就是重复做同样的事 | 每次迭代都是一个完整的开发过程，但目标和内容不同，是"增量"式的推进 |
| 混淆"迭代"和"增量" | 迭代是工作流中的步骤，增量是系统中增加的部分 |
| 认为 RUP 只适用于大型项目 | RUP 可以通过裁剪应用于各种规模的项目 |
| 忽略里程碑的作用 | 里程碑是阶段结束前的评估点，用于决定项目是否继续 |
| 认为所有项目都需要 9 个工作流 | 需要根据项目特点裁剪，如嵌入式项目可省略业务建模 |

### 7.4 与其他开发过程的对比

| 维度 | 瀑布模型 | RUP | 极限编程（XP） |
|------|----------|-----|---------------|
| 开发方式 | 线性顺序 | 迭代增量 | 迭代增量 |
| 需求变更 | 难以适应 | 通过迭代容纳 | 拥抱变化 |
| 文档要求 | 重文档 | 适度文档 | 轻文档 |
| 体系结构 | 前期设计 | 以体系结构为中心 | 简单设计 |
| 适用规模 | 小型、需求明确 | 中大型项目 | 小型、需求多变 |
| 核心驱动 | 文档驱动 | 用例驱动 | 测试驱动 |
