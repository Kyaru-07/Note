## 第1章 面向对象技术概述

### 1.1 面向对象的基本概念

#### 对象（Object）
- **定义**：系统中用来描述客观事物的一个实体，是用来构成系统的一个基本单位
- **组成**：由一组属性和一组行为构成
- **特性**：
  - **标识**：对象可识别的特性
  - **状态**：对象属性的值
  - **行为**：对象能执行的操作

#### 类（Class）
- **定义**：具有相同属性和行为的一组对象的集合
- **组成**：名称、属性、操作
- **类与对象的关系**：
  - 对象是类的实例
  - 类是对象的定义模板
  - 类是对象集合的再抽象

#### 属性（Attribute）
- **定义**：描述对象静态特征的一个数据项
- **类属性**：描述类的所有对象的共同特征，对于任何对象实例，属性值都相同

#### 服务/操作（Service/Operation）
- **定义**：描述对象动态特征（行为）的一个操作序列

### 1.2 面向对象的四大特征

| 特征 | 定义 | 关键点 |
|------|------|--------|
| **封装** | 把对象的属性和服务结合成独立单位，隐蔽内部细节 | 信息隐藏 |
| **继承** | 特殊类拥有其一般类的全部属性和操作 | 代码复用 |
| **多态** | 同一操作在不同类中有不同实现 | 灵活性 |
| **分类** | 将具有相同特征的对象归为一类 | 抽象化 |

### 1.3 面向对象方法与结构化方法的区别

| 维度 | 结构化方法 | 面向对象方法 |
|------|------------|--------------|
| 分析角度 | 功能分解 | 从现实对象角度 |
| 基本单位 | 功能/过程 | 对象 |
| 数据与操作 | 分离 | 封装在一起 |
| 复用机制 | 函数调用 | 继承、多态 |

### 1.4 面向对象的软件工程

- **OOA**（面向对象分析）
- **OOD**（面向对象设计）
- **OOP**（面向对象编程）
- **OOT**（面向对象测试）
- **OOSM**（面向对象的软件维护）

### 1.5 面向对象设计原则（6大原则）

1. **开放封闭原则**：对扩展开放，对修改关闭
2. **单一职责原则**：一个类只有一个引起变化的原因
3. **依赖倒置原则**：依赖于抽象，不依赖于具体
4. **Liskov替换原则**：子类必须能够替换父类
5. **迪米特法则**：一个对象应该对其他对象有最少的了解
6. **接口隔离原则**：客户端不应该依赖它不需要的接口

### 1.6 系统模型

面向对象的系统分析要确立3个系统模型：
- **对象模型**（静态模型）
- **行为模型**（动态模型）
- **功能模型**

## 第2章 UML概述

### 2.1 UML的定义

UML（Unified Modeling Language）是**统一建模语言**，是面向对象软件的标准化建模语言。

### 2.2 UML的组成

UML由3个部分组成：
1. **基本构造块**：事物、关系、图
2. **规则**：如何将基本构造块放在一起
3. **公共机制**：规格说明、修饰、通用划分

### 2.3 UML的事物

| 类型 | 说明 | 示例 |
|------|------|------|
| 结构事物 | 描述物理或概念元素 | 类、接口、协作、用例、组件、节点 |
| 行为事物 | 描述跨越时间和空间的行为 | 交互、活动 |
| 分组事物 | 描述组织结构 | 包 |
| 注释事物 | 描述解释说明 | 注释 |

### 2.4 UML的关系

| 关系 | 说明 | 符号 |
|------|------|------|
| **依赖** | 一个元素的变化影响另一个元素 | 虚线箭头 |
| **关联** | 类之间的结构关系 | 实线 |
| **泛化** | 一般-特殊关系（继承） | 空心三角箭头 |
| **实现** | 接口与实现类之间的关系 | 虚线空心三角箭头 |

### 2.5 UML的视图

| 视图 | 用途 | 对应的图 |
|------|------|----------|
| **用例视图** | 描述系统功能需求 | 用例图 |
| **逻辑视图** | 描述系统静态结构 | 类图、对象图 |
| **进程视图** | 描述系统并发执行 | 顺序图、协作图、状态图 |
| **实现视图** | 描述系统组件结构 | 组件图 |
| **部署视图** | 描述系统物理部署 | 部署图 |

### 2.6 UML的10种基本图

| 分类 | 图名称 | 用途 |
|------|--------|------|
| **结构图** | 类图 | 描述系统静态结构 |
| | 对象图 | 描述某一时刻的对象实例 |
| | 包图 | 描述系统包结构 |
| | 组件图 | 描述系统组件结构 |
| | 部署图 | 描述系统物理部署 |
| **行为图** | 用例图 | 描述系统功能需求 |
| | 活动图 | 描述系统的业务流程 |
| | 状态图 | 描述对象的状态变化 |
| **交互图** | 顺序图 | 描述对象交互的时间顺序 |
| | 协作图 | 描述对象交互的组织结构 |

### 2.7 UML的扩展机制

- **构造型（Stereotype）**：扩展UML词汇
- **标记值（Tagged Value）**：扩展元素的属性
- **约束（Constraint）**：扩展元素的语义

### 2.8 UML的优点

唯一性、连续性、维护性、复用性、逐步完善

## 第3章 用例图

### 3.1 用例图的组成

用例图由4个要素组成：
1. **参与者（Actor）**：系统外部与系统交互的角色
2. **用例（Use Case）**：系统提供的功能单元
3. **系统边界（System Boundary）**：系统与外界的分界线
4. **关联（Association）**：参与者与用例之间的关系

### 3.2 参与者

#### 参与者的定义
- 是系统外部的、需要与系统进行交互的客观实体
- 可以是**人**、**外部系统**或**外部设备**

#### 参与者之间的关系
- **泛化关系**：一般-特殊关系（如"管理员"是"用户"的特例）

#### 识别参与者的问题
- 谁使用系统？
- 谁安装/维护系统？
- 谁启动/关闭系统？
- 系统需要与其他系统交互吗？
- 系统需要从外部获取数据吗？

### 3.3 用例

#### 用例的定义
- 描述参与者与系统之间的一次交互
- 用例是系统中的一个功能单元

#### 用例的特征
- 用例是**独立**的
- 用例由**参与者**触发
- 用例描述**参与者可见**的结果
- 用例是一个**完整的流程**

#### 用例粒度
- 用例所包含的系统服务或功能单元的多少
- 粒度太大：描述不够详细
- 粒度太小：模型过于复杂

### 3.4 用例之间的关系

| 关系 | 定义 | 符号 | 示例 |
|------|------|------|------|
| **包含（Include）** | 基本用例的部分功能被提取出来 | `<<include>>` | "购物"包含"结算" |
| **扩展（Extend）** | 扩展用例为基本用例添加额外功能 | `<<extend>>` | "购物"扩展"使用优惠券" |
| **泛化（Generalization）** | 用例之间的一般-特殊关系 | 空心三角箭头 | "支付"泛化"信用卡支付" |

#### 包含 vs 扩展 vs 泛化

| 维度 | 包含 | 扩展 | 泛化 |
|------|------|------|------|
| 执行 | 一定会执行 | 条件满足才执行 | 子用例可单独执行 |
| 依赖 | 基本用例依赖包含用例 | 扩展用例依赖基本用例 | 子用例依赖父用例 |
| 箭头方向 | 基本→包含 | 扩展→基本 | 子→父 |

### 3.5 用例建模步骤

1. **确定系统边界**
2. **确定参与者**
3. **确定用例**
4. **细化用例**
5. **编写用例描述**
6. **审核用例模型**

### 3.6 用例描述的内容

- 用例名称
- 参与者
- 前置条件
- 后置条件
- 基本操作流程
- 可选操作流程
- 被泛化的用例
- 被包含的用例
- 被扩展的用例

## 第4章 类图与对象图

### 4.1 类图的组成

类图由以下元素组成：
1. **类（Class）**
2. **类之间的关系**
3. **接口（Interface）**

### 4.2 类的表示

```
+------------------+
|      类名        |
+------------------+
| 属性1 : 类型     |
| 属性2 : 类型     |
+------------------+
| 操作1() : 返回值 |
| 操作2() : 返回值 |
+------------------+
```

#### 类的可见性

| 符号 | 可见性 | 说明 |
|------|--------|------|
| + | public | 公有 |
| - | private | 私有 |
| # | protected | 受保护 |
| ~ | package | 包内可见 |

#### 类的三要素
- **名称**：类的标识
- **属性**：描述类的静态特征
- **操作**：描述类的动态行为

### 4.3 类之间的关系

| 关系 | 定义 | 符号 | 多重性 |
|------|------|------|--------|
| **依赖** | 一个类的变化影响另一个类 | 虚线箭头 | - |
| **关联** | 类之间的结构关系 | 实线 | 可标注 |
| **聚合** | 整体-部分关系（弱） | 空心菱形 | 可标注 |
| **组合** | 整体-部分关系（强） | 实心菱形 | 可标注 |
| **泛化** | 一般-特殊关系（继承） | 空心三角箭头 | - |
| **实现** | 接口与实现类之间的关系 | 虚线空心三角 | - |

#### 关联 vs 聚合 vs 组合

| 维度 | 关联 | 聚合 | 组合 |
|------|------|------|------|
| 生命周期 | 独立 | 部分可独立 | 部分不可独立 |
| 耦合度 | 低 | 中 | 高 |
| 示例 | 学生-课程 | 部门-员工 | 车-发动机 |

### 4.4 多重性

| 符号 | 含义 |
|------|------|
| 1 | 恰好1个 |
| 0..1 | 0或1个 |
| 0..* | 0或多个 |
| 1..* | 1或多个 |
| * | 0或多个（等同0..*） |
| m..n | m到n个 |

### 4.5 接口

- **定义**：可以在整个模型中反复使用的一组行为
- **特点**：没有属性，只有操作
- **表示**：`<<interface>>` 构造型

### 4.6 对象图

- **定义**：描述某一时刻对象实例的图
- **用途**：验证类图的正确性
- **组成**：对象、链（关联的实例）

## 第5章 顺序图与协作图

### 5.1 交互图概述

交互图描述系统中对象间的交互行为，是描述一组对象如何合作完成某个行为的模型化工具。

| 类型 | 特点 | 侧重点 |
|------|------|--------|
| **顺序图** | 强调消息的时间顺序 | 时间顺序 |
| **协作图** | 强调参与交互对象的组织 | 空间关系 |

### 5.2 顺序图的组成

| 元素 | 说明 | 符号 |
|------|------|------|
| **对象** | 交互的参与者 | 矩形框（带下划线） |
| **生命线** | 对象存在的时间段 | 垂直虚线 |
| **激活** | 对象执行操作的时间 | 矩形条 |
| **消息** | 对象之间的通信 | 箭头线 |

### 5.3 消息类型

| 类型 | 说明 | 符号 |
|------|------|------|
| **调用消息（同步）** | 发送者等待接收者完成 | 实心箭头 |
| **异步消息** | 发送者不等待 | 开放箭头 |
| **简单消息** | 不关心消息类型 | 普通箭头 |
| **返回消息** | 返回控制 | 虚线箭头 |

#### 消息的组成
- **发送者**
- **接收者**
- **活动**

#### 消息编号
- 顺序图中使用多层次编号表示消息的嵌套
- 例如：1.1, 1.2, 2.1

### 5.4 协作图的组成

| 元素 | 说明 |
|------|------|
| **对象** | 交互的参与者 |
| **链** | 对象之间的连接 |
| **消息** | 带有序号的消息 |

### 5.5 顺序图与协作图的比较

| 维度 | 顺序图 | 协作图 |
|------|--------|--------|
| 侧重点 | 时间顺序 | 空间组织 |
| 消息编号 | 不需要（用位置表示） | 需要（用序号表示） |
| 生命线 | 有 | 没有 |
| 激活 | 有 | 没有 |
| 可相互转换 | 是 | 是 |

## 第6章 状态图与活动图

### 6.1 状态图

#### 状态图的定义
描述一个对象在其生命周期里随事件的发生而改变状态的变化过程。

#### 状态的分类

| 类型 | 定义 | 示例 |
|------|------|------|
| **简单状态** | 不包含子状态的状态 | "空闲" |
| **组合状态** | 包含子状态的状态 | "运动状态"（含加速、匀速、减速） |
| **历史状态** | 上次离开组合状态时的最后活动子状态 | 用H表示 |

#### 事件的类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **信号事件** | 接收到信号 | 鼠标点击 |
| **改变事件** | 条件满足时触发 | 温度>30℃ |
| **调用事件** | 调用操作时触发 | 方法调用 |
| **时间事件** | 指定时间到达 | after(20 seconds) |

#### 转移（Transition）
- 从一个状态到另一个状态的变化
- 包含：源状态、目标状态、触发事件、监护条件、动作

#### 状态图的基本元素
- 初始状态（实心圆）
- 终止状态（带圈的实心圆）
- 状态（圆角矩形）
- 转移（箭头）

### 6.2 活动图

#### 活动图的定义
描述系统从一个活动到另一个活动的流程，用于描述业务流程或操作流程。

#### 活动图的组成

| 元素 | 说明 | 符号 |
|------|------|------|
| **活动状态** | 一个工作步骤 | 圆角矩形 |
| **决策** | 条件判断 | 菱形 |
| **分叉/汇合** | 并发执行 | 粗横线 |
| **泳道** | 职责分组 | 矩形分区 |
| **对象流** | 对象在活动间的传递 | 虚线箭头+对象 |
| **初始节点** | 流程开始 | 黑色实心圆 |
| **终止节点** | 流程结束 | 带圈的实心圆 |

#### 活动状态 vs 动作状态

| 维度 | 活动状态 | 动作状态 |
|------|----------|----------|
| 原子性 | 非原子性 | 原子性 |
| 可中断 | 可以 | 不可以 |
| 内部转移 | 可以有 | 不可以 |

#### 分支 vs 分叉

| 维度 | 分支 | 分叉 |
|------|------|------|
| 目的 | 条件选择 | 并发执行 |
| 符号 | 菱形 | 粗横线 |
| 结果 | 互斥 | 并行 |

### 6.3 状态图与活动图的区别

| 维度 | 状态图 | 活动图 |
|------|--------|--------|
| 描述对象 | 单个对象 | 多个对象/整个流程 |
| 侧重点 | 状态变化 | 活动流程 |
| 应用场景 | 反应型对象 | 业务流程/算法流程 |
| 转移触发 | 事件 | 活动完成自动触发 |

## 第7章 包图

### 7.2 包的组成

- **包（Package）**：模型元素的容器
- **包之间的关系**：依赖、泛化

### 7.3 包的可见性

| 符号 | 可见性 | 说明 |
|------|--------|------|
| + | public | 公有 |
| - | private | 私有 |
| # | protected | 受保护 |

### 7.4 包之间的关系

| 关系 | 说明 | 符号 |
|------|------|------|
| **依赖** | 一个包的变化可能影响另一个包 | 虚线箭头 |
| **泛化** | 包之间的继承关系 | 空心三角箭头 |

## 第9章 面向对象设计原则

### 9.1 六大设计原则

#### 1. 单一职责原则（SRP）
- **定义**：一个类应该只有一个引起它变化的原因
- **目的**：降低类的复杂度，提高可读性和可维护性

#### 2. 开放封闭原则（OCP）
- **定义**：对扩展开放，对修改关闭
- **目的**：通过抽象和多态实现可扩展性

#### 3. 里氏替换原则（LSP）
- **定义**：子类必须能够替换其父类
- **目的**：确保继承的正确使用

#### 4. 依赖倒置原则（DIP）
- **定义**：依赖于抽象，不依赖于具体
- **目的**：降低模块间的耦合度

#### 5. 接口隔离原则（ISP）
- **定义**：客户端不应该依赖它不需要的接口
- **目的**：减少接口的冗余

#### 6. 迪米特法则（LoD）
- **定义**：一个对象应该对其他对象有最少的了解
- **目的**：降低耦合度

## 第10章 设计模式

### 10.1 设计模式的分类

| 类型 | 目的 | 常用模式 |
|------|------|----------|
| **创建型** | 对象创建机制 | 单例、工厂方法、抽象工厂、建造者、原型 |
| **结构型** | 类和对象的组合 | 适配器、桥接、组合、装饰器、外观、享元、代理 |
| **行为型** | 对象之间的通信 | 责任链、命令、迭代器、观察者、策略、模板方法 |

### 10.2 常用设计模式简介

| 模式 | 类型 | 核心思想 |
|------|------|----------|
| **单例模式** | 创建型 | 确保一个类只有一个实例 |
| **工厂方法** | 创建型 | 定义创建对象的接口，由子类决定实例化哪个类 |
| **观察者模式** | 行为型 | 一对多依赖关系，对象状态变化时通知所有依赖者 |
| **策略模式** | 行为型 | 定义一系列算法，使它们可以互相替换 |
| **适配器模式** | 结构型 | 将一个类的接口转换成客户期望的接口 |
| **装饰器模式** | 结构型 | 动态地给对象添加职责 |

## 附录：核心概念速查表

### 面向对象核心概念

| 概念 | 定义 | 关键特征 |
|------|------|----------|
| 类 | 具有相同属性和行为的一组对象的集合 | 封装、继承、多态 |
| 对象 | 类的实例，系统中描述客观事物的实体 | 标识、状态、行为 |
| 封装 | 把对象的属性和服务结合成独立单位 | 信息隐藏 |
| 继承 | 特殊类拥有其一般类的全部属性和操作 | 代码复用 |
| 多态 | 同一操作在不同类中有不同实现 | 灵活性 |
| 关联 | 类之间的结构关系 | 持久性 |
| 聚合 | 整体-部分关系（弱） | 可分离 |
| 组合 | 整体-部分关系（强） | 不可分离 |
| 依赖 | 一个类的变化影响另一个类 | 临时性 |
| 泛化 | 一般-特殊关系 | 继承性 |
| 实现 | 接口与实现类之间的关系 | 抽象性 |
